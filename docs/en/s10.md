# Designing Workflows and Automation

> "Trigger, condition, action. Start simple — but know that complexity can grow fast."

## The Problem

Repetitive manual work accumulates.

Every new ticket gets a tag added by hand.

Every high-priority issue triggers a Slack message typed by a human.

Every SLA breach requires someone to notice it and escalate.

This isn't just inefficient — it's error-prone.

The moments when automation matters most are the same moments people are busiest and most likely to miss a step.

## The Solution

DevRev's Automations engine lets you define rules that execute automatically when conditions are met.

Simple automations take minutes to configure.

The engine also supports AI condition nodes and parallel execution paths for complex scenarios — making it one of the most powerful layers of what DevRev calls the AI Trifecta alongside Search and Analytics.

**Note**: This session covers the conceptual foundation for s14 (Agent Studio).

Workflows are rule-based; Agent Studio is goal-based.

Understanding Workflows first makes the distinction clear.

## How It Works

**The Three-Element Structure**

Every automation in DevRev is built from three elements.

1. **Trigger**: What event starts the automation
2. **Condition**: An optional additional check after the trigger fires
3. **Action**: What to execute when the condition is met

**Trigger Examples**

| Trigger | When It Fires |
|---------|--------------|
| Ticket Created | A new ticket is submitted |
| Ticket Updated | Any field on a ticket changes |
| Issue Stage Changed | An issue moves to a new stage |
| SLA Breached | A ticket exceeds its SLA target |

**Condition Examples**

- Priority is Blocker
- The ticket's linked Part is "Billing"
- The ticket body contains a specific keyword

**Action Examples**

- Add a tag
- Assign to a specific team member
- Post a message to a Slack channel
- Change ticket priority
- Create a linked issue

**A Simple Workflow Example**

"When a ticket is created, if the body contains the word 'urgent', set Priority to Blocker and post to #urgent-tickets in Slack."

Broken into the three elements:

- Trigger: Ticket Created
- Condition: Body contains "urgent"
- Action 1: Set Priority to Blocker
- Action 2: Post to Slack #urgent-tickets

**Where to find it**

Workflows are created and managed from **Settings > Workflows**. This is the "Workflows" category in the Settings map covered in s02.

---

**Workflow Node Types**

DevRev's workflow engine goes beyond simple if-then. You can combine these node types to build complex flows:

| Node | Description | Example use |
|------|-------------|-------------|
| **Condition** | Branch (if-then-else) | Route Blocker tickets to a special path |
| **AI Condition** | AI analyzes text and decides semantically | Classify "angry customer" vs "casual question" |
| **Parallel** | Execute multiple paths simultaneously | Send notification AND reassign at the same time |
| **Wait** | Pause for a duration before continuing | Remind after 24 hours if no response |
| **Loop** | Repeat until a condition is met | Retry until approval is received |
| **Action** | Execute a concrete operation | Add tag, assign owner, send notification |

**Where Complexity Grows**

Basic rule automations are approachable. Complexity increases sharply when you combine the nodes above:

- **AI condition nodes**: Instead of keyword matching, an AI model evaluates the ticket's meaning and intent. More flexible, but harder to test deterministically.
- **Parallel paths**: After a condition, multiple action branches execute simultaneously.
- **Wait nodes**: Pause execution for a set duration before continuing.
- **Loops**: Repeat until a condition resolves.

When you find yourself designing a Workflow with more than three levels of branching, or with conditions that require understanding context rather than matching patterns, that is a signal that Agent Studio (s14) may be the better tool.

---

**Durability and Versioning**

DevRev's workflow engine has two operationally important properties:

1. **Durable State**: If a workflow run encounters an error or is paused, its state persists. You can **resume execution** after fixing the issue — no need to restart from scratch.

2. **Versioning**: When you publish a new version of a workflow, **in-flight runs continue on the old version until they complete**. The new version applies only to the next trigger. This lets you safely iterate on live workflows.

```
Workflow v1 starts executing
  ↓ You publish v2 while it's running
  ↓ The in-flight run finishes on v1
  ↓ Next trigger uses v2
```

This means you can confidently improve workflows in production without breaking active runs.

## What You Will Learn

**Before this session:**
- Automation has not been configured, or only simple rules exist
- Trigger / condition / action as a framework is not part of your mental model
- AI condition nodes are an unknown concept
- Workflow versioning behavior is unknown

**After this session:**
- You can create workflows from Settings > Workflows
- You can build a working automation using trigger, condition, and action
- You can describe the available node types (Condition / AI Condition / Parallel / Wait / Loop)
- You understand versioning (in-flight runs complete on the old version)
- You understand durability (runs can resume after errors)
- You recognize the boundary where Workflow complexity suggests switching to Agent Studio
