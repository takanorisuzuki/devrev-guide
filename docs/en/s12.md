# Getting Started with the DevRev API

> "A Personal Access Token and one curl command. That is all you need to start."

## The Problem

You want to build a custom integration with DevRev — pulling ticket data into an internal dashboard, creating issues programmatically from an external system, or automating something the marketplace does not cover. But the API documentation is extensive, authentication options are multiple, and it is unclear where to begin.

## The Solution

Start with a Personal Access Token (PAT) and a single curl command. The PAT gives you immediate API access without any OAuth setup. Once you have made your first successful call and seen the response structure, the broader API becomes navigable.

## How It Works

**Authentication Methods**

DevRev supports two authentication approaches.

| Method | When to Use | How to Obtain |
|--------|-------------|--------------|
| Personal Access Token (PAT) | Development, testing, personal scripts | Settings > API Tokens |
| OAuth 2.0 | Production Snap-ins and apps used by others | Configured during Snap-in setup |

For getting started: use a PAT. For building Snap-ins that will run on behalf of multiple users: use OAuth.

**Issuing a PAT**

Navigate to Settings > API Tokens in your DevRev workspace. Generate a new token and store it securely — DevRev will not show it again after creation.

**Internal type hierarchy (for developers)**

To use the API confidently, it helps to know how object types relate. Every object ultimately derives from a shared base type called **Atom**. Read this alongside the conceptual model in [s03](/en/s03).

```
Atom (base — shared fields such as id, created_date, modified_date)
├── Work
│   ├── Ticket
│   ├── Issue
│   ├── Task
│   └── Incident
├── Part
│   ├── Product / Capability / Feature / Enhancement (customer parts)
│   └── Code / Service / Runnable / Linkable (builder parts)
├── Org
│   ├── DevOrg
│   └── RevOrg
├── User
│   ├── DevUser
│   └── RevUser
└── Link (relationships between objects)
```

Why this matters for APIs:

- `works.list` can return Ticket, Issue, and Task together because they are all subtypes of **Work**.
- Common JSON fields (`id`, `created_date`, `modified_date`, `display_id`, …) come from **Atom**.
- The `type` field identifies the concrete object (`ticket`, `issue`, …).

See the [object model reference](/en/architecture) for link rules. If you are not ready to integrate yet, you can skip this subsection and return when you start calling APIs.

**Endpoint Structure**

DevRev's API is REST-based, served from:

```
https://api.devrev.ai/
```

An important characteristic: list operations and create operations both use POST, not GET. This is different from many REST APIs and is worth noting before you start.

| Endpoint | Method | Description |
|----------|--------|-------------|
| /works.list | POST | List tickets and issues |
| /works.create | POST | Create a ticket or issue |
| /works.update | POST | Update a ticket or issue |
| /dev-users.list | POST | List Dev Users (internal team) |
| /rev-users.list | POST | List Rev Users (customers) |
| /parts.list | POST | List Parts |

**Public API vs Beta API**

| Type | Stability | Change Policy |
|------|-----------|--------------|
| Public (GA) | Stable. Breaking changes have advance notice | Announced in release notes |
| Beta | May change without notice | No guarantees |

Use public APIs for production integrations. Beta APIs are for exploration and testing only.

**Your First API Call**

Retrieve a list of tickets:

```bash
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"]}'
```

The response is JSON. Each object contains an `id` field — this ID is used in all subsequent operations on that object (updating, linking, commenting).

## What You Will Learn

**Before this session:**
- The DevRev API authentication model is unclear
- The endpoint structure is unfamiliar
- The difference between public and beta APIs is unknown

**After this session:**
- You can issue a PAT and use it to authenticate API calls
- You understand the main endpoint families (works, dev-users, rev-users, parts)
- You can decide whether a given integration should use the public API or a beta endpoint
