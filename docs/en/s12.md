# Getting Started with the DevRev API

> "A Personal Access Token and one curl command. That is all you need to start."

## The Problem

You want to build a custom integration with DevRev — pulling ticket data into an internal dashboard, creating issues programmatically from an external system, or automating something the marketplace does not cover.

But the API documentation is extensive, authentication options are multiple, and it is unclear where to begin.

## The Solution

Start with a Personal Access Token (PAT) and a single curl command.

The PAT gives you immediate API access without any OAuth setup.

Once you have made your first successful call and seen the response structure, the broader API becomes navigable.

## How It Works

**Authentication Methods**

DevRev supports two authentication approaches.

| Method | When to Use | How to Obtain |
|--------|-------------|--------------|
| Personal Access Token (PAT) | Development, testing, personal scripts | Settings > API Tokens |
| Application Access Token (AAT) | Obtaining session tokens for customer identification in PLuG | Issued by Dev Org admin |
| System User Token (SUT) | External apps acting as a system user | Issued by Dev Org admin |
| Session Token | API calls on behalf of a specific customer (Rev User) | Obtained from STS using an AAT |

**Choosing the right token**

- **PAT** — Start here. It carries the same privileges as the token owner. Issue one from Settings > API Tokens.
- **AAT** — Used when your PLuG widget needs to identify customers (Rev Users). The AAT itself does not call APIs directly — it authenticates to the Security Token Service (STS) to obtain a Session Token.
- **SUT** — For external applications (not Snap-ins) that need to act as a service account.
- **Session Token** — A short-lived token obtained via AAT, tied to a specific customer. This is what powers Verified Identification when you pass a `session_token` to the PLuG SDK.

For getting started: use a PAT. For building Snap-ins that will run on behalf of multiple users: use OAuth (configured in the Snap-in manifest).

**Issuing a PAT**

Navigate to Settings > API Tokens in your DevRev workspace. Generate a new token and store it securely — DevRev will not show it again after creation.

**Internal type hierarchy (for developers)**

To use the API confidently, it helps to know how object types relate. Every object ultimately derives from a shared base type called **Atom**. Read this alongside the conceptual model in [s03](/en/s03).

```
Atom (base — shared fields such as id, created_date, modified_date)
├── Work
│   ├── Ticket
│   ├── Issue
│   ├── Task
│   └── Incident
├── Part
│   ├── Product / Capability / Feature / Enhancement (customer parts)
│   └── Code / Service / Runnable / Linkable (builder parts)
├── Org
│   ├── DevOrg
│   └── RevOrg
├── User
│   ├── DevUser
│   └── RevUser
└── Link (relationships between objects)
```

Why this matters for APIs:

- `works.list` can return Ticket, Issue, and Task together because they are all subtypes of **Work**.
- Common JSON fields (`id`, `created_date`, `modified_date`, `display_id`, …) come from **Atom**.
- The `type` field identifies the concrete object (`ticket`, `issue`, …).

See the [object model reference](/en/architecture) for link rules. If you are not ready to integrate yet, you can skip this subsection and return when you start calling APIs.

**Endpoint Structure**

DevRev's API is REST-based, served from:

```
https://api.devrev.ai/
```

An important characteristic: list operations and create operations both use POST, not GET. This is different from many REST APIs and is worth noting before you start.

| Endpoint | Method | Description |
|----------|--------|-------------|
| /works.list | POST | List tickets and issues |
| /works.create | POST | Create a ticket or issue |
| /works.update | POST | Update a ticket or issue |
| /dev-users.list | POST | List Dev Users (internal team) |
| /rev-users.list | POST | List Rev Users (customers) |
| /parts.list | POST | List Parts |

**Public API vs Beta API**

| Type | Stability | Change Policy |
|------|-----------|--------------|
| Public (GA) | Stable. Breaking changes have advance notice | Announced in release notes |
| Beta | May change without notice | No guarantees |

Use public APIs for production integrations. Beta APIs are for exploration and testing only.

**Your First API Call**

Retrieve a list of tickets:

```bash
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"]}'
```

The response is JSON. Each object contains an `id` field — this ID is used in all subsequent operations on that object (updating, linking, commenting).

**Common response fields (Atom base type)**

Because every object inherits from the Atom base type, responses share a consistent structure.

```json
{
  "work": [
    {
      "id": "don:core:dvrv-us-1:devo/0:ticket/123",
      "display_id": "TKT-123",
      "type": "ticket",
      "title": "Cannot log in",
      "created_date": "2025-01-15T09:00:00Z",
      "modified_date": "2025-01-15T10:30:00Z",
      "created_by": { "id": "don:identity:dvrv-us-1:devo/0:devu/1", "display_name": "Taro Yamada" },
      "owned_by": [{ "id": "don:identity:dvrv-us-1:devo/0:devu/2", "display_name": "Hanako Sato" }],
      "applies_to_part": { "id": "don:core:dvrv-us-1:devo/0:product/1", "display_name": "MyApp" },
      "stage": { "name": "in_progress" },
      "priority": "p1"
    }
  ]
}
```

- `id` — The unique identifier (DON format). Used in all API operations.
- `display_id` — Human-readable short ID (`TKT-123`, `ISS-456`, etc.).
- `type` — The concrete object type (`ticket`, `issue`, `task`, etc.).
- `created_date` / `modified_date` — ISO 8601 timestamps.

**Pagination**

DevRev uses cursor-based pagination. If the response includes a `next_cursor` field, more pages exist.

```bash
# First request
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"]}'

# If response contains next_cursor, fetch the next page
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"], "cursor": "ufhe492s..."}'
```

- When `next_cursor` is absent from the response → you have reached the last page.
- Use the `limit` parameter to control page size (a default applies if unspecified).

**TypeScript SDK**

Once you have validated your API calls with curl, move to the official TypeScript SDK for production development. It is strongly recommended over raw HTTP — it provides type safety and built-in retry logic.

```bash
npm install @devrev/typescript-sdk
```

Initialization differs by context.

```typescript
import { client } from "@devrev/typescript-sdk";

// Pattern 1: Inside a Snap-in (authenticating with service account token)
const devrevSDK = client.setup({
  endpoint: event.execution_metadata.devrev_endpoint,
  token: event.context.secrets.service_account_token,
});

// Pattern 2: When you need Beta APIs
const devrevBeta = client.setupBeta({
  endpoint: event.execution_metadata.devrev_endpoint,
  token: event.context.secrets.service_account_token,
});
```

`client.setup()` returns a client for public (GA) APIs. `client.setupBeta()` returns a client that includes beta endpoints. Use `setup()` in production; only switch to `setupBeta()` when you need an API that has not yet graduated.

SDK method names map directly to API endpoints.

| API Endpoint | SDK Method |
|-------------|-----------|
| /works.list | `devrevSDK.worksList(...)` |
| /works.create | `devrevSDK.worksCreate(...)` |
| /works.get | `devrevSDK.worksGet(...)` |
| /works.update | `devrevSDK.worksUpdate(...)` |
| /timeline-entries.create | `devrevSDK.timelineEntriesCreate(...)` |
| /dev-users.list | `devrevSDK.devUsersList(...)` |

Using the SDK over raw HTTP unifies error handling across your integrations.

## What You Will Learn

**Before this session:**
- The DevRev API authentication model is unclear
- The endpoint structure is unfamiliar
- The difference between public and beta APIs is unknown
- Pagination behavior is unknown
- The SDK exists but you have never used it

**After this session:**
- You can issue a PAT and use it to authenticate API calls
- You understand the main endpoint families (works, dev-users, rev-users, parts)
- You can decide whether a given integration should use the public API or a beta endpoint
- You can distinguish between token types (PAT / AAT / SUT / Session Token) and when each applies
- You can paginate through full result sets using `cursor` / `next_cursor`
- You know how to initialize the TypeScript SDK and when to use `setup()` vs `setupBeta()`
