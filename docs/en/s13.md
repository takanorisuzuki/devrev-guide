# Building Your First Snap-in

> "Version-controlled, testable, deployable. Snap-ins are packages, not scripts."

## The Problem

Your team has a workflow that no marketplace Snap-in covers.

Every time a ticket is created in a specific product area, you need to send a structured payload to an internal webhook.

Or you want to post a formatted daily digest of unresolved blockers to a Slack channel at 9am.

These are not exotic requirements — they are the kind of custom integration every growing team eventually needs.

## The Solution

DevRev's Snap-in framework lets you build these integrations in TypeScript using the DevRev CLI.

Snap-ins are versioned, testable locally before deployment, and distributable like npm packages.

If you can write a serverless function, you can write a Snap-in.

## How It Works

**What You Need**

| Tool | Purpose |
|------|---------|
| Node.js | Runtime |
| DevRev CLI (`@devrev/cli`) | Create, test, and deploy Snap-ins |
| TypeScript | Write the Snap-in logic |

Install the CLI:

```bash
npm install -g @devrev/cli
```

**Project Structure**

Running `devrev snap_in init` generates:

```
my-snap-in/
  ├── manifest.yaml        # Snap-in definition
  ├── src/
  │   └── functions/
  │       └── handler.ts   # Your function logic
  ├── package.json
  └── tsconfig.json
```

**manifest.yaml**

The manifest is the blueprint. It declares what your Snap-in does and what it needs.

Key sections:

- `version`: Snap-in version string
- `functions`: The function(s) this Snap-in exposes
- `event_sources`: Which DevRev events trigger each function
- `keyrings`: External credentials the Snap-in requires at install time

**Writing a Function Handler**

The handler receives an event object and executes your logic. A minimal example that posts a comment when a ticket is created:

```typescript
import { client } from "@devrev/typescript-sdk";

export async function handleEvent(event: any) {
  const devrevClient = client.setupBeta({
    endpoint: event.context.dev_oid,
    token: event.context.secrets.service_account_token,
  });

  await devrevClient.timelineEntriesCreate({
    object: event.payload.work_id,
    body: "Snap-in triggered: ticket received.",
    type: "timeline_comment",
  });
}
```

**Local Testing**

Before deploying, test locally:

```bash
devrev snap_in test
```

This runs your function against a simulated event payload. Fix issues here, not in production.

**Deployment Flow**

1. `devrev snap_in draft` — upload as a draft version
2. Activate the Snap-in in the DevRev UI
3. Validate behavior in a test workspace
4. Install in your production workspace

**Snap-kit: UI Components**

Snap-ins are not limited to background processing. You can embed custom interfaces directly inside the DevRev UI using **Snap-kit**.

Snap-kit lets you build the configuration screens and interaction modals that ship with your Snap-in. Common use cases:

| Use case | Description |
|----------|-------------|
| Setup form | Input fields for credentials and settings shown at install time |
| Action modal | Confirmation or input dialog shown when a command is triggered |
| Custom panel | Information embedded in the Ticket or Issue detail view |

Snap-kit components are declared in manifest.yaml and written in a React-like component model. Paired with a Function handler, they let you build a complete UI flow: receive input → execute logic → display result.

If your Snap-in only needs to react to events and call external APIs, Snap-kit is not required. Consider it when your Snap-in needs an interface that humans interact with — setup, configuration, or inline actions within DevRev.
