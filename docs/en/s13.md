# Building Your First Snap-in

> "Version-controlled, testable, deployable. Snap-ins are packages, not scripts."

## The Problem

Your team has a workflow that no marketplace Snap-in covers.

Every time a ticket is created in a specific product area, you need to send a structured payload to an internal webhook.

Or you want to post a formatted daily digest of unresolved blockers to a Slack channel at 9am.

These are not exotic requirements — they are the kind of custom integration every growing team eventually needs.

## The Solution

DevRev's Snap-in framework lets you build these integrations in TypeScript using the DevRev CLI.

Snap-ins are versioned, testable locally before deployment, and distributable like npm packages.

If you can write a serverless function, you can write a Snap-in.

## How It Works

### Development Environment Setup

| Tool | Purpose |
|------|---------|
| Node.js | Runtime (v18+ recommended) |
| DevRev CLI (`devrev`) | Authentication, template generation, version management, deployment |
| jq | Formatting CLI output (recommended) |
| TypeScript | Write the Snap-in logic |

**Installing the DevRev CLI**

The DevRev CLI is distributed as an OS-specific binary. Follow the [official installation guide](https://developer.devrev.ai/snapin-development/references/cli-install) for your operating system.

```bash
# Verify installation
devrev --version
# Example output: devrev version v0.4.13 ...
```

**Authentication**

Log in to DevRev via the CLI. A browser window opens for the auth flow.

```bash
devrev profiles authenticate -o <dev-org-slug> -u <your-email@example.com>
```

The `<dev-org-slug>` is visible in your DevRev URL: `https://app.devrev.ai/<slug>/...`

### Creating a Project

**Step 1: Initialize the template**

```bash
devrev snap_in_version init
```

This generates a `devrev-snaps-typescript-template` directory.

**Step 2: Create a Snap-in package**

A package is the global container for your Snap-in. The `slug` must be globally unique.

```bash
devrev snap_in_package create-one --slug my-first-snap-in | jq .
```

### Project Structure

The generated directory structure:

```
devrev-snaps-typescript-template/
  ├── manifest.yaml              # Snap-in definition (the blueprint)
  └── code/
      ├── src/
      │   ├── function-factory.ts  # Function registry (entry point)
      │   ├── functions/
      │   │   └── on_work_creation/
      │   │       ├── index.ts     # Function handler
      │   │       └── index.test.ts
      │   ├── fixtures/            # Test event payloads
      │   └── test-runner/
      ├── package.json
      └── tsconfig.json
```

Key points:
- **manifest.yaml** lives at the template root
- Source code lives inside the **code/** directory
- Every function must be registered in **`function-factory.ts`**

---

### manifest.yaml

The manifest is the blueprint. Current version is `"2"`.

```yaml
version: "2"

name: "My First Snap-in"
description: "Adds an automatic comment when a ticket is created"

service_account:
  display_name: "My Snap-in Bot"

functions:
  - name: on_work_creation
    description: "Handler for work creation events"

automations:
  - name: on_work_creation_automation
    source: devrev-webhook
    event_type: work_created
    function: on_work_creation
```

**Key manifest sections**

| Section | Description |
|---------|-------------|
| `version` | Manifest version (currently `"2"`) |
| `name` / `description` | Snap-in name and description |
| `service_account` | Display name for the Snap-in's service account |
| `functions` | Function definitions |
| `automations` | Bindings between event sources and functions |
| `event_sources` | Custom event source definitions (timer-events, etc.) |
| `keyrings` | External credentials configured at install time |
| `developer_keyrings` | Developer-provided secrets (hidden from installers) |
| `commands` | Slash command definitions (executed from Discussions) |

---

### Event Sources (Trigger Types)

Event sources define when a Snap-in fires.

| Event Source | Description | Config example |
|---|---|---|
| `devrev-webhook` | DevRev internal events (work created/updated, etc.) | `event_type: work_created` |
| `timer-events` | Scheduled execution (CRON or interval) | `config: { cron: "0 9 * * *" }` |
| `custom-webhook` | Incoming HTTP requests from external systems | A webhook URL is provisioned |

**timer-events example** (runs daily at 9am):

```yaml
event_sources:
  - name: daily_trigger
    type: timer-events
    config:
      cron: "0 9 * * *"
```

### Commands (Slash Commands)

Execute a Snap-in manually from a Discussions thread with `/command-name`.

```yaml
commands:
  - name: summarize
    namespace: my-snap-in
    description: "Post a summary of unresolved tickets"
    surfaces:
      - surface: discussions
        object_types:
          - part
    usage_hint: "[days] (e.g. 7)"
    function: summarize_handler
```

- `surfaces`: Where the command is available (`discussions`, etc.)
- `object_types`: Object types the command can be invoked on
- `namespace`: Prevents name collisions with other Snap-ins

### Function Handler

The function handler contains your business logic.

**`function-factory.ts` (function registry)**

Every function must be registered here. When you add a new function, add it to this file.

```typescript
import on_work_creation from './functions/on_work_creation';

export const functionFactory = {
  on_work_creation,
} as const;

export type FunctionFactoryType = keyof typeof functionFactory;
```

**Function handler implementation**

```typescript
import { client, publicSDK } from "@devrev/typescript-sdk";

export const run = async (events: any[]) => {
  for (const event of events) {
    // Initialize the DevRev SDK
    const devrevClient = client.setup({
      endpoint: event.execution_metadata.devrev_endpoint,
      token: event.context.secrets.service_account_token,
    });

    // Add a comment to the ticket
    await devrevClient.timelineEntriesCreate({
      object: event.payload.work_created.work.id,
      body: "Snap-in triggered: ticket received.",
      type: publicSDK.TimelineEntriesCreateRequestType.TimelineComment,
    });
  }
};

export default run;
```

**Event payload structure**

| Field | Description |
|-------|-------------|
| `event.execution_metadata.devrev_endpoint` | API endpoint |
| `event.context.secrets.service_account_token` | Short-lived token (auto-provisioned) |
| `event.payload` | Event-specific data (`work_created.work`, etc.) |
| `event.input_data.keyrings` | Configured keyring values |

### developer_keyrings (Developer Secrets)

Manage secrets like external API keys securely. These are provided by the developer and hidden from Snap-in installers.

```yaml
developer_keyrings:
  - name: external_api_key
    description: "API key for the external service"
    display_name: "External API Key"
```

Register via CLI:

```bash
echo "sk-your-secret-key" | devrev developer_keyring create snap-in-secret external_api_key
```

Access in your function via `event.input_data.keyrings.external_api_key`.

### Local Testing

Test your function locally before deploying.

```bash
cd code
npm install
npm run start:watch -- --functionName=on_work_creation --fixturePath=fixtures/on_work_creation_event.json
```

- `--functionName`: Function name as registered in `function-factory.ts`
- `--fixturePath`: Path to a test event payload JSON file

Place test payloads in `src/fixtures/`. Include keyring values in the payload to test external API integrations locally.

### Build and Deploy

**Step 1: Build & package**

```bash
cd code
npm install
npm run build     # Compile TypeScript → JavaScript
npm run package   # Generate build.tar.gz
```

**Step 2: Create a Snap-in version**

Return to the template root and create a version.

```bash
cd ..  # Back to devrev-snaps-typescript-template/ root
devrev snap_in_version create-one --path . | jq .
```

> A package can only have one unpublished version at a time. Delete the existing one with `devrev snap_in_version delete-one` if needed.

**Step 3: Draft install**

```bash
devrev snap_in draft
```

> Draft installation requires **Admin privileges**.

**Step 4: Configure and Deploy**

1. Open the Snap-in via the URL output by the CLI, or navigate to Settings > Snap-ins in the DevRev UI
2. Configure any required Keyring Connections
3. Click the **Deploy snap-in** button to activate

**Upgrading (re-deploy after code changes)**

After modifying code, upgrade with a single command:

```bash
devrev snap_in upgrade --path ./
```

**Viewing logs**

Check logs after deployment:

```bash
devrev snap_in_package logs | jq .
```

**Snap-kit: UI Components**

Snap-ins are not limited to background processing. You can embed custom interfaces directly inside the DevRev UI using **Snap-kit**.

Snap-kit lets you build the configuration screens and interaction modals that ship with your Snap-in. Common use cases:

| Use case | Description |
|----------|-------------|
| Setup form | Input fields for credentials and settings shown at install time |
| Action modal | Confirmation or input dialog shown when a command is triggered |
| Custom panel | Information embedded in the Ticket or Issue detail view |

Snap-kit components are declared in manifest.yaml and written in a React-like component model. Paired with a Function handler, they let you build a complete UI flow: receive input → execute logic → display result.

If your Snap-in only needs to react to events and call external APIs, Snap-kit is not required. Consider it when your Snap-in needs an interface that humans interact with — setup, configuration, or inline actions within DevRev.
