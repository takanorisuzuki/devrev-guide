# Building a Custom AirSync Connector (ADaaS)

> "Even external systems not covered by the marketplace can be synced — build a connector with the ADaaS SDK."

> **Prerequisite**: This session builds on [s13 (Building Your First Snap-in)](/en/s13). Complete the basic Snap-in development flow (manifest.yaml, Function handler, deployment) before tackling this session.

## What You Will Learn

**Before this session:**
- No experience building ADaaS / AirSync connectors
- Unclear on the difference between a Custom Snap-in and an AirSync Connector
- Not sure how to import large volumes of data or set up bidirectional sync

**After this session:**
- You can explain when to use an AirSync Connector (ADaaS) vs a Custom Snap-in (s13)
- You understand the project structure of an ADaaS connector built with the SDK
- You can describe the four Extraction phases and their purpose
- You understand what Initial Domain Mapping (IDM) is and how to create it
- You can explain the control protocol (DONE / PROGRESS / ERROR) and timeout management
- You can implement the Batch-then-Push pattern and the state reset pattern
- You can list the five Computer Ready connector requirements
- You know the most common failure patterns and how to avoid them

## The Problem

The marketplace covers major tools, but it does not handle internal systems or niche SaaS products.

"We need to import data from our in-house system." "We want to pull in the full history from a specific SaaS tool." "We need bidirectional sync with a system we built ourselves." Pre-built connectors cannot meet these requirements.

The Custom Snap-ins from s13 are good at real-time processing, but they are not designed for bulk historical data imports or bidirectional sync.

## The Solution

Build an AirSync Connector using the **ADaaS (Airdrop as a Service) SDK**.

The ADaaS SDK is distributed as the `@devrev/ts-adaas` npm package. You build on top of the template repository in TypeScript. The complex orchestration — Orchestrator control, state management, retries — is handled by the SDK and platform automatically.

## How It Works

### AirSync Connector vs Custom Snap-in

| | Custom Snap-in (s13) | AirSync Connector (ADaaS) |
|---|---|---|
| **SDK** | Snap-in SDK (general) | `@devrev/ts-adaas` |
| **Trigger** | Webhook, event, Cron | Events from AirSync Orchestrator |
| **Data flow** | Arbitrary | Structured ETL pipeline |
| **State management** | Implement yourself | Provided by SDK (S3 backend) |
| **Field mapping** | None | Yes (IDM + Blueprint + recipe UI) |
| **Long-running tasks** | Implement yourself | Orchestrator + control protocol |

**When to use ADaaS:**

| Use ADaaS if you need | Custom Snap-in is enough if |
|---|---|
| Periodic bulk data import | Real-time / event-driven: fetch or send small amounts of data on each event |
| Full historical data import | Data enrichment only (appending info to existing records) |
| Bidirectional sync (keep state in sync between external system ↔ DevRev) | One-directional action only (no sync state to manage) |
| Field mapping UI (recipe) | New events only, no historical data needed |

### Terminology

AirSync has accumulated several overlapping terms from its history.

| Term | Meaning |
|------|---------|
| **AirSync** | Brand name for the full data sync engine (formerly Airdrop) |
| **ADaaS** | Technical name for the SDK / service layer that accesses the AirSync platform. Still used by developers |
| **AirSync Connector** | A Snap-in built with the ADaaS SDK (synonymous with ADaaS Connector) |
| **Native Connector** | Connectors for Jira / SFDC built directly into the platform before ADaaS |

### Architecture Overview

```
[External System] ←→ [ADaaS Snap-in (Lambda)] ←→ [AirSync Platform]
                                                         │
                                                ┌────────┴────────┐
                                                │                 │
                                          [Orchestrator]    [Recipe Manager]
                                                │                 │
                                          Control protocol   Mapping management
                                                │
                                            [DevRev]
                                         (Knowledge Graph)
```

ADaaS Snap-ins run on AWS Lambda and are invoked phase by phase according to instructions from the AirSync Orchestrator.

### Domain Model

#### External side

| Concept | Description | Example |
|---------|-------------|---------|
| **External System Name** | Identifier for the tool | Jira, Salesforce, Zendesk |
| **External System Instance** | Individual tenant of the tool | yourcompany.atlassian.net |
| **External Sync Unit** | Data container within an instance | A Jira project, an SFDC org |

Two types of external systems:

- **Single Workspace** (Salesforce, Zendesk): the entire org is one External Sync Unit
- **Multiple Workspace** (Jira, Linear): each project / team is a separate External Sync Unit

#### DevRev side

| Concept | Description |
|---------|-------------|
| **Sync Unit** | Abstract unit combining an External Sync Unit + Sync Run config + Blueprint |
| **Sync Run** | One execution of a Sync Unit |
| **Blueprint** | Defines transformation functions for multiple record types |
| **Keyring** | Stores authentication credentials (shown as "Connection" in the UI) |

---

### Extraction Lifecycle (4 phases)

E2DR (External to DevRev) sync runs through four phases.

| # | Phase | Purpose | When it runs |
|---|-------|---------|-------------|
| 1 | **External Sync Units Extraction** | Fetch available sync units | Initial sync only |
| 2 | **Metadata Extraction** | Fetch external system schema | Every sync |
| 3 | **Data Extraction** | Extract record data | Every sync |
| 4 | **Attachments Extraction** | Extract attachments | Every sync |

DR2E (DevRev to External) has no Extraction phases. It consists of two Loading phases: Data Loading → Attachments Loading.

> **Note**: To sync a new External Sync Unit (e.g. a new Jira project), create a new import from the DevRev UI. Existing Sync Units do not auto-detect new workspaces.

For the full picture of sync types (Initial / Incremental / 2-way / Time-scoped) covering both Extraction and Loading, see the "Sync Types" section below.

### Getting Started

**Template and setup**

```bash
# Clone the template repository (SDK is already in package.json)
git clone https://github.com/devrev/airdrop-template
cd airdrop-template/code
npm install
```

**Project structure**

```
manifest.yaml                          # Snap-in configuration
code/
  src/
    functions/
      external-system/
        external_domain_metadata.json  # External domain metadata
        initial_domain_mapping.json    # Initial Domain Mapping (IDM)
        http-client.ts                 # External API communication
        data-normalization.ts          # Data normalization (E2DR)
        data-denormalization.ts        # Data denormalization (DR2E)
      extraction/
        workers/
          external-sync-units-extraction.ts
          metadata-extraction.ts
          data-extraction.ts
          attachments-extraction.ts
      loading/
        workers/
          load-data.ts
          load-attachments.ts
```

**Entry point example**

```typescript
import { AirdropEvent, spawn } from "@devrev/ts-adaas";
import initialDomainMapping from "../external-system/initial_domain_mapping.json";

export interface ExtractorState {
  todos: { completed: boolean };
  users: { completed: boolean };
  attachments: { completed: boolean };
}

export const initialExtractorState: ExtractorState = {
  todos: { completed: false },
  users: { completed: false },
  attachments: { completed: false },
};

const run = async (events: AirdropEvent[]) => {
  for (const event of events) {
    await spawn<ExtractorState>({
      event,
      initialState: initialExtractorState,
      initialDomainMapping,
      baseWorkerPath: __dirname,  // SDK resolves workers based on event type
    });
  }
};

export default run;
```

> **v1.13.0+**: `workerPath` is deprecated. Use `baseWorkerPath: __dirname`.

**Recommended development order**

1. Implement the HTTP client that fetches data from the external system
2. Create an initial version of External Domain Metadata (EDM) and validate it with `chef-cli` (`chef-cli` is a separate tool from the DevRev CLI — see [installation and usage](https://developer.devrev.ai/airsync/initial-domain-mapping#chef-ui-setup))
3. Prepare the Initial Domain Mapping
4. **Start with one item type (users recommended)**, confirm it works, then expand to other types (users are recommended because they have no dependencies on other entities, making them the simplest starting point)

### Initial Domain Mapping (IDM)

IDM is the file that defines how external data schemas map to DevRev native record types. Once set, it applies to all users who install this Snap-in.

```json
{
  "additional_mappings": {
    "record_type_mappings": {
      "todos": {
        "default_mapping": {
          "target_type": "issue",
          "fields": { "title": "title", "description": "body" }
        },
        "possible_record_type_mappings": []
      },
      "users": {
        "default_mapping": { "target_type": "rev_user" },
        "possible_record_type_mappings": []
      }
    },
    "format_version": "v0.2.0",
    "devrev_metadata_version": 13
  }
}
```

> **Generate your actual IDM with chef-cli.** The example above is a minimal illustration of the structure. The contents of `default_mapping` depend on your EDM (external schema definition) and the target DevRev object type.

How to create an IDM:

| Method | Command | Use case |
|--------|---------|----------|
| **Chef UI** | `chef-cli configure-mappings --env prod --idm path/to/idm.json` | Interactive. Recommended for production |
| **MCP** | AI-assisted mapping (experimental) | Fast local prototyping |

When you pass `initialDomainMapping` to `spawn()`, the SDK automatically installs it on the next stateful event (metadata extraction, etc.) when it detects a Snap-in version change.

### Object Mappers

A bidirectional lookup table that tracks the relationship between external objects and DevRev entities. Accessed via `adapter.mappers`.

```typescript
// External ID → DevRev entity lookup
await adapter.mappers.getByExternalId({
  sync_unit: adapter.event.payload.event_context.sync_unit,
  external_id: "EXT-123",
  target_type: SyncMapperRecordTargetType.WORK,
});

// DevRev entity → external ID lookup (used in Loading)
const record = await adapter.mappers.getByTargetId({
  sync_unit: adapter.event.payload.event_context.sync_unit,
  target: devrevWorkId,
});
const externalId = record.data.sync_mapper_record.external_ids[0];

// Create a new mapping
await adapter.mappers.create({
  sync_unit: adapter.event.payload.event_context.sync_unit,
  external_ids: [externalSystemId],
  targets: [devrevEntityId],
  status: SyncMapperRecordStatus.OPERATIONAL,
});
```

**When implementing 2-way sync (Loading)**

In the Loading worker (`load-data.ts`), receive change data from DevRev and write it back to the external system. The flow is:

1. Use `getByTargetId` to look up DevRev ID → external ID
2. Call the external API to update the record
3. Record `modified_date` in `external_versions` to prevent sync loops

```typescript
// Step 3: Loop prevention (without this, DevRev-originated changes get re-extracted)
await adapter.mappers.update({
  id: syncMapperRecordId,
  sync_unit: adapter.event.payload.event_context.sync_unit,
  external_versions: {
    // modified_date must come from the external API response — never a hardcoded value or new Date()
    add: [{ recipe_version: 0, modified_date: externalItem.updated_at }],
  },
});
```

See the [Load data docs](https://developer.devrev.ai/airsync/load-data) for details.

### Control Protocol and Long-Running Tasks

ADaaS Lambdas can take more than 15 minutes when processing millions of records. The SDK handles this with three mechanisms.

**1. State Persistence**

Progress state is stored in an S3 backend. After a Lambda restart, processing resumes from where it left off.

**2. Control Protocol**

At the end of each Lambda execution (normal completion or timeout), the Snap-in returns a status to the platform:

| Message | Behavior |
|---------|---------|
| **DONE** | Processing complete |
| **PROGRESS** | More work needed. Lambda is restarted with the same input |
| **ERROR** | Error occurred. Shown to the end user |

**3. Timeout Management**

The SDK sets a soft timeout at approximately 90% of the Lambda timeout. When it fires, `adapter.isTimeout` becomes `true`.

```
1. Lambda invoked
2. SDK sets soft timeout timer (90% of hard timeout)
3. Soft timeout fires → adapter.isTimeout = true
4. onTimeout callback is called
5. onTimeout MUST emit a PROGRESS event (not ERROR or DONE)
6. Platform restarts Lambda; saved state is carried over
```

> **Important**: Any `repo.push()` call after `adapter.isTimeout` is `true` is silently ignored — data is not uploaded to S3. Pushing after timeout means data is lost.

### Sync Types

Use `event_context.mode` to determine the current sync type (`SyncMode.INITIAL` / `SyncMode.INCREMENTAL`). For detailed extraction logic per mode, see [ADaaS Reference — Sync Modes](/en/adaas-reference#sync-modes-reference).

> **Note**: In AirSync, "2-way sync" refers to the DR2E direction (reverse). Enabling both E2DR (Forward) and DR2E (Reverse) achieves full bidirectional sync.

| Type | Direction | Description |
|------|-----------|-------------|
| **Initial Sync** | E2DR | First run. Extract and import all data |
| **Incremental Sync** | E2DR | Import only changes since the last sync |
| **2-way Sync (Reverse)** | DR2E | Write DevRev changes back to the external system |
| **Time-scoped Sync** | E2DR | Extract only data after a specified timestamp |

**Implementing Time-Scoped Sync**

`TIME_SCOPED_SYNCS` support is required for Computer Ready connectors. Handle two parameters from `event_context`:

| Parameter | Description |
|-----------|-------------|
| `extract_from` | Start timestamp for extraction |
| `reset_extract_from` | When `true`, re-extract from `extract_from` even during incremental sync |

```typescript
const { extract_from, reset_extract_from, mode } = adapter.event.payload.event_context;

let sinceTimestamp: string | null;
if (mode === SyncMode.INITIAL) {
  // Initial sync: use extract_from if provided, otherwise extract all data
  sinceTimestamp = extract_from ?? null;
} else if (reset_extract_from) {
  // Incremental + reset instruction: re-extract from extract_from
  sinceTimestamp = extract_from ?? null;
} else {
  // Normal incremental sync: use the last successful sync timestamp
  sinceTimestamp = adapter.state.lastSuccessfulSyncStarted;
}

// When sinceTimestamp is null, extract all data (no filter)
```

Declare the capability in your manifest:

```yaml
imports:
  - slug: airdrop-my-connector-snap-in
    capabilities:
      - TIME_SCOPED_SYNCS
```

### Implementation Patterns

**Batch-then-Push (recommended)**

Process a full page, then push. Advance the cursor only after a successful push.

```typescript
for (const item of page.items) {
  if (adapter.isTimeout) {
    // Timeout: return without pushing → next invocation resumes from same page
    return;
  }
  batch.push(normalize(item));
}

// Push only after the full page is processed
await adapter.getRepo('entity').push(batch);
state.cursor = page.nextCursor; // Advance cursor only after successful push
```

**State Reset Pattern**

Reset completion flags at the start of a new Sync Run, but preserve the high-water mark for incremental sync.

```typescript
if (event_type === EventType.StartExtractingData) {
  // Reset flags and cursor. Do not overwrite lastModified (high-water mark).
  adapter.state.entityA = { ...adapter.state.entityA, completed: false, nextPage: 0 };
  adapter.state.entityB = { ...adapter.state.entityB, completed: false, nextPage: 0 };
}
```

**Retry and Backoff**

Ensure total maximum backoff time does not exceed the Lambda timeout (5 retries × max 30s = 150s is safe; 5 × 60s = 300s is dangerous).

### Manifest Configuration

**Determining external_org_id**

`external_org_id` is used for Sync Mapper record scoping. Getting it wrong causes data to bleed between customers. It is set alongside `external_system_name` in the manifest (the identifier for the connected system — e.g. `"my-system"`).

| Condition | How to set |
|-----------|-----------|
| Instance identified by subdomain | `is_subdomain: true` (subdomain value becomes `external_org_id` automatically) |
| Otherwise | Use `organization_data` to call the API and extract a unique ID from the response |

- Must be unique and **immutable** per customer instance
- Do not use display names or any value that can be renamed
- Never hardcode a static string

**token_verification**

Specifies the endpoint used to validate credentials when a connection is created. Must be an endpoint that requires authentication (not a public endpoint that always returns 200).

**Computer Ready Connector Requirements**

A "Computer Ready" connector can be fully set up without user intervention.

| # | Requirement | Details |
|---|-------------|---------|
| 1 | TIME_SCOPED_SYNCS | Declared in manifest; `extract_from` handled correctly |
| 2 | OAuth authentication | OAuth flow, not manual API keys |
| 3 | Permission awareness | Validate connection permissions; report errors on insufficient access |
| 4 | Default configuration | Installable without required custom inputs |
| 5 | Automatic recipe submission | Full `initial_domain_mapping.json` passed to `spawn()` |

```yaml
imports:
  - slug: airdrop-my-connector-snap-in
    capabilities:
      - TIME_SCOPED_SYNCS
    allowed_connection_types:
      - my-system-oauth
```

### Common Failures

The three highest-impact failures. See [ADaaS Reference](/en/adaas-reference) for all 10 patterns.

| Failure | Impact | Fix |
|---------|--------|-----|
| `external_org_id` hardcoded or set to a mutable value | Data bleeds between customers / duplicates on rename | Fetch an immutable GUID dynamically from the external API |
| Completion flags not reset between Sync Runs | Second and subsequent syncs extract nothing | Always reset on `StartExtractingData` |
| `onTimeout` emits Error event | Recoverable timeout becomes a fatal error | Always emit a Progress event |
