# Understanding the DevRev Data Model

> "Parts, Rev Users, Dev Users. Understand these three axes and every DevRev feature clicks into place."

## The Problem

DevRev's object model is fundamentally different from Jira or Zendesk. Jira has "projects," "issues," and "epics." Zendesk has "tickets," "users," and "organizations." Trying to map these concepts directly onto DevRev leads to confusion.

The concepts of "Parts," "Rev Users," and "Dev Users" have no direct equivalent in other tools, making them the first conceptual hurdle.

## The Solution

People often describe DevRev as having "three axes," but the official mental model is three **core object families**: **Identity** (who is involved), **Parts** (what you build and deliver), and **Work** (what activity is happening).

- **Identity**: Who is involved — DevUser, RevUser, DevOrg, RevOrg, Account
- **Parts**: What you build and deliver — **RevParts** (customer-facing) and **DevParts** (internal-only)
- **Work**: What activity is happening — Conversation, Ticket, Issue, Task, Enhancement, Incident, and related types

You may still hear "three axes" in conversation; mapping that phrase to **identity / parts / work** makes it easier to align with docs and the product UI.

For the full object list, link rules, and diagrams, see the **[Object model reference](/en/architecture)**. This session builds the mental model first.

## How It Works

### Identity (who is in the system)

DevRev clearly separates the side that **builds the product** from the side that **uses it**.

| Type | Belongs to | Role |
|------|------------|------|
| DevUser | DevOrg (your company) | Builds and supports the product — engineers, PMs, support agents |
| RevUser | RevOrg (customer org) | Customers and partners who use the product |
| DevOrg | — | Your organization as a whole |
| RevOrg | — | A customer organization unit (there may be several under one Account) |
| Account | Managed by DevUsers | The customer record that groups RevOrgs |

Hierarchy at a glance:

- Your side: **DevOrg** → DevUsers
- Customer side: **Account** → RevOrg → RevUsers

### Parts — RevParts and DevParts

Parts split into what **customers see** (**RevParts**) and what **engineering maintains internally** (**DevParts**). That split is central to how DevRev is designed.

**RevParts (customer-facing product structure)**

Product → Capability → Feature → Enhancement (improvement theme, similar to an epic)

**DevParts (internal technical structure)**

Code / Service → Runnable (runnable services), Linkable (libraries and shared components)

| Part kind | Category | What it is | Visible to customers |
|-----------|----------|------------|----------------------|
| Product | RevPart | Top of the product tree | Yes |
| Capability | RevPart | A billable or deliverable capability area | Yes |
| Feature | RevPart | Smallest unit for enablement and measurement | Yes |
| Enhancement | RevPart | Improvements scoped to a Feature (epic-like) | No |
| Code / Service | DevPart | Internal services and API implementations | No |
| Runnable | DevPart | Runnable services | No |
| Linkable | DevPart | Libraries and shared components | No |

**RevParts** describe what customers buy and use; **DevParts** describe the internal stack. Customers primarily interact with the RevPart view; DevParts keep engineering work structured behind the scenes.

All Tickets and Issues attach to a Part so you can see volume and quality by product area.

### Tickets and Issues

| Object | App | Created by | Purpose |
|--------|-----|-----------|---------|
| Ticket | Support App | Rev User (customer) | Customer inquiries and bug reports |
| Issue | Build App | Dev User (internal) | Work the development team must do |

You can create an Issue from a Ticket — the core path for getting customer signal to engineering. Multiple Tickets can link to one Issue so you see how many customers hit the same problem.

### The full set of work objects

DevRev's definition of work is broad: any artifact that requires activity. Besides Tickets and Issues, you will use types such as:

| Object | Description | Typical creator |
|--------|-------------|-----------------|
| Conversation | A synchronous discussion; can escalate to a Ticket | Anyone |
| Ticket | Customer inquiry or bug report | Rev User (customer) |
| Issue | Work the dev team must complete | Dev User (internal) |
| Enhancement | Parent grouping for multiple Issues; themes and larger initiatives | Dev User (internal) |
| Task | A subdivided unit of work | Dev User (internal) |
| Incident | Outage or incident response tracked internally | Dev User (internal) |

These objects support parents, children, and links — for example Enhancement → many Issues → Tickets tied to each Issue for full traceability.

### Who sees what (visibility overview)

Rough visibility for **DevUsers** vs **RevUsers** (exact behavior can vary by role and settings):

| Object | DevUser | RevUser | Notes |
|--------|---------|---------|-------|
| Conversation | Yes | Yes (their own) | |
| Ticket | Yes | Yes (their own) | Visible in portal where applicable |
| Issue | Yes | No | Engineering-internal |
| Task | Yes | No | Engineering-internal |
| Incident | Yes | No | Engineering-internal |
| Account | Yes | No | Managed by DevUsers |
| Opportunity | Yes | No | Sales-internal |
| Product / Capability / Feature | Yes | Yes (reference) | As product information |
| Enhancement | Yes | No | Roadmap-style work |
| Code / Service | Yes | No | Internal structure |
| Article | Yes | Yes (published) | Controlled by access level |

### Mapping to other tools

If you are coming from another stack, this rough mapping can help:

| DevRev | Jira | Zendesk | Salesforce |
|--------|------|---------|------------|
| Ticket | — | Ticket | Case |
| Issue | Issue (story/bug) | — | — |
| Enhancement | Epic | — | — |
| Part (Product/Feature) | Project + component | — | — |
| Account | — | Organization | Account |
| RevUser | — | End user | Contact |
| DevUser | User | Agent | — |

### Example flow: from chat to fix

A SaaS customer reports a payment error:

1. The customer (RevUser) starts PLuG chat → a **Conversation** is created
2. Support escalates → becomes a **Ticket** tied to the **Account**
3. Engineering confirms a bug → an **Issue** is created from the Ticket (linked to the billing **Part**)
4. The engineer ships the fix → the **Issue** moves to Done
5. The linked **Ticket** updates → the customer is notified automatically

**Before this session:**

- You cannot separate Rev Users from Dev Users
