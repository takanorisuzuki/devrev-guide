# Understanding the DevRev Data Model

> "Parts, Rev Users, Dev Users. Understand these three axes and every DevRev feature clicks into place."

## The Problem

DevRev's object model is fundamentally different from Jira or Zendesk. Jira has "projects," "issues," and "epics." Zendesk has "tickets," "users," and "organizations." Trying to map these concepts directly onto DevRev leads to confusion.

The concepts of "Parts," "Rev Users," and "Dev Users" have no direct equivalent in other tools, making them the first conceptual hurdle.

## The Solution

DevRev's data model is built on three axes:

- **Parts**: Represent your product structure. They have a hierarchy of Products (top level) and Features (subdivisions)
- **Rev Users**: Customer-side users. The people who file support tickets and use your product
- **Dev Users**: Internal users. Developers, support agents, and administrators

Understanding the relationship between these three axes is the foundation for using DevRev effectively.

## How It Works

**Parts (Product Structure)**

Parts have a hierarchical structure:

- **Product**: The top-level product unit (e.g., "Our SaaS")
- **Feature**: Subdivisions under a Product (e.g., "Dashboard," "API," "Billing")

All Tickets and Issues are linked to a Part. This lets you structurally see "how many inquiries each feature receives" and "where bugs concentrate."

**Tickets and Issues**

| Object | App | Created by | Purpose |
|--------|-----|-----------|---------|
| Ticket | Support App | Rev User (customer) | Customer inquiries and bug reports |
| Issue | Build App | Dev User (internal) | Tasks the development team needs to address |

You can create an Issue from a Ticket. This is the core mechanism for "making customer voices reach the development team." Multiple Tickets can be linked to a single Issue, automatically showing "how many customers are reporting the same problem."

**The Full Work Object Hierarchy**

DevRev's official definition of a work item is "any artifact in the system that requires some activity." In addition to Tickets and Issues, the platform includes several other work object types. The following table summarizes all five types:

| Object | Description | Typical Creator |
|--------|-------------|----------------|
| Conversation | A synchronous discussion. Can be escalated to a Ticket when needed | Anyone |
| Ticket | Customer inquiry or bug report | Rev User (customer) |
| Issue | A task the development team needs to address | Dev User (internal) |
| Enhancement | A parent item grouping multiple Issues. Used for feature improvements or larger development themes | Dev User (internal) |
| Task | A subdivided unit of a larger work item | Dev User (internal) |

These objects support parent-child relationships and can be linked to each other. For example: Enhancement (a feature theme) → multiple Issues → each Issue linked to Tickets (customer reports). This creates a vertical chain of traceability across the entire platform.

**Rev Users and Dev Users**

- Rev Users: External users. They belong to an Account (company unit)
- Dev Users: Internal users. They are assigned roles and permissions

This separation makes access control between external and internal users clear and explicit.

## What You Will Learn

**Before this session:**
- You do not know what "Parts" refers to in DevRev
- The difference between Tickets and Issues is unclear
- You cannot distinguish Rev Users from Dev Users

**After this session:**
- You can explain the Parts hierarchy (Products / Features)
- You understand the Ticket-to-Issue linking flow and can explain its significance
- You can diagram the relationships between Rev Users, Dev Users, and Accounts
