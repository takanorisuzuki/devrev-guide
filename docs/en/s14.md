# Designing Autonomous Agents with Agent Studio

> "Workflows follow rules. Agents pursue goals. Know when each is the right tool."

## The Problem

You have mastered Workflows.

You know how to connect triggers, conditions, and actions.

But some problems do not fit the rule-based model.

"Read the ticket, understand whether the customer is frustrated or just curious, find the most relevant knowledge base article, and draft a response — escalating if no good answer exists." This requires judgment, not pattern matching.

A Workflow that attempts this via keyword conditions becomes unmaintainable within weeks.

## The Solution

**Agent Studio and the four foundations**

Agent Studio is one of the **four foundations of Computer by DevRev** introduced in [s01](/en/s01).

The table below relates each foundation to this session.

| Foundation | How it connects here |
|------------|----------------------|
| AirSync (external sync) | Data the agent reasons over — synced from 50+ tools |
| Memory (knowledge graph) | The agent's "memory" — see **Memory** below |
| Foundational Services (shared platform) | Search, workflows, and other services the agent uses as **tools** |
| **Agent Studio (this session)** | The environment that ties the above together to design no-code autonomous agents |

Agent Studio is DevRev's environment for designing AI-native autonomous agents.

Where Workflows execute a predefined sequence of steps, agents receive a goal and decide how to achieve it.

For Being AI-native and the **Discern / Deflect / Deduplicate** frame, see **[The Book of DevRev (Perspectives) Reference](/en/perspectives)**.

Agent Studio is where you configure these behavior patterns for your specific context.

**Agent Studio** became generally available in May 2026.

This session covers design methodology and use case development. You can now build and test agents directly in the Agent Studio UI.

## How It Works

**Workflow vs Agent Studio**

| Dimension | Workflow | Agent Studio |
|-----------|----------|--------------|
| Operating model | Rule-based (if-then-else) | Goal-based (AI decides the path) |
| Flexibility | Executes defined paths only | Adapts to the situation |
| Design input | Trigger + conditions + actions | Goal + tools + constraints |
| Best for | Predictable, repeating tasks | Judgment-required, variable tasks |
| Prerequisite | s10 | s10 + s03 (data model) + s07 (dev cycle) |

**The Three Design Elements**

Every agent you build in Agent Studio is defined by three things.

**1. Goal**

What the agent is trying to accomplish. Goals should be specific and measurable.

Poor: "Help customers" (too vague)

Better: "Read the incoming ticket, search the knowledge base for relevant articles, and post a suggested response. If no relevant article exists with a confidence score above 0.7, escalate to the Tier 2 queue."

**2. Tools**

What the agent is allowed to use to accomplish the goal. Give agents only what they need.

Examples of tools an agent might use:
- Read a ticket's content and metadata
- Search the knowledge base
- Post a comment to a ticket
- Update the ticket's priority
- Create a linked issue
- Trigger a Workflow

Fewer tools means more predictable agent behavior. Start minimal and add only when needed.

**3. Constraints**

What the agent must never do. Constraints prevent well-intentioned agents from taking actions that create problems.

Examples:
- "Do not delete any objects"
- "Do not share internal engineering notes with Rev Users"
- "Do not close a ticket without human confirmation if Priority is Blocker"

**When to Use Agent Studio vs Workflow**

| Scenario | Recommendation |
|----------|---------------|
| Tag a ticket when a keyword appears | Workflow |
| Escalate when SLA is breached | Workflow |
| Branching logic with 3 or fewer levels | Workflow |
| Understand ticket intent before routing | Agent Studio |
| Respond to novel questions using knowledge base | Agent Studio |
| Complex branching that changes frequently | Agent Studio |
| Synthesize information from multiple objects | Agent Studio |

### Skills (Tool Implementation)

In design, we talk about "tools." In Agent Studio, these are implemented as **Skills** — the specific operations an agent can perform.

**Built-in Skills examples**

| Skill | Description |
|-------|-------------|
| `get_ticket` | Retrieve ticket content and metadata |
| `get_timeline` | Retrieve ticket timeline (comment history) |
| `create_timeline_entry` | Post a comment to a ticket |
| `search_past_resolutions` | Search previously resolved tickets for similar cases |
| `search_tickets` | Search tickets by criteria |
| `search_knowledge_base` | Search KB for relevant articles |
| `update_work` | Update ticket/issue fields |
| `create_work` | Create a new ticket/issue |

Adding or removing Skills controls the agent's capability boundary. **Fewer Skills means more predictable behavior** — this is the implementation-level expression of the "minimal tools" principle above.

### Agent Lifecycle: Draft → Live

Agents move through stages before reaching production.

```
Draft (design & test)
  ↓ Testing complete, behavior verified
Live (production)
  ↓ If needed
Paused (temporarily stopped)
```

| State | Description |
|-------|-------------|
| **Draft** | Under development. Can be tested with simulated events but does not react to real tickets |
| **Live** | Active in production. Responds to real events and executes actions |
| **Paused** | Temporarily suspended. Used when issues arise or design changes are needed |

**Test thoroughly in Draft before promoting to Live.** A Live agent acts on real customer tickets — unintended behavior directly impacts customers.

---

### Guardrails (Machine-Enforceable Constraints)

The "constraints" from the design framework are implemented as **Guardrails** in Agent Studio. Guardrails are not just instructional text — they are **machine-enforceable rules**.

| Type | Example | Enforcement method |
|------|---------|-------------------|
| Action restriction | "Prohibit object deletion" | Do not grant the relevant Skill |
| Scope restriction | "Only handle tickets linked to a specific Part" | Input filters |
| Escalation condition | "Escalate if confidence below threshold" | Branch condition in the flow |
| Timeout | "Timeout if no response within 5 minutes" | Per-agent timeout setting |
| Information boundary | "Do not expose internal-scope articles to Rev Users" | Memory's permission model applies automatically |

The most reliable guardrail is **making the action physically impossible by not granting the Skill** — stronger than relying on instructional constraints alone.

### Multi-Agent Composition (Agent Chaining)

For complex use cases, rather than one agent doing everything, **chain multiple agents with separate roles**.

**Typical pattern: Agent 1 (triage) → Agent 2 (investigate & resolve)**

```
Incoming ticket
  ↓
Agent 1 (Triage & Routing)
  - Gather context
  - Deflect known issues to self-service
  - Route unknown issues to Agent 2
  ↓
Agent 2 (Investigation & Resolution)
  - Deep investigation (search past cases, analyze logs, etc.)
  - Validate and present solution
  - Escalate to humans if unresolvable
```

**Agent 1 Skills**:
- `get_ticket`, `get_timeline`, `search_past_resolutions`, `search_knowledge_base`, `create_timeline_entry`

**Agent 2 Skills**:
- All of the above, plus `update_work`, `create_work`, and deeper investigation tools

Agents pass context between each other via Snapshots (below).

---

### Snapshots (State Management)

Agents maintain a **Snapshot** per ticket. This enables delta processing when the agent is re-triggered on the same ticket.

| Property | Description |
|----------|-------------|
| Key | `{ticket_id}_{run_number}` |
| Contents | Previous analysis, actions taken, collected context |
| On re-trigger | Reads the latest snapshot and processes only the delta |
| Retention | Guideline: last 10 per ticket, 90-day TTL |

Snapshots enable "stateful conversations." Rather than one-shot processing, agents accumulate context across the entire ticket lifecycle.

---

### Memory

Agents in Agent Studio maintain memory across interactions. That memory is realized by **Memory** — one of the four foundations — an AI-native knowledge graph. It sits on top of DevRev's data model from s03 (**identity, parts, and work**), so the agent understands relationships between objects and can reason about them over time.

A support agent, for example, can remember that a particular Rev User has had three similar tickets in the past month, and adjust its response accordingly.

Memory operates within the permission model — information accessible to an agent is constrained by the same permission boundaries as human users (see [Article Access Control Reference](/en/article-access-control)).
