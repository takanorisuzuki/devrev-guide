# Designing Autonomous Agents with Agent Studio

> "Workflows follow rules. Agents pursue goals. Know when each is the right tool."

## The Problem

You have mastered Workflows. You know how to connect triggers, conditions, and actions. But some problems do not fit the rule-based model.

"Read the ticket, understand whether the customer is frustrated or just curious, find the most relevant knowledge base article, and draft a response — escalating if no good answer exists." This requires judgment, not pattern matching. A Workflow that attempts this via keyword conditions becomes unmaintainable within weeks.

## The Solution

Agent Studio is DevRev's environment for designing AI-native autonomous agents. Where Workflows execute a predefined sequence of steps, agents receive a goal and decide how to achieve it.

**Context from The Book of DevRev**

DevRev's philosophy of being AI-native goes beyond adding AI features to existing workflows. The platform is designed around three AI behaviors that agents embody:

- **Discern**: Classify and understand incoming information (understanding ticket intent, categorizing issues)
- **Deflect**: Route, recommend, and summarize before human intervention is needed
- **Deduplicate**: Identify redundancy in real time (similar tickets, duplicate issues)

Agent Studio is where these behaviors become configurable for your specific context.

**Note**: This capability is planned for general availability as **Agent Studio** around the end of May 2026. This session focuses on design methodology and use case development. When Agent Studio launches, you will have the conceptual foundation to move quickly.

## How It Works

**Workflow vs Agent Studio**

| Dimension | Workflow | Agent Studio |
|-----------|----------|--------------|
| Operating model | Rule-based (if-then-else) | Goal-based (AI decides the path) |
| Flexibility | Executes defined paths only | Adapts to the situation |
| Design input | Trigger + conditions + actions | Goal + tools + constraints |
| Best for | Predictable, repeating tasks | Judgment-required, variable tasks |
| Prerequisite | s10 | s10 + s03 (data model) + s07 (dev cycle) |

**The Three Design Elements**

Every agent you build in Agent Studio is defined by three things.

**1. Goal**

What the agent is trying to accomplish. Goals should be specific and measurable.

Poor: "Help customers" (too vague)

Better: "Read the incoming ticket, search the knowledge base for relevant articles, and post a suggested response. If no relevant article exists with a confidence score above 0.7, escalate to the Tier 2 queue."

**2. Tools**

What the agent is allowed to use to accomplish the goal. Give agents only what they need.

Examples of tools an agent might use:
- Read a ticket's content and metadata
- Search the knowledge base
- Post a comment to a ticket
- Update the ticket's priority
- Create a linked issue
- Trigger a Workflow

Fewer tools means more predictable agent behavior. Start minimal and add only when needed.

**3. Constraints**

What the agent must never do. Constraints prevent well-intentioned agents from taking actions that create problems.

Examples:
- "Do not delete any objects"
- "Do not share internal engineering notes with Rev Users"
- "Do not close a ticket without human confirmation if Priority is Blocker"

**When to Use Agent Studio vs Workflow**

| Scenario | Recommendation |
|----------|---------------|
| Tag a ticket when a keyword appears | Workflow |
| Escalate when SLA is breached | Workflow |
| Branching logic with 3 or fewer levels | Workflow |
| Understand ticket intent before routing | Agent Studio |
| Respond to novel questions using knowledge base | Agent Studio |
| Complex branching that changes frequently | Agent Studio |
| Synthesize information from multiple objects | Agent Studio |

**Memory**

Agents in Agent Studio maintain memory across interactions. This is built on DevRev's data model from s03 (**identity, parts, and work**) — the agent understands relationships between objects and can reason about them over time.

A support agent, for example, can remember that a particular Rev User has had three similar tickets in the past month, and adjust its response accordingly.
