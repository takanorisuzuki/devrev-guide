# Designing Customer Support

> "Ticket management, SLA policies, and auto-triage. Design these three and your support team transforms."

## The Problem

Tickets pile up day after day. SLA deadlines get missed. Deciding which ticket to handle first takes too long. Assigning agents is manual, and nobody can see how much load each person carries.

These problems arise when support operations are poorly designed. Simply adopting a tool does not solve them.

## The Solution

DevRev's Support App lets you establish the foundation of support operations by designing three elements:

1. **Ticket management**: Define the ticket lifecycle (creation, triage, handling, resolution, closure)
2. **SLA policies**: Set response and resolution time targets by priority
3. **Auto-triage**: Automate priority assignment and agent routing with rules

## How It Works

**Ticket Lifecycle**

DevRev tickets go through these stages:

1. **New**: Ticket just created
2. **Triaged**: Priority and assignee determined
3. **In Progress**: Being worked on
4. **Awaiting Customer**: Waiting for customer reply
5. **Resolved**: Solution provided
6. **Closed**: Closed

Designing the transition conditions between stages is the core of ticket management.

**SLA Policies**

SLA policies set time targets based on ticket severity:

| Priority | First Response | Resolution Target |
|----------|---------------|-------------------|
| Blocker | Within 1 hour | Within 4 hours |
| High | Within 4 hours | Within 24 hours |
| Medium | Within 8 hours | Within 72 hours |
| Low | Within 24 hours | Within 1 week |

SLA policies are configured in the workspace settings. Make sure to also set Business Hours, as SLA calculations are based on business hours. Incorrect settings will skew your targets.

**Auto-Triage**

DevRev can automatically apply rules when tickets are created. Examples:

- If the ticket contains "urgent" or "outage," set priority to Blocker
- If the ticket is linked to a specific Part, auto-assign to the responsible team
- If the ticket comes from a specific Account, raise priority by one level

These rules are configured in Settings. Combined with the workflow automation covered in s10, more complex conditions become possible.

**Incidents and escalation**

Separate from the day-to-day ticket lifecycle, you can track outages and major incidents with **Incident** records. A typical escalation flow looks like this:

```
Normal ticket
  ↓ SLA breach or severity decision
Create Incident (set Severity 1–4)
  ↓ Link related Tickets / Issues with is_dependent_on
Assign response team (PIA: Primary Incident Assignee)
  ↓ Investigate and respond
Status: Identified → Acknowledged → Mitigated → Resolved
```

Here, Incident severity uses a 1–4 scale (roughly: Severity 1 ≈ Blocker in the SLA table above).

**Incident stages**

| Stage | Meaning |
|-------|---------|
| Identified | The outage or incident is recognized |
| Acknowledged | The owning team has taken ownership and started response |
| Mitigated | A workaround or partial fix reduces impact |
| Resolved | Root cause is addressed and the incident is closed out |

**Escalation design tips**

- **Trigger on SLA breach**: Use a Workflow (see s10) to create an Incident automatically when a Blocker ticket exceeds its resolution target (for example, four hours).
- **Set Severity**: Incidents use Severity 1–4. Severity 1 is company-wide response; Severity 4 is minor.
- **Link related work**: Use `is_dependent_on` to tie multiple Tickets and Issues to one Incident so related customer threads stay in one place.
- **PIA (Primary Incident Assignee)**: Assign a single owner per Incident so accountability is clear.

For object shapes and link rules, see the [object model reference](/en/architecture) together with [s03](/en/s03).

## What You Will Learn

**Before this session:**
- Ticket management is done manually
- You do not know how to configure SLA policies
- Triage depends on individual judgment

**After this session:**
- You can define the ticket lifecycle and explain transition conditions for each stage
- You can configure SLA policies by priority level
- You can design basic auto-triage rules
- You can design an escalation flow that uses Incidents appropriately
