# Using Snap-ins and the Marketplace

> "Before building a custom integration, check if someone has already built it."

## The Problem

DevRev does not exist in isolation.

Your team uses Slack for communication, GitHub or GitLab for code, Jira for legacy project tracking, Salesforce for CRM.

Building these integrations from scratch takes weeks of engineering time.

When tools stay disconnected, the cost shows up in two places:

- **Context switching**: work bounces between systems, and updates get lost or delayed
- **Integration cost**: every “small” connection becomes an engineering project

## The Solution

Connecting these tools to DevRev means your team can stay in one place instead of switching contexts constantly.

The DevRev marketplace hosts pre-built Snap-ins for the most common tools.

Install one, configure the connection, and the integration is live — without writing a line of code.

Snap-ins come in two forms:

- **Marketplace Snap-ins**: Install-ready, built by DevRev or partners
- **Custom Snap-ins**: Built by your team in TypeScript (covered in s13)

Start with marketplace Snap-ins. Build custom ones only when the marketplace does not cover your use case.

## How It Works

**The Snap-ins Screen**

Navigate to Settings > Snap-ins to find three tabs.

- **All**: Browse and search the full catalog of available Snap-ins
- **Featured**: Snap-ins recommended by DevRev — a curated shortlist of what teams use most
- **Installed**: Snap-ins currently active in your workspace

**Installing a Snap-in**

1. Go to Settings > Snap-ins
2. Find the Snap-in you want in the **All** or **Featured** tab
3. Click the Snap-in card to open the detail page
4. Click **Install**
5. Complete the required setup (Keyring Connection, configuration options)
6. Once installed, the Snap-in appears in the **Installed** tab

**Managing Installed Snap-ins**

The **Installed** tab gives you a single view of everything active in your workspace. From here you can:

- Toggle a Snap-in on or off
- Change configuration settings (Keyring Connections, behavior options)
- Apply available updates
- Uninstall

**Snap-in Architecture Concepts**

Even when using marketplace Snap-ins, understanding the underlying architecture helps when something does not behave as expected.

- **Functions**: Serverless functions that execute the Snap-in's logic
- **Event Sources**: Define which events trigger the Snap-in
- **Keyring Connections**: Store and manage authentication credentials for external services

**Keyring Connections**

Every integration with an external service requires authentication. Keyring Connections is DevRev's secure credential store. When you install a Snap-in, the setup wizard prompts you to create the required Keyring Connections — typically an API key or OAuth flow.

Once configured, the Snap-in uses the stored credentials automatically. You set it up once; it runs indefinitely.

**Automatic Customer Reply: Disable It First**

Automatic Customer Reply sends a pre-written message to customers when no agent is available — for example, outside business hours.

When you install this Snap-in, **leave it disabled** until you have completed the configuration. If you enable it immediately after install, the default or unconfigured template gets sent to real customers.

Complete these steps before enabling:

1. Write the auto-reply message templates
2. Set the hours during which auto-reply is active (for example, outside business hours)
3. Define which tickets should receive auto-replies

Once the configuration is complete, enable the Snap-in.

**AirSync**

AirSync is DevRev's bidirectional sync framework, used by the Jira Snap-in and others. Unlike a one-way push, AirSync keeps records in sync in both directions.

Key AirSync behaviors:

- A change in DevRev propagates to the external tool, and vice versa
- Field mapping defines which DevRev fields correspond to which external fields
- Conflict resolution rules determine which side wins when both are updated simultaneously
- **Coexist mode**: Run Jira and DevRev in parallel during a gradual migration — teams can move at their own pace

**Connecting Jira with AirSync**

Jira is the most common AirSync target. Here is the full walkthrough.

**Before you start**: You need an Atlassian API token. Generate one from Atlassian Account Settings (id.atlassian.com) > Security > API tokens.

1. Go to Settings > Snap-ins > **All** tab, search for "Jira"
2. Click the Jira card and click **Install**
3. In the Keyring Connection setup, enter:
   - **Jira URL**: `https://yourcompany.atlassian.net`
   - **Email**: Your Atlassian account email
   - **API Token**: The token generated above
4. DevRev runs a connection test. If it passes, proceed to sync configuration
5. Configure the AirSync sync settings:
   - Which object types to sync (Jira Issue ↔ DevRev Issue, Jira Project ↔ DevRev Part)
   - Field mapping (Jira status values ↔ DevRev stage names)
   - Sync direction (bidirectional / Jira→DevRev only / DevRev→Jira only)
6. Click **Activate** to start syncing

Once active, Jira Issues are pulled into DevRev automatically. Updates made in either system propagate to the other.

**How Authentication Differs by Connector**

The setup flow varies by Snap-in. Here is a summary of the common patterns.

| Connector | Auth method | Notes |
|-----------|-------------|-------|
| Jira | API token + email | AirSync bidirectional sync available |
| GitHub | OAuth (GitHub account) | Select which repositories to connect |
| Slack | OAuth (Slack workspace) | Choose notification channels at install |
| Salesforce | OAuth (Salesforce account) | Test in Sandbox environment first |
| Zendesk | API token + subdomain | Migration wizard included |

OAuth connectors (GitHub, Slack, Salesforce) open the external service's authorization screen during install — approve access and setup is complete. API token connectors (Jira, Zendesk) require you to generate a token in the external service first.
