# Using Snap-ins and the Marketplace

> "Before building a custom integration, check if someone has already built it."

## The Problem

DevRev does not exist in isolation. Your team uses Slack for communication, GitHub or GitLab for code, Jira for legacy project tracking, Salesforce for CRM. Connecting these tools to DevRev means your team can stay in one place instead of switching contexts constantly.

Building these integrations from scratch takes weeks of engineering time.

## The Solution

The DevRev marketplace hosts pre-built Snap-ins for the most common tools. Install one, configure the connection, and the integration is live — without writing a line of code.

Snap-ins come in two forms:

- **Marketplace Snap-ins**: Install-ready, built by DevRev or partners
- **Custom Snap-ins**: Built by your team in TypeScript (covered in s13)

Start with marketplace Snap-ins. Build custom ones only when the marketplace doesn't cover your use case.

## How It Works

**What Is in the Marketplace**

The marketplace organizes Snap-ins by category.

| Category | Examples |
|----------|---------|
| Communication | Slack, Microsoft Teams |
| Source Control | GitHub, GitLab, Bitbucket |
| Project Management | Jira (via AirSync) |
| CRM | Salesforce, HubSpot |
| Monitoring | PagerDuty, Datadog |

**Snap-in Architecture Concepts**

Even when using marketplace Snap-ins, understanding the underlying architecture helps when something does not behave as expected.

- **Functions**: Serverless functions that execute the Snap-in's logic
- **Event Sources**: Define which events trigger the Snap-in
- **Keyring Connections**: Store and manage authentication credentials for external services

**Keyring Connections**

Every integration with an external service requires authentication. Keyring Connections is DevRev's secure credential store. When you install a Snap-in, the setup wizard prompts you to create the required Keyring Connections — typically an API key or OAuth flow.

Once configured, the Snap-in uses the stored credentials automatically. You set it up once; it runs indefinitely.

**AirSync**

AirSync is DevRev's bidirectional sync framework, used by the Jira Snap-in and others. Unlike a one-way push, AirSync keeps records in sync in both directions.

Key AirSync behaviors:

- A change in DevRev propagates to the external tool, and vice versa
- Field mapping defines which DevRev fields correspond to which external fields
- Conflict resolution rules determine which side wins when both are updated simultaneously

AirSync removes the need for manual updates in two places. When an engineer closes an issue in Jira, the linked DevRev issue updates automatically.

## What You Will Learn

**Before this session:**
- Snap-ins are an unfamiliar concept
- The marketplace has not been explored
- External integrations are handled through manual workarounds or custom scripts

**After this session:**

- You understand how Keyring Connections store and manage external credentials
- You can describe what AirSync does and when to use it
