# Snap-inとマーケットプレイスを活用する

> "まずはマーケットプレイスの既製Snap-inを活用する。それが最速の拡張方法。"

## 問題

DevRev単体では完結しない業務があります。

Slackでの通知、GitHubとのコード連携、Jiraからのデータ移行。

既存のツールエコシステムとDevRevをつないで使いたいですが、自前で連携を開発する工数は取れません。

## 解決策

DevRevのマーケットプレイスには、主要ツールとの連携を実現する既製のSnap-inが用意されています。

インストールするだけで連携が始まるため、開発工数ゼロで拡張できます。

Snap-inには3種類あります。

- **既製Snap-in**: マーケットプレイスからインストールして使うもの
- **カスタムSnap-in**: Webhook・イベント・Cronをトリガーに動く独自ロジックを実装するもの（s13で扱う）
- **AirSyncコネクタ**: 外部システムとの定期・双方向データ同期に特化したSnap-in。ADaaS SDKを使って開発する（s15で扱う）

カスタムSnap-inとAirSyncコネクタはどちらも自作しますが、用途が異なります。リアルタイムなイベント処理や通知はカスタムSnap-in、大量データの取り込みや双方向同期はAirSyncコネクタを使います。

まずは既製Snap-inの活用から始めるのが最速の拡張方法です。

## 仕組み

**Snap-ins画面の構成**

Settings > Snap-ins を開くと、3つのタブが表示されます。

- **All**: 利用可能な全Snap-inを検索・閲覧できます
- **Featured**: DevRevが推奨するSnap-inの一覧。よく使われるものが集まっています
- **Installed**: 現在のワークスペースにインストール済みのSnap-in一覧

**Snap-inをインストールする手順**

1. Settings > Snap-ins を開く
2. **All** タブまたは **Featured** タブで目的のSnap-inを探す
3. Snap-inのカードをクリックして詳細ページを開く
4. **Install** ボタンをクリックする
5. インストール時に必要な設定（Keyring Connectionなど）を求められるので入力する
6. インストール完了後、**Installed** タブで確認できます

**インストール済みSnap-inの管理**

**Installed** タブでは、インストール済みのSnap-inをまとめて確認できます。各Snap-inに対して以下の操作が可能です。

- 有効化・無効化のトグル
- 設定の変更（Keyring Connection、動作設定など）
- アップデートの適用
- アンインストール

**Snap-inのアーキテクチャ概念**

Snap-inは以下の要素で構成されます。

- **Functions**: Snap-inのロジックを実行するサーバーレス関数
- **Event Sources**: トリガーとなるイベントの定義
- **Keyring Connections**: 外部サービスとの認証情報の管理

**Keyring Connections**

外部サービスとの連携には認証情報が必要になります。DevRevではKeyring Connectionsという仕組みで、APIキーやOAuthトークンを安全に管理します。

Snap-inのインストール時に、必要なKeyring Connectionの設定を求められます。設定は一度行えば、Snap-inが自動的に認証情報を利用します。

---

**よく使われるSnap-in（Featured）**

FeaturedタブにはDevRevが推奨するSnap-inが並んでいます。以下は代表的なものとその用途です。

| Snap-in | 用途 |
|---------|------|
| **Slack** | SlackのメッセージからDevRevチケットを作成。エージェントへの通知配信 |
| **Jira（Computer AirSync）** | JiraのIssueとDevRevのIssueを双方向同期。Jiraから段階的に移行するための共存モードあり |
| **GitHub** | PRのマージ・コメントとDevRevのIssueを自動連携。コードと課題を一元管理 |
| **Salesforce** | SalesforceのAccount/Contactを顧客データとして同期。サポートと営業情報を統合 |
| **Zendesk** | ZendeskのチケットとDevRevのチケットを同期。Zendeskからの移行パスとして利用 |
| **HubSpot** | HubSpotのコンタクト・会社情報をRevOrg/Rev Userとして取り込む |
| **ServiceNow** | ServiceNowのITSMチケットをDevRevと連携。ITオペレーションとサポートを統合 |
| **PagerDuty** | PagerDutyのインシデントアラートをDevRevのIssueとして取り込む |
| **Automatic Customer Reply** | エージェント不在時に顧客へ自動返信。**インストール後は設定が完了するまで無効化しておく**（後述） |
| **GitHub Actions Integration** | GitHub ActionsのCI/CDステータスをDevRevのIssueに反映 |
| **SLA Status Change Slack Notifier** | SLA違反が近づいた際にSlackへ通知 |

---

**Automatic Customer Reply の注意点**

Automatic Customer Reply は、エージェントが不在の時間帯（営業時間外など）に、顧客からのチケットに対して自動返信するSnap-inです。

このSnap-inをインストールした直後は**無効化した状態のままにしてください**。有効化する前に以下の設定を完了させてください。

1. 返信に使うメッセージテンプレートを作成する
2. 自動返信を送信する時間帯（営業時間外など）を設定する
3. 対象とするチケットの条件を設定する

設定を完了してから有効化してください。インストール直後に有効化してしまうと、デフォルトのメッセージや未設定のテンプレートが顧客に送信されてしまう可能性があります。

---

**Computer AirSync フレームワーク**

Computer AirSync はDevRevが提供する双方向同期フレームワークです。Jiraなどの外部プロジェクト管理ツールとDevRevの間で、データを双方向に同期します。

Computer AirSync の特徴は以下の通りです。

- **双方向同期**: DevRevで変更した内容が外部ツールに反映され、その逆も同様
- **マッピング設定**: 外部ツールのフィールドとDevRevのフィールドの対応を定義します
- **コンフリクト解決**: 両方で同時に変更された場合のルールを設定します
- **共存モード（Coexist）**: JiraとDevRevを並行運用しながら段階的に移行します（チームのペースに合わせて移行可能）

**Jira Computer AirSync の接続手順**

Jiraは最もよく使われる Computer AirSync 対象ツールのひとつです。以下の手順で接続できます。

**前提**: AtlassianアカウントのAPIトークンが必要です。Atlassian Account Settings（id.atlassian.com）> Security > API tokens から事前に発行しておいてください。

1. Settings > Snap-ins > All タブを開き、「Jira」を検索する
2. Jira Snap-inのカードをクリックし、**Install** をクリックする
3. Keyring Connection の設定画面で以下を入力する
   - **Jira URL**: `https://yourcompany.atlassian.net`
   - **Email**: Atlassianアカウントのメールアドレス
   - **API Token**: 事前に発行したAPIトークン
4. 接続テストが実行され、成功すると次のステップへ進む
5. Computer AirSync の同期設定を行う
   - 同期するオブジェクトの種類（Jira Issue ↔ DevRev Issue, Jira Project ↔ DevRev Part など）
   - フィールドマッピング（Jiraのステータス ↔ DevRevのステージ など）
   - 同期方向（双方向 / Jira→DevRev のみ / DevRev→Jira のみ）
6. **Activate** をクリックして同期を開始する

同期が始まると、対象のJira Issueが自動的にDevRev Issueとして取り込まれます。JiraとDevRevどちらで更新しても、もう一方に反映されます。

**他のコネクタの接続方法の違い**

接続手順はコネクタによって異なります。代表的なパターンは以下の通りです。

| コネクタ | 認証方式 | 特記事項 |
|---------|---------|---------|
| Jira | APIトークン + メールアドレス | Computer AirSync（双方向同期）に対応 |
| GitHub | OAuth（GitHubアカウントで認証） | リポジトリ単位で連携対象を選択 |
| Slack | OAuth（Slackワークスペースで認証） | 通知先チャンネルをインストール時に設定 |
| Salesforce | OAuth（Salesforceアカウントで認証） | Sandbox環境での事前テストを推奨 |
| Zendesk | APIトークン + Zendeskサブドメイン | 移行ウィザードあり |

OAuthを使うコネクタ（GitHub、Slack、Salesforceなど）は、インストール時に外部サービスの認証画面が開き、DevRevへのアクセスを許可するだけで設定が完了します。APIトークン方式のコネクタ（Jira、Zendeskなど）は、外部サービス側でトークンを事前発行する必要があります。

## このセッションで学ぶこと

**始める前の状態：**
- Snap-inの概念を知らない
- マーケットプレイスの存在を知らない
- 外部ツール連携を自前で開発しようとしている

**このセッション後の状態：**
- All、Featured、Installedタブの違いを理解し、Snap-inを検索・インストールできます
- Automatic Customer ReplyなどのSnap-inを安全にインストール・設定できます
- JiraをComputer AirSyncで接続する手順を説明できます
- Keyring Connectionsの設定方法を理解しています

## 試してみる

Snap-inを1つインストールして、動作を確認してみましょう。Jiraを使っているチームはComputer AirSync の接続まで挑戦してみてください。

```
【Snap-inインストール演習】

□ 1. Settings > Snap-ins を開き、3つのタブを確認した

□ 3. インストールしたいSnap-inを1つ選んだ
□ 4. Snap-inの詳細ページで必要な権限を確認した
□ 5. Snap-inをインストールした
□ 6. Keyring Connectionを設定した（OAuth or APIトークン）
□ 7. テスト操作を行い、連携が動作することを確認した

インストールしたSnap-in: ______________
設定したConnection: ______________
確認した動作: ______________

【Jira Computer AirSync（Jiraを利用している場合）】

□ 8. AtlassianアカウントでAPIトークンを発行した
□ 9. Jira Snap-inをインストールしてKeyring Connectionを設定した
□ 10. 同期するオブジェクトとフィールドマッピングを設定した
□ 11. Computer AirSync を有効化し、Jira Issueが同期されることを確認した
```
