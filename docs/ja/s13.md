# Snap-inを自作する

> "npmパッケージのように公開・配布できる。ローカルテストから本番デプロイまでを一気通貫で学ぶ。"

## 問題

マーケットプレイスにない機能を自社で実装したいことがあります。

たとえば「チケットの優先度が変わったら社内の承認ワークフローAPIを呼び出す」「特定顧客からのチケット作成を検知して独自のスコアリングシステムを更新する」など、自社固有のロジックを含む連携が必要になります。

## 解決策

DevRev CLIとTypeScriptを使ってSnap-inを開発します。

Snap-inはnpmパッケージと同じようにバージョン管理・公開・配布できる構造になっています。

## 仕組み

### 開発環境のセットアップ

Snap-in開発には以下のツールが必要です。

| ツール | 用途 |
|--------|------|
| Node.js | ランタイム（v18 以上推奨） |
| DevRev CLI (`devrev`) | 認証、テンプレート生成、バージョン管理、デプロイ |
| jq | CLI 出力の整形（推奨） |
| TypeScript | Snap-inのロジック記述 |

**DevRev CLI のインストール**

DevRev CLI は OS 別のバイナリとして提供されます。[公式インストールガイド](https://developer.devrev.ai/snapin-development/references/cli-install)に従い、OS に合ったバイナリをダウンロードして PATH に追加してください。

```bash
# インストール確認
devrev --version
# 出力例: devrev version v0.4.13 ...
```

**認証**

CLI で DevRev にログインします。ブラウザが開き認証フローが始まります。

```bash
devrev profiles authenticate -o <dev-org-slug> -u <your-email@example.com>
```

`<dev-org-slug>` は DevRev の URL（`https://app.devrev.ai/<slug>/...`）から確認できます。

---

### プロジェクトの作成

**ステップ1: テンプレート初期化**

```bash
devrev snap_in_version init
```

`devrev-snaps-typescript-template` ディレクトリが生成されます。

**ステップ2: Snap-in パッケージの作成**

パッケージはSnap-inのグローバルな入れ物です。`slug` はグローバルで一意にする必要があります。

```bash
devrev snap_in_package create-one --slug my-first-snap-in | jq .
```

---

### プロジェクト構造

生成されるディレクトリ構造は以下の通りです。

```
devrev-snaps-typescript-template/
  ├── manifest.yaml              # Snap-inの定義ファイル（設計図）
  └── code/
      ├── src/
      │   ├── function-factory.ts  # 関数の登録（エントリポイント）
      │   ├── functions/
      │   │   └── on_work_creation/
      │   │       ├── index.ts     # Function handler
      │   │       └── index.test.ts
      │   ├── fixtures/            # テスト用イベントペイロード
      │   └── test-runner/
      ├── package.json
      └── tsconfig.json
```

重要なポイント:
- **manifest.yaml** はテンプレートのルートに置きます
- **code/** ディレクトリ内でソースを管理します
- 関数名は **`function-factory.ts`** に登録する必要があります

---

### manifest.yaml

manifest.yaml はSnap-inの設計図です。現行バージョンは `"2"` です。

```yaml
version: "2"

name: "My First Snap-in"
description: "チケット作成時に自動コメントを追加する"

service_account:
  display_name: "My Snap-in Bot"

functions:
  - name: on_work_creation
    description: "Work作成時のハンドラー"

automations:
  - name: on_work_creation_automation
    source: devrev-webhook
    event_type: work_created
    function: on_work_creation
```

**manifest.yaml の主要セクション**

| セクション | 説明 |
|-----------|------|
| `version` | manifest のバージョン（現行は `"2"`） |
| `name` / `description` | Snap-in の名前と説明 |
| `service_account` | Snap-in が使用するサービスアカウントの表示名 |
| `functions` | 実行する Function 群の定義 |
| `automations` | Event Source とFunction の紐付け |
| `event_sources` | カスタムイベントソースの定義（timer-events 等） |
| `keyrings` | インストール時に設定する外部認証情報 |
| `developer_keyrings` | 開発者が提供する秘密情報（インストーラーには非表示） |
| `commands` | スラッシュコマンドの定義（Discussions から実行） |

---

### Event Sources（トリガーの種類）

Snap-in がいつ発動するかを定義します。

| Event Source | 説明 | 使う場面 | 設定例 |
|---|---|---|---|
| `devrev-webhook` | DevRev内のイベント（Work作成・更新など） | DevRev内のオブジェクト変更に反応したいとき | `event_type: work_created` |
| `timer-events` | スケジュール実行（CRON式 or 秒間隔） | 定期バッチ処理や集計が必要なとき | `config: { cron: "0 9 * * *" }` |
| `custom-webhook` | 外部からのHTTPリクエスト | 外部システムからDevRevに情報を送り込みたいとき | Webhook URLが払い出されます |

> **Note**: 単純な「毎日の未対応チケットレポートをSlackに投稿」のようなユースケースは、マーケットプレイスの Slack Notifier Agent で対応できます。`timer-events` でカスタムSnap-inを作る前に、既製品で実現できないか確認してください。

**timer-events の例**（毎朝9時に実行）:

```yaml
event_sources:
  - name: daily_trigger
    type: timer-events
    config:
      cron: "0 9 * * *"
```

---

### Commands（スラッシュコマンド）

Discussions 画面から `/コマンド名` で Snap-in を手動実行できます。

```yaml
commands:
  - name: summarize
    namespace: my-snap-in
    description: "未対応チケットのサマリーを投稿する"
    surfaces:
      - surface: discussions
        object_types:
          - part
    usage_hint: "[days] (例: 7)"
    function: summarize_handler
```

- `surfaces`: コマンドが使える場所（`discussions` など）
- `object_types`: コマンドを実行できるオブジェクト種別
- `namespace`: 同名コマンドとの衝突を避けるための名前空間

---

### Function handler

Function handler は Snap-in のビジネスロジックを記述する場所です。

**`function-factory.ts`（関数の登録）**

すべての Function は `function-factory.ts` に登録します。新しい Function を追加したら必ずここに追記してください。

```typescript
import on_work_creation from './functions/on_work_creation';

export const functionFactory = {
  on_work_creation,
} as const;

export type FunctionFactoryType = keyof typeof functionFactory;
```

**Function handler の実装例**

```typescript
import { client, publicSDK } from "@devrev/typescript-sdk";

export const run = async (events: any[]) => {
  for (const event of events) {
    // DevRev SDK の初期化
    const devrevClient = client.setup({
      endpoint: event.execution_metadata.devrev_endpoint,
      token: event.context.secrets.service_account_token,
    });

    // チケットにコメントを追加
    await devrevClient.timelineEntriesCreate({
      object: event.payload.work_created.work.id,
      body: "Snap-inからの自動コメント",
      type: publicSDK.TimelineEntriesCreateRequestType.TimelineComment,
    });
  }
};

export default run;
```

**イベントペイロードの構造**

Function が受け取るイベントには以下のフィールドが含まれます。

| フィールド | 説明 |
|-----------|------|
| `event.execution_metadata.devrev_endpoint` | API エンドポイント |
| `event.context.secrets.service_account_token` | 短期トークン（自動発行） |
| `event.payload` | イベント固有のデータ（`work_created.work` など） |
| `event.input_data.keyrings` | 設定された Keyring の値 |

---

### developer_keyrings（開発者用シークレット）

外部 API キーなど、開発者が提供する秘密情報を安全に管理する仕組みです。インストーラー（利用者）には見えません。

```yaml
developer_keyrings:
  - name: external_api_key
    description: "外部サービスのAPIキー"
    display_name: "External API Key"
```

CLI で登録します:

```bash
echo "sk-your-secret-key" | devrev developer_keyring create snap-in-secret external_api_key
```

Function 内では `event.input_data.keyrings.external_api_key` でアクセスできます。

---

### ローカルテスト

デプロイ前に手元の環境で Function の動作を確認します。

```bash
cd code
npm install
npm run start:watch -- --functionName=on_work_creation --fixturePath=fixtures/on_work_creation_event.json
```

- `--functionName`: テストする Function 名（`function-factory.ts` に登録した名前）
- `--fixturePath`: テスト用イベントペイロードの JSON ファイル

テスト用のイベントペイロードは `src/fixtures/` ディレクトリに配置します。Keyring の値もペイロード内に含めることでローカルで API 連携のテストが可能です。

---

### ビルドとデプロイ

**ステップ1: ビルド & パッケージ**

```bash
cd code
npm install
npm run build     # TypeScript → JavaScript にコンパイル
npm run package   # build.tar.gz を生成
```

**ステップ2: Snap-in バージョンの作成**

テンプレートのルートに戻り、バージョンを作成します。

```bash
cd ..  # devrev-snaps-typescript-template/ のルートに戻る
devrev snap_in_version create-one --path . | jq .
```

> 1つのパッケージに未公開のバージョンは1つまでです。既存バージョンがある場合は `devrev snap_in_version delete-one` で削除してから再作成してください。

**ステップ3: ドラフトインストール**

```bash
devrev snap_in draft
```

> ドラフトインストールには **Admin 権限**が必要です。

**ステップ4: 設定と Deploy**

1. CLI が出力する URL、または DevRev UI の Settings > Snap-ins で Snap-in を開く
2. Keyring Connection の設定を行う（必要な場合）
3. **Deploy snap-in** ボタンをクリックして有効化する

**アップグレード（コード変更後の再デプロイ）**

コードを変更した後は、1コマンドでアップグレードできます。

```bash
devrev snap_in upgrade --path ./
```

**ログの確認**

デプロイ後のログは CLI で確認します。

```bash
devrev snap_in_package logs | jq .
```

**Snap-kit：UIコンポーネント**

Snap-inはバックグラウンド処理だけでなく、DevRevのUI内にカスタムインターフェースを埋め込むこともできます。この仕組みが**Snap-kit**です。

Snap-kitを使うと、Snap-inの設定画面やアクション実行時のモーダルを独自のコンポーネントで構築できます。たとえば、Snap-inのインストール時に認証情報を入力するフォームや、チケット詳細画面に表示するカスタムパネルなどが該当します。

| 用途 | 説明 |
|------|------|
| 設定フォーム | Snap-inインストール時の認証情報・設定値入力UI |
| アクションモーダル | コマンド実行時に表示する確認・入力ダイアログ |
| カスタムパネル | チケット・Issue詳細画面への情報埋め込み |

Snap-kitはmanifest.yamlの中で宣言し、ReactライクなコンポーネントモデルでUIを記述します。バックエンドのFunction handlerとセットで使うことで、「入力を受け取って処理を実行し、結果を表示する」一連のUIフローを完結させられます。

UIが必要ないシンプルな自動化（イベント検知→外部API呼び出し等）ではSnap-kitは不要です。チームや管理者が操作するインターフェースが必要な場合に採用を検討してください。

## このセッションで学ぶこと

**始める前の状態：**
- Snap-in開発の経験がない
- manifest.yaml の構造を知らない
- ローカルテストの方法を知らない
- DevRev CLI の使い方を知らない

**このセッション後の状態：**
- DevRev CLI でテンプレートを初期化し、パッケージを作成できます
- manifest.yaml の主要セクション（functions / automations / event_sources / commands / keyrings）を説明できます
- `function-factory.ts` に Function を登録し、handler を記述できます
- ローカルテスト（`start:watch`）でデプロイ前に動作確認できます
- ビルド → バージョン作成 → ドラフト → Deploy の一連のフローを実行できます
- Event Sources の種類（devrev-webhook / timer-events / custom-webhook）を説明できます
- Commands（スラッシュコマンド）の仕組みを理解しています
