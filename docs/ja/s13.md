# Agent Builderで自律エージェントを設計する

> "Workflowより高度なAI自律エージェントを設計する。ゴール設計とツール選択が鍵。"

## 問題

s09で学んだWorkflow自動化は、ルールベースで確実に動作する反面、「判断」が必要な場面では限界がある。たとえば「チケットの内容を読み解き、適切な対応方針を判断し、必要に応じて顧客に質問を返す」といった、状況に応じた柔軟な対応はWorkflowだけでは実現が難しい。

複雑な条件分岐をWorkflowで組もうとすると、分岐が増えすぎてメンテナンス不可能になる。

## 解決策

Agent Builderは、DevRevプラットフォーム上で目標ベースのAI自律エージェントを設計できる機能である。Workflowが「if-then-elseの連鎖」であるのに対し、Agent Builderは「ゴールを与えてAIが判断する」アプローチをとる。

**注意**: Agent Builderは2026年3月時点では一般公開前である。このセッションでは設計思想とユースケース設計を中心に扱い、UIハンズオンは含まない。Agent Builderが公開された際にすぐ活用できるよう、概念を先行して学ぶ。

## 仕組み

**WorkflowとAgent Builderの違い**

| 観点 | Workflow | Agent Builder |
|------|----------|--------------|
| 動作原理 | ルールベース（if-then-else） | ゴールベース（AIが判断） |
| 柔軟性 | 定義されたパスのみ実行 | 状況に応じて動的に判断 |
| 設計方法 | トリガー・条件・アクションを定義 | ゴール・ツール・制約を定義 |
| 適用場面 | パターンが明確な繰り返し作業 | 判断が必要な複雑な対応 |
| 前提知識 | s09のWorkflow基礎 | s09 + s03（データモデル） + s07（権限） |

**エージェント設計の3要素**

Agent Builderでエージェントを設計する際、3つの要素を定義する。

1. **ゴール（Goal）**: エージェントが達成すべき目標。「チケットを適切な担当者にアサインする」「顧客の質問に回答する」など
2. **ツール（Tools）**: エージェントが使える機能。「チケットの読み取り」「担当者の変更」「コメントの投稿」「ナレッジベースの検索」など
3. **制約（Constraints）**: エージェントがやってはいけないこと。「チケットを削除してはならない」「顧客に技術的な内部情報を開示してはならない」など

**ゴール設計のポイント**

ゴールは具体的かつ測定可能であるべきである。

悪い例: 「顧客を満足させる」（曖昧すぎる）

良い例: 「チケットの内容を分析し、ナレッジベースから関連する回答を検索して、顧客に返信案を提示する。該当する記事がない場合は適切なチームにエスカレーションする」

**ツール選択の考え方**

エージェントに与えるツールは「必要最小限」が原則。ツールが多すぎるとエージェントの判断精度が下がり、意図しない操作を実行するリスクが増える。

**Workflowとの使い分け**

すべてをAgent Builderで置き換える必要はない。使い分けの目安は以下の通り。

| シナリオ | 推奨 |
|---------|------|
| パターンが固定された定型作業 | Workflow |
| 条件分岐が3段階以内 | Workflow |
| テキスト理解が必要な判断 | Agent Builder |
| 条件分岐が複雑で頻繁に変わる | Agent Builder |
| 複数情報源を統合して回答する | Agent Builder |

**メモリの概念**

Agent Builderのエージェントは「メモリ」を持つ。過去の対話や判断の履歴をメモリに保持することで、文脈を踏まえた継続的な対応ができる。

これはs03で学んだDevRevのデータモデルの上に構築される。エージェントはParts、Rev Users、Dev Usersの関係性を理解した上で判断を行う。

## このセッションで学ぶこと

**始める前の状態：**
- Agent Builderの概念を知らない
- WorkflowとAgent Builderの違いが分からない
- エージェント設計の方法論を持っていない

**このセッション後の状態：**
- WorkflowとAgent Builderの違いを明確に説明できる
- ゴール・ツール・制約の3要素でエージェントを設計できる
- 自社ユースケースでWorkflowとAgent Builderの使い分けを判断できる
