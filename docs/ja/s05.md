# ナレッジベースとサポートポータルを構築する

> "信頼できる情報だけを登録し、顧客が自分で解決できる場を作る。"

## 問題

チケット対応の多くは、同じ質問の繰り返しである。「パスワードを忘れた」「請求書の見方がわからない」「APIの認証方法を知りたい」——これらは一度ドキュメント化すれば、顧客が自己解決できるはずだ。

しかし情報が散在していたり、古いドキュメントが混在していたりすると、むしろ混乱を招く。

顧客が「どれが正しい情報かわからない」状態になると、自己解決は進まず、同じ質問がチケットとして戻ってくる。

## 解決策

DevRevのナレッジベース（KB）は、信頼できる情報を**体系化して**管理し、サポートポータルを通じて顧客に公開する仕組みである。

KBにArticleを登録し、Collectionで整理し、Support Portalで公開する。この3ステップで顧客向けセルフサービスの基盤が完成する。

- **Article**: ナレッジの最小単位。必ずPartに紐付ける
- **Collection**: Articleを束ねるカテゴリ。階層構造を持てる
- **Support Portal**: 顧客向けの公開ページ。チケット起票とKB検索を提供する

## 仕組み

### KB Articleの登録

Articleの登録方法は3つある。

| 方法 | 説明 |
|------|------|
| リッチテキストエディタ | UIで直接執筆する。画像・コードブロック・テーブルを埋め込める |
| URLリンク | 外部でホストされているページをリンクとして登録する |
| ファイルアップロード | PDFやWordドキュメントをそのままArticleとして取り込む |

**登録時の必須フィールド**

- **Part**: このArticleが対象とするプロダクト領域（必須）。「どの機能の情報か」を明示する
- **Owned by**: 記事のオーナーとなるDev User（必須）。情報の鮮度維持に責任を持つ

ArticleのStatusは **Draft → Published → Archived** の3段階で管理する。Published状態のArticleのみSupport Portalに表示される。

**信頼できる情報だけを登録する原則**

KBは量より質である。古い情報、不確かな情報、未検証の情報を登録しないこと。顧客はKBの情報を正しいと信じて操作する。誤った情報は、何も情報がない状態より悪い結果をもたらす。

### URLクローリングによる一括登録

既存のヘルプサイトやドキュメントサイトがある場合、GUIからURLを登録すると自動でページを読み込んでナレッジとして取り込む。

**手順**

1. 左サイドバー最下部の **Setup 0/8** をクリック
2. Setup画面（`/settings/setup/platform`）の **Connect your Knowledge Base** セクションを開く
3. 「Automatically load your knowledge base」の **Add URL Source** をクリック
4. モーダルでURLを入力し、設定を行って **Start training** をクリック

**Add URL Sourceモーダルの設定項目**

| 項目 | 説明 |
|------|------|
| URL | ヘルプセンターのトップレベルドメインを入力。サブドメイン以下のページもすべて読み込まれる |
| Link type | URLまたはSitemapを選択 |
| Override Regex | 特定パターンのURLを除外したい場合に使用 |
| Which part should the articles be linked to? | 読み込んだArticleを紐付けるPartを選択 |

**注意事項**

- トップレベルドメインを指定すると、配下のすべてのサブドメインページが読み込まれる
- `robots.txt` などでAIクローラーをブロックしているサイト、またはJavaScriptで動的にレンダリングされているページは読み込めない可能性がある
- 自動取り込みされたArticleは必ずレビューしてから公開すること。不要なページや誤情報が含まれる可能性がある

### Collectionの設計

CollectionはArticleを整理するカテゴリである。Support Portalではトップレベルのナビゲーション構造となる。

**作成場所**: Settings > Support > Article Collections > +Collection

**設定項目**

- **Title**: Collectionの表示名
- **Description**: 簡単な説明
- **Parent**: 親Collectionを指定してネスト構造を作れる
- **Publish Collection**: オンにすると顧客向けに公開される

**公開のルール**

Articleがポータルに表示されるためには、以下の3つが揃っている必要がある。

1. ArticleのStatusが **Published**
2. 所属するCollectionの **Publish Collection** がオン
3. ArticleのVisibility設定が **Public** または対象ユーザーを含む設定

Collectionは階層化できる。たとえば「Getting Started > Account Setup > 2段階認証」のような深いナビゲーション構造を作れる。並び順はドラッグ＆ドロップで変更可能だ。

### サポートポータルの設定

**デフォルトURL**

```
https://support.devrev.ai/<yourcompany>
```

カスタムドメイン（例: `help.example.com`）への変更はDevRevサポートへのチケット起票が必要である。

**外観のカスタマイズ**

Settings > Plug & Portal > Portal Settings > Appearance から以下を設定できる。

| 項目 | 内容 |
|------|------|
| ロゴ・ファビコン | 自社ロゴの設定 |
| カラー・テーマ | ブランドカラーとライト/ダークテーマ |
| ヘッダー・フッター | ナビゲーションリンクの追加 |
| ヒーローセクション | トップページのキービジュアル・見出し文 |

**ポータルの機能**

- チケットの起票と進捗トラッキング
- KBの検索（構文検索 + セマンティック検索の両方に対応）

### 言語設定

DevRevは6言語をサポートしている。

| 言語 | コード |
|------|--------|
| 英語 | en |
| スペイン語 | es |
| 日本語 | ja |
| オランダ語 | nl |
| ポルトガル語 | pt |
| 中国語（簡体字） | zh-CN |

**多言語対応の仕組み**

各言語に対応した親Collectionをフォルダとして設定する。翻訳済みのArticleをその言語のCollectionに登録することで、ポータルが言語別に表示を切り替える。ポータルでは言語・デバイス別にプレビューを確認できる。

## このセッションで学ぶこと

**始める前の状態：**
- KB Articleをどこでどのように登録するかわからない
- CollectionとPartの違いが曖昧
- サポートポータルのURLや設定場所を知らない

**このセッション後の状態：**
- Part指定とオーナー設定を含むArticleを登録できる
- Collectionの階層構造を設計し、ArticleをPublishできる
- サポートポータルの外観をカスタマイズし、公開URLを確認できる

## 試してみる

自社プロダクトのKBを最小構成で立ち上げてみよう。

```
【KB立ち上げチェックリスト】

□ 1. KB ArticleをDraft状態で1本作成した（Partを必ず指定）
□ 2. Articleのオーナー（Owned by）を設定した
□ 3. Collectionを1つ作成し、Articleを所属させた
□ 4. CollectionのPublish Collectionをオンにした
□ 5. ArticleをPublishedにした
□ 6. サポートポータルのURLでArticleが表示されることを確認した
□ 7. ポータルの外観（ロゴまたはカラー）を1項目変更した
□ 8. （発展）URLクローリングのパラメータを確認し、既存ドキュメントサイトに適用できるか検討した

【設計メモ】
登録したArticle:
  タイトル: ______________
  Part: ______________
  Collection: ______________

Collectionの階層案:
  親: ______________
    └── 子: ______________
```
