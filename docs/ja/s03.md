# DevRevのデータモデルを理解する

> "Parts、Rev Users、Dev Users。この3軸を理解すれば、DevRevの全機能が腑に落ちる。"

## 問題

DevRevのオブジェクト体系は、JiraやZendeskとは根本的に異なる。Jiraには「プロジェクト」「課題」「エピック」があり、Zendeskには「チケット」「ユーザー」「組織」がある。これらの概念をそのままDevRevに当てはめようとすると混乱する。

特に「Parts」「Rev Users」「Dev Users」という3つの概念は、他のツールに直接対応するものがないため、最初の関門になりやすい。

## 解決策

DevRevのデータモデルは、よく「3つの軸」と説明されるが、公式の整理では **Identity（誰が関わるか）**、**Parts（何を作り、何を提供するか）**、**Work（どんな作業が発生しているか）** の3つのコアオブジェクト群で構成されている。

- **Identity**: 誰が関わるか（DevUser、RevUser、DevOrg、RevOrg、Account）
- **Parts**: 何を作り、何を提供するか。顧客向けの **RevParts** と、社内だけの **DevParts** に分かれる
- **Work**: どんな作業が発生しているか（Conversation、Ticket、Issue、Task、Enhancement、Incident など）

「3軸」という言い方は引き続き使われることがあるが、頭の中では **identity / parts / work** の3つに整理すると、ドキュメントやUIとも対応しやすい。

オブジェクト一覧・リンクの詳細は **[オブジェクト構造リファレンス](/ja/architecture)** でまとめている。まずはこのセッションで全体像をつかむ。

## 仕組み

DevRevの主要オブジェクトとその関係を整理する。

### Identity（登場人物）

DevRev では「製品を作る側」と「製品を使う側」を明確に分離している。

| タイプ | 所属 | 役割 |
|--------|------|------|
| DevUser | DevOrg（自社） | 製品を作る・サポートする側。エンジニア、PM、サポートエージェント |
| RevUser | RevOrg（顧客企業） | 製品を使う顧客・パートナー。エンドユーザーや顧客担当者 |
| DevOrg | — | 自社組織全体 |
| RevOrg | — | 顧客企業の組織単位（Account の下に複数存在しうる） |
| Account | DevUser が管理 | 顧客企業の台帳。RevOrg をまとめる上位概念 |

組織の階層構造:

- 自社側: **DevOrg** → DevUser
- 顧客側: **Account** → RevOrg → RevUser

### Parts — RevParts と DevParts

DevRev の Parts は、「顧客から見える側（**RevParts**）」と「開発者が作る側（**DevParts**）」の2種類に分かれる。これが DevRev の設計上の大きな特徴である。

**RevParts（顧客向け製品構成）**

Product（製品）→ Capability（機能領域）→ Feature（個別機能）→ Enhancement（改善・Epic に相当する単位）

**DevParts（内部技術構成）**

Code / Service → Runnable（実行可能サービス）、Linkable（ライブラリ等）

| Part 種別 | カテゴリ | 説明 | 顧客に見えるか |
|-----------|---------|------|----------------|
| Product | RevPart | 製品の最上位 | ○ |
| Capability | RevPart | 独立して提供・課金できる機能群 | ○ |
| Feature | RevPart | 有効化・計測の最小単位 | ○ |
| Enhancement | RevPart | Feature への改善・追加（Epic 相当） | × |
| Code / Service | DevPart | 内部マイクロサービス・API 実装 | × |
| Runnable | DevPart | 実行可能なサービス | × |
| Linkable | DevPart | ライブラリ・共通コンポーネント | × |

ポイント: **RevParts** は「顧客が購入・利用する機能」、**DevParts** は「それを動かす内部技術」。顧客には RevParts を中心に見せ、技術的な内訳は DevParts で整理する。

すべての Ticket と Issue は Part に紐付けられる。これにより「どの機能にどれだけの問い合わせが来ているか」「どの機能にバグが集中しているか」を構造的に把握できる。

### TicketとIssueの関係

| オブジェクト | 所属 | 起票者 | 目的 |
|------------|------|--------|------|
| Ticket | Support App | Rev User（顧客） | 顧客からの問い合わせ・不具合報告 |
| Issue | Build App | Dev User（社内） | 開発チームが対応すべき課題 |

TicketからIssueを作成できる。これが「顧客の声が開発チームに届く」仕組みの核心部分である。1つのIssueに複数のTicketを紐付けることもできるため、「同じ問題を報告している顧客が何人いるか」が自動的に可視化される。

### Workオブジェクトの全体像

DevRevでは、TicketとIssue以外にも複数のWorkオブジェクトが存在する。公式定義は「活動を必要とする、システム内のあらゆる成果物（artifact）」である。

| オブジェクト | 説明 | 主な起票者 |
|------------|------|----------|
| Conversation | 同期的な議論。必要に応じてTicketにエスカレート可能 | 誰でも |
| Ticket | 顧客からの問い合わせ・不具合報告 | Rev User（顧客） |
| Issue | 開発チームが対応すべき課題 | Dev User（社内） |
| Enhancement | 複数のIssueをまとめる親項目。機能改善や大きな開発テーマに使う | Dev User（社内） |
| Task | より大きな作業を細分化した単位 | Dev User（社内） |
| Incident | 障害・インシデント対応。社内の対応管理に使う | Dev User（社内） |

これらは親子関係を持ち、相互にリンクできる。たとえば「Enhancement（機能改善）→ 複数のIssue → 各IssueにリンクされたTicket（顧客報告）」という縦断的なつながりが構造上で実現される。

### 誰に何が見えるか（可視性まとめ）

次の表は、主要オブジェクトが **DevUser** と **RevUser** のどちらから見えるかの目安である（設定やロールにより変わる場合がある）。

| オブジェクト | DevUser | RevUser | 備考 |
|-------------|---------|---------|------|
| Conversation | ○ | ○（自分のもの） | |
| Ticket | ○ | ○（自分のもの） | ポータルで確認可 |
| Issue | ○ | × | 開発内部専用 |
| Task | ○ | × | 開発内部専用 |
| Incident | ○ | × | 開発内部専用 |
| Account | ○ | × | DevUser が管理 |
| Opportunity | ○ | × | 営業内部専用 |
| Product / Capability / Feature | ○ | ○（参照） | 製品情報として参照可 |
| Enhancement | ○ | × | 開発ロードマップ管理 |
| Code / Service | ○ | × | 内部技術構成 |
| Article | ○ | ○（公開分） | アクセスレベルで制御 |

### 他ツールとの対照

既存ツールの用語と対応づけたい場合の目安である。

| DevRev | Jira | Zendesk | Salesforce |
|--------|------|---------|------------|
| Ticket | — | チケット | ケース |
| Issue | チケット（Story/Bug） | — | — |
| Enhancement | Epic | — | — |
| Part (Product/Feature) | プロジェクト + コンポーネント | — | — |
| Account | — | 組織 | 取引先 |
| RevUser | — | エンドユーザー | 取引先責任者 |
| DevUser | ユーザー | エージェント | — |

### 典型フロー: 問い合わせから開発対応まで

例: SaaS 企業の顧客が決済エラーを報告した場合

1. 顧客（RevUser）が PLuG チャットで問い合わせ → **Conversation** が作成される
2. サポート担当がエスカレーション → **Ticket** に昇格（Account と紐づく）
3. 調査の結果バグと判明 → Ticket から **Issue** を作成（Part「決済機能」に紐づく）
4. エンジニアが修正を完了 → Issue のステージを Done に変更
5. 紐づいた Ticket のステータスが更新 → 顧客に自動通知

## このセッションで学ぶこと

**始める前の状態：**
- DevRevの「Parts」が何を指すかわからない
- TicketとIssueの違いが曖昧
- Rev UsersとDev Usersの区別がつかない

**このセッション後の状態：**
- Identity（Account / RevOrg / DevOrg）と RevParts / DevParts の違いを説明できる
- Ticket→Issue連携の流れを理解し、その意義を説明できる
- 主要Workオブジェクトと可視性の目安を説明できる

オブジェクトの一覧表・リンクルール・全体図は **[オブジェクト構造リファレンス](/ja/architecture)** を参照する。

## 試してみる

### Parts階層設計

自社プロダクトのParts階層を設計してみよう。

```
【Parts階層設計シート】

Product: ____________________

  ├── Feature: ______________
  │     └── 想定されるTicket例: ______________
  │
  ├── Feature: ______________
  │     └── 想定されるTicket例: ______________
  │
  ├── Feature: ______________
  │     └── 想定されるTicket例: ______________
  │
  └── Feature: ______________
        └── 想定されるTicket例: ______________

【考察】
- この構造でTicketの分類は十分か？
- Featureの粒度は適切か（細かすぎ / 粗すぎ）？
```
