# DevRevのデータモデルを理解する

> "Parts、Rev Users、Dev Users。この3軸を理解すれば、DevRevの全機能が腑に落ちる。"

## 問題

DevRevのオブジェクト体系は、JiraやZendeskとは根本的に異なる。Jiraには「プロジェクト」「課題」「エピック」があり、Zendeskには「チケット」「ユーザー」「組織」がある。これらの概念をそのままDevRevに当てはめようとすると混乱する。

特に「Parts」「Rev Users」「Dev Users」という3つの概念は、他のツールに直接対応するものがないため、最初の関門になりやすい。

## 解決策

DevRevのデータモデルは3つの軸で構成されている。

- **Parts**: プロダクトの構造を表す。Products（大分類）とFeatures（小分類）の階層を持つ
- **Rev Users**: 顧客側のユーザー。サポートチケットを起票する人、プロダクトを使う人
- **Dev Users**: 社内側のユーザー。開発者、サポート担当者、管理者

この3軸の関係性を理解することが、DevRevを使いこなす基盤になる。

## 仕組み

DevRevの主要オブジェクトとその関係を整理する。

**Parts（プロダクト構造）**

Partsは階層構造を持つ。

- **Product**: 最上位のプロダクト単位（例: 「自社SaaS」）
- **Feature**: Productの下位に属する機能単位（例: 「ダッシュボード」「API」「課金」）

すべてのTicketとIssueはPartに紐付けられる。これにより「どの機能にどれだけの問い合わせが来ているか」「どの機能にバグが集中しているか」を構造的に把握できる。

**TicketとIssueの関係**

| オブジェクト | 所属 | 起票者 | 目的 |
|------------|------|--------|------|
| Ticket | Support App | Rev User（顧客） | 顧客からの問い合わせ・不具合報告 |
| Issue | Build App | Dev User（社内） | 開発チームが対応すべき課題 |

TicketからIssueを作成できる。これが「顧客の声が開発チームに届く」仕組みの核心部分である。1つのIssueに複数のTicketを紐付けることもできるため、「同じ問題を報告している顧客が何人いるか」が自動的に可視化される。

**Rev UsersとDev Users**

- Rev Users: 外部ユーザー。Account（会社単位）に属する
- Dev Users: 内部ユーザー。ロールと権限が割り当てられる

この分離により、外部ユーザーと内部ユーザーのアクセス制御が明確になる。

## このセッションで学ぶこと

**始める前の状態：**
- DevRevの「Parts」が何を指すかわからない
- TicketとIssueの違いが曖昧
- Rev UsersとDev Usersの区別がつかない

**このセッション後の状態：**
- Parts（Products / Features）の階層構造を説明できる
- Ticket→Issue連携の流れを理解し、その意義を説明できる
- Rev Users / Dev Users / Accountsの関係を図に描ける

## 試してみる

自社プロダクトのParts階層を設計してみよう。

```
【Parts階層設計シート】

Product: ____________________

  ├── Feature: ______________
  │     └── 想定されるTicket例: ______________
  │
  ├── Feature: ______________
  │     └── 想定されるTicket例: ______________
  │
  ├── Feature: ______________
  │     └── 想定されるTicket例: ______________
  │
  └── Feature: ______________
        └── 想定されるTicket例: ______________

【考察】
- この構造でTicketの分類は十分か？
- Featureの粒度は適切か（細かすぎ / 粗すぎ）？
```
