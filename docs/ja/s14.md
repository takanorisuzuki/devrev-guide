# Agent Studioで自律エージェントを設計する

> "Workflowはルールに従う。エージェントはゴールを追う。どちらを使うべきかを判断できるようになる。"

## 問題

s10でWorkflowを習得しました。

トリガー・条件・アクションをつなぐ方法を知っています。

しかしルールベースのモデルに収まらない問題があります。

「チケットを読んで、顧客が困っているのか単に興味があるのかを判断し、最も関連性の高いナレッジベース記事を見つけ、回答を下書きする。良い回答がなければエスカレーションする」——これは判断であり、パターンマッチングではありません。

キーワード条件でWorkflowを組もうとすると、数週間でメンテナンス不可能になります。

## 解決策

Agent Studio は、[s01](/ja/s01) で紹介した **Computer by DevRev の 4 つの基盤** のうちの 1 つです。

Computer の全体像を復習しておくと理解しやすいです。

| 基盤 | このセッションとの関係 |
|------|----------------------|
| Computer AirSync（外部同期） | エージェントが参照するデータの入口。50 以上のツールから同期されたデータがエージェントの判断材料になります |
| Memory（ナレッジグラフ） | エージェントの「記憶」の実体。下記「メモリ」セクションで詳しく触れます |
| Foundational Services（共通基盤） | 検索・ワークフローなど、エージェントが「ツール」として利用するサービス群 |
| **Agent Studio（本セッション）** | 上記 3 つの基盤を束ね、ノーコードで自律エージェントを設計する環境 |

Agent Studioは、AIネイティブな自律エージェントを設計するためのDevRev上の環境です。

Workflowが定義済みのステップを実行するのに対し、エージェントはゴールを受け取り、それをどう達成するかを自ら判断します。

Being AI-native の捉え方と **Discern（見極める） / Deflect（自己解決に導く） / Deduplicate（重複を排除する）** の整理は **[The Book of DevRev（Perspectives）リファレンス](/ja/reference/perspectives)** を参照してください。

Agent Studioは、これらの動作パターンを自社のコンテキストに合わせて設定できる場所です。

**Agent Studio** は2026年5月に正式提供が開始されました。

このセッションでは設計手法とユースケース開発を学びます。Agent Studio の UI で直接エージェントを構築・テストできます。

## 仕組み

**WorkflowとAgent Studioの違い**

| 観点 | Workflow | Agent Studio |
|------|----------|--------------|
| 動作原理 | ルールベース（if-then-else） | ゴールベース（AIが判断） |
| 柔軟性 | 定義されたパスのみ実行 | 状況に応じて動的に判断 |
| 設計方法 | トリガー・条件・アクションを定義 | ゴール・ツール・制約を定義 |
| 適用場面 | パターンが明確な繰り返し作業 | 判断が必要な複雑な対応 |
| 前提知識 | s10 | s10 + s03（データモデル） + s07（開発サイクル） |

**エージェント設計の3要素**

Agent Studioで作るエージェントは、必ず3つの要素で定義します。

**1. ゴール（Goal）**

エージェントが達成しようとしていること。ゴールは具体的かつ測定可能であるべきです。

悪い例: 「顧客を満足させる」（曖昧すぎる）

良い例: 「受信チケットを読んでナレッジベースから関連記事を検索し、返信案を投稿する。信頼度スコア0.7以上の記事がなければ、Tier 2キューにエスカレーションする」

**2. ツール（Tools）**

エージェントがゴール達成に使ってよいもの。必要最小限のツールだけを与えてください。

ツールの例：
- チケットの内容とメタデータを読む
- ナレッジベースを検索する
- チケットにコメントを投稿する
- チケットの優先度を変更する
- 紐付けられたIssueを作成する
- Workflowをトリガーする

ツールが少ないほど、エージェントの動作は予測しやすくなります。最小限から始め、必要なときだけ追加してください。

**3. 制約（Constraints）**

エージェントが絶対にしてはならないこと。制約は、善意のエージェントが問題を起こす行動をとらないよう防ぎます。

例：
- 「いかなるオブジェクトも削除してはならない」
- 「内部のエンジニアリングメモをRev Userに共有してはならない」
- 「PriorityがBlockerの場合、人間の確認なしにチケットをクローズしてはならない」

**WorkflowとAgent Studioの使い分け**

| シナリオ | 推奨 |
|---------|------|
| キーワードが含まれるときにチケットにタグを付ける | Workflow |
| SLA違反時にエスカレーションする | Workflow |
| 条件分岐が3段階以内 | Workflow |
| ルーティング前にチケットの意図を理解する | Agent Studio |
| ナレッジベースを使って未知の質問に回答する | Agent Studio |
| 条件分岐が複雑で頻繁に変わる | Agent Studio |
| 複数のオブジェクトから情報を統合して回答する | Agent Studio |

### Tools / Workflows / NL Skills — 3つの層

Agent Studio では、エージェントの「行動する能力」を3つの層で構成します。

**層の概要**

| 層 | 抽象度 | 役割 |
|----|--------|------|
| **Tools（ツール）** | 低 | 単一のAPI操作。エージェントが実行できる最小単位 |
| **Workflows（ワークフロー）** | 中 | 複数のアクションを決まった順序で実行するシーケンス |
| **NL Skills（スキル）** | 高 | 自然言語の指示＋ツール/ワークフロー参照。推論しながらタスクを遂行 |

---

**Tools（ツール）— 最小単位の操作**

Tools はエージェントが実行できる最小単位のアクションです。1つのツール = 1つの操作です。

組み込み Tools の例：

| Tool | 説明 |
|------|------|
| `get_ticket` | チケットの内容・メタデータを取得 |
| `get_timeline` | チケットのタイムライン（コメント履歴）を取得 |
| `create_timeline_entry` | チケットにコメントを投稿 |
| `search_past_resolutions` | 過去の解決済みチケットから類似事例を検索 |
| `search_tickets` | 条件に合うチケットを検索 |
| `search_knowledge_base` | ナレッジベースから関連記事を検索 |
| `update_work` | チケット / Issue のフィールドを更新 |
| `create_work` | 新しいチケット / Issue を作成 |

Tools の追加・削除でエージェントの能力範囲を制御します。**与える Tools が少ないほど動作は予測しやすくなります**——前述の「ツールは最小限」原則の実装レベルの表現です。

---

**Workflows（ワークフロー）— 決まった手順のシーケンス**

Workflows は複数のアクションを決まった順序で実行する自動化シーケンスです。ノードをドラッグ&ドロップでつなぐエディタで、コードを書かずに構築できます。

Workflows をエージェントのツールとして使う場合は、**Manual Trigger** を持つワークフローを作成します。Manual Trigger のスキーマで入力パラメータを定義し、「Set skill output」ノードで実行結果をエージェントに返却します。エージェントが会話のコンテキストから適切なパラメータを推測して渡します。

例：「チケットエスカレーション」ワークフロー — 優先度を判定 → 担当チームに通知 → ステータスを更新 → Slack 通知送信。

---

**NL Skills（スキル）— 推論しながら実行する能力**

NL Skills は自然言語の指示（プロンプト）と Tools / Workflows への参照をパッケージ化した、再利用可能な能力です。サブエージェントとして独自の推論を行い、自然言語のゴールを受け取って自律的にステップを分解・実行します。

NL Skill は3つの要素で構成されます：

| 要素 | 役割 |
|------|------|
| **Skill Description** | メインエージェントに表示される短いトリガー説明（最大1024文字）。いつこのスキルを発動すべきかを記述 |
| **Plan Guidance** | スキル発動時にロードされる詳細な指示。方法論・ツール呼び出し順序・出力構造・制約を記述 |
| **Tools** | スキルが使用するワークフローや操作の参照 |

例：「顧客問い合わせ対応」スキル — ナレッジ検索 → 過去チケット確認 → 解決策の提示 → 解決不可能ならエスカレーション判断。

複数のエージェントで同一のスキルを再利用できます。一度構築すれば様々なエージェントに適用できます。

---

### エージェントのライフサイクル: Draft → Live

エージェントは段階的に本番へ移行します。

```
Draft（設計・テスト）
  ↓ テスト完了・動作確認
Live（本番稼働）
  ↓ 必要に応じて
Paused（一時停止）
```

| ステート | 説明 |
|---------|------|
| **Draft** | 設計・開発中。テストイベントで動作確認できますが、実際のチケットには反応しません |
| **Live** | 本番稼働中。実際のイベントに反応してアクションを実行します |
| **Paused** | 一時停止中。問題が発生した場合や設計変更時に使用します |

**Draft で十分なテストを行ってから Live にプロモートすること**が重要です。Live 状態のエージェントは実際の顧客チケットに対してアクションを実行するため、意図しない動作は顧客影響に直結します。

### Guardrails（マシンで強制可能な制約）

前述の「制約（Constraints）」は、Agent Studio では **Guardrails** として実装されます。Guardrails は単なる指示テキストではなく、**マシンで強制可能なルール**として設計されます。

| 種類 | 例 | 強制方法 |
|------|---|---------|
| アクション制限 | 「オブジェクトの削除を禁止」 | 該当 Skill を付与しない |
| スコープ制限 | 「特定の Part に紐づくチケットのみ対象」 | 入力フィルタで制御 |
| エスカレーション条件 | 「信頼度が閾値未満ならエスカレーション」 | フロー内の分岐条件として設定 |
| タイムアウト | 「5分以内に応答できなければタイムアウト」 | エージェントごとのタイムアウト設定 |
| 情報境界 | 「Internal scope の記事を Rev User に公開しない」 | Memory の権限モデルが自動適用 |

「制約を自然言語で書く」だけでなく、**Skill の付与範囲で物理的に不可能にする**ことが最も確実なガードレールになります。

### 複数エージェント構成（Agent Chaining）

複雑なユースケースでは、1つのエージェントですべてを処理するのではなく、**役割を分担した複数のエージェントをチェーン**する設計が有効です。

**典型パターン: Agent 1（振り分け）→ Agent 2（調査・解決）**

```
受信チケット
  ↓
Agent 1（振り分け・ルーティング）
  - コンテキスト収集
  - 既知の問題なら自己解決に導く（Deflect）
  - 未知なら Agent 2 にルーティング
  ↓
Agent 2（調査・解決）
  - 詳細調査（過去の類似事例検索、ログ分析等）
  - 検証・解決策の提示
  - 解決できなければ人間にエスカレーション
```

**Agent 1 に付与する Skills**:
- `get_ticket`, `get_timeline`, `search_past_resolutions`, `search_knowledge_base`, `create_timeline_entry`

**Agent 2 に付与する Skills**:
- 上記に加えて、`update_work`, `create_work`, より深い調査系のツール

各エージェント間はスナップショット（後述）を介してコンテキストを受け渡します。

---

### Snapshot（状態管理）

エージェントはチケットごとに **Snapshot（スナップショット）** を保持します。これにより、同じチケットに対してエージェントが再度トリガーされた際に、前回の処理結果を踏まえた差分処理が可能になります。

| 項目 | 説明 |
|------|------|
| キー | `{ticket_id}_{run_number}` |
| 保持内容 | 前回の分析結果、実行したアクション、収集したコンテキスト |
| 再トリガー時 | 最新のスナップショットを読み、差分のみ処理 |
| 保持期間 | 目安: チケットごとに直近10件、90日間 |

スナップショットにより、エージェントは「状態を持つ対話」を実現できます。1回限りの処理ではなく、チケットのライフサイクル全体を通じてコンテキストを蓄積します。

---

### メモリ

Agent Studioのエージェントは、やり取りをまたいでメモリを保持します。このメモリの実体が、Computer の 4 基盤の 1 つである **Memory**（AI ネイティブなナレッジグラフ）です。s03で学んだDevRevのデータモデル（**identity / parts / work**）の上に構築されており、エージェントはオブジェクト間の関係性を理解し、時間をかけて推論できます。

たとえばサポートエージェントは、ある Rev User が過去1ヶ月に3件の類似チケットを作成していたことを記憶し、それを踏まえた回答をすることができます。

Memory は権限モデルに従って動作するため、エージェントが参照できる情報は人間のユーザーと同じ権限境界に制約されます（[Article アクセス制御リファレンス](/ja/reference/article-access-control) 参照）。

## このセッションで学ぶこと

**始める前の状態：**
- Agent Studioの概念を知らない
- WorkflowとAgent Studioの違いが分からない
- エージェント設計の方法論を持っていない
- エージェントのライフサイクルや状態管理を知らない

**このセッション後の状態：**
- WorkflowとAgent Studioの違いを明確に説明できます
- ゴール・ツール・制約のフレームワークでエージェントを設計できます
- Tools（組み込み操作）・Workflows（シーケンス）・NL Skills（推論実行）の3層の違いを説明できます
- Draft → Live のライフサイクルを理解し、安全にプロモートする手順を説明できます
- Guardrails をマシンで強制可能な制約として設計できます
- 複数エージェント構成（Agent Chaining）の設計パターンを説明できます
- Snapshot による状態管理の仕組みを理解しています
- 自社ユースケースでWorkflowとAgent Studioの使い分けを判断できます
- Agent Studio が Computer by DevRev の 4 基盤のどこに位置するかを説明できます
