# Memory vs fetch型AI：正確性とコスト

このページは DevRev CTO office の公開記事（LinkedIn）2本を、ガイド内で参照しやすいように要点だけまとめたリファレンスである。テーマは「“つながっているAI”でも、データモデルを読めないと推測してしまう」「セッションごとにschema探索が必要だとトークン/時間が複利で増える」。

ポイントは「モデルの賢さ」より **取得アーキテクチャ** にある。

- **fetch型（API/MCPで“つながっている”AI）**: セッションのたびに schema を探索し、生データを読み込み、関係をモデル内で再構築しがち
- **共有Memoryを持つ設計**: 型付きリレーション（edge）を権威ソースから読み、構造化データは決定的に（SQL/CPU）処理して、必要な結果だけ返す

## 1) それっぽく正しいが、実は間違う（推測で作られた構造）

権威あるリレーション（親子・リンク）を読めない場合、AIはIDの規則性や名称などの“パターン”から構造を推測しがちである。見た目は整っていても間違う可能性があり、検証の負担が人間側に寄る。

実務上の示唆:

- 文字列パターンではなく、SoR上の**型付きリレーション**（edge/join）から構造を組み立てる
- 意思決定に使うなら「**どう構築したか**」を必ず要求し、再現できる根拠（辿った関係）を残す

## 2) 毎朝ゼロから始める：トークン/時間が複利で増える理由

セッションごとに schema を探索し、さらに生データをコンテキストに載せて結合・抽出する方式は、コストが**元データ量**や探索の繰り返しに引きずられやすい。一方、永続的で権限対応のMemory層（例：ナレッジグラフ + 決定的なクエリ）を持つ場合、コストはより**結果セット**に比例しやすい。

現場でコストを押し上げる要因（典型）:

- **schema探索**: プロジェクト一覧、カスタムフィールドID、join key、リンクの意味（親子/依存/対応付け）の発見
- 探索・サンプリングでツールが返す **大きなペイロード**
- セッションごとに「地図」を持ち直せず、同じ探索を **繰り返す**

記事内で示されている比較（目安・同一ビジネスクエリの反復）:

- ある比較では、1回あたり **約3.2M tokens / 約8分50秒** に対し、**約157k tokens / 約1分36秒**（トークン約95%減、約5.5倍高速）
- 別の例では、（修正後の権威ある実行で）**616,843 tokens** に対し **169,986 tokens**（約72%減）

実務上の示唆:

- schema理解と関係マッピングを、モデル内推論ではなく**決定的な層（CPU処理）**として扱う
- 取得は「必要な行だけ返す」設計にし、根拠を一緒に返す

## 差別化ポイント（Computer + Memoryで何が変わるか）

この2本の記事が強調しているのは「推測しないで“読む”」「根拠を残す」ためのレイヤー設計である。

- **権威ある関係を辿る**: パーツ階層や issue→part / ticket→account の対応は、明示された edge から読む（ID規則性から推測しない）
- **構造化は決定的に処理**: join/filter はモデル内で頑張らず、決定的な問い合わせで結果セットだけ返す
- **意味検索は内容を詰め込まない**: 記事本文を全部コンテキストに載せず、索引＋ランキングで絞り込む
- **根拠ファースト**: どのリンク（edge）を辿ったかが説明可能で、検証がしやすい
