# カスタマーサポートを設計する

> "チケット管理・SLA・自動トリアージ。この3つを設計すれば、サポートチームの動き方が変わる。"

## 問題

チケットが日々積み上がっていく。

SLAの期限を過ぎてしまうものが増える。

どのチケットを先に対応すべきかの判断に時間がかかる。

担当者のアサインが手動で、誰がどれだけの負荷を抱えているかが見えない。

こうした問題は、サポート運用の設計が不十分な場合に起きる。

ツールを導入しただけでは解決しない。

## 解決策

DevRevのSupport Appでは、3つの要素を設計することでサポート運用の基盤が整う。

1. **チケット管理**: チケットのライフサイクル（作成→トリアージ→対応→解決→クローズ）を定義する
2. **SLAポリシー**: 優先度別の応答時間・解決時間の目標を設定する
3. **自動トリアージ**: ルールベースで優先度の判定と担当者のアサインを自動化する

## 仕組み

**チケットのライフサイクル**

DevRevのチケットは以下のステージを経る。

1. **New**: チケットが作成された状態
2. **Triaged**: 優先度と担当者が決まった状態
3. **In Progress**: 対応中
4. **Awaiting Customer**: 顧客の返答待ち
5. **Resolved**: 解決済み
6. **Closed**: クローズ

各ステージ間の遷移条件を設計することが、チケット管理の要になる。

**SLAポリシー**

SLAポリシーでは、チケットの優先度（Severity）に応じた時間目標を設定する。

| 優先度 | 初回応答時間 | 解決目標時間 |
|--------|------------|------------|
| Blocker | 1時間以内 | 4時間以内 |
| High | 4時間以内 | 24時間以内 |
| Medium | 8時間以内 | 72時間以内 |
| Low | 24時間以内 | 1週間以内 |

SLAポリシーはワークスペースの設定画面から定義する。営業時間（Business Hours）の設定も忘れずに行う。SLAの計算は営業時間ベースで行われるため、ここを正しく設定しないと目標時間がずれる。

**自動トリアージ**

DevRevでは、チケット作成時に自動でルールを適用できる。基本的なトリアージルールの例は以下の通り。

- キーワードに「緊急」「障害」が含まれる場合、優先度をBlockerに設定
- 特定のPartに紐付くチケットは、該当チームに自動アサイン
- 特定のAccountからのチケットは優先度を1段階引き上げ

これらのルールはSettings画面から設定できる。s10で学ぶワークフロー自動化と組み合わせることで、より高度な条件分岐も実現できる。

**障害・インシデント対応とエスカレーション**

チケットのライフサイクルとは別に、障害やインシデントを **Incident** オブジェクトで追跡できる。重大な問題が発生した場合のエスカレーションフローは以下の流れになる。

```
通常チケット
  ↓ SLA 違反または重大度判定
Incident 作成（Severity 1-4 を設定）
  ↓ is_dependent_on で関連 Ticket/Issue を紐づけ
対応チーム割り当て（PIA: Primary Incident Assignee）
  ↓ 調査・対応
ステータス遷移: Identified → Acknowledged → Mitigated → Resolved
```

ここでは Incident の重大度（Severity）を 1-4 の尺度で表す（目安として Severity 1 は SLA 表の Blocker に相当する）。

**Incident のステージ**

| ステージ | 意味 |
|---------|------|
| Identified | 障害が認識された |
| Acknowledged | 対応チームが確認し、対応を開始した |
| Mitigated | 影響を緩和する暫定対策を実施した |
| Resolved | 根本対応が完了した |

**エスカレーションの設計ポイント**

- **SLA 違反をトリガーにする**: Blocker チケットの解決目標（4 時間）を超過した場合に自動で Incident を作成するワークフローを設定する（s10 のワークフロー機能を利用）。
- **Severity 設定**: Incident には Severity（1-4）を設定する。Severity 1 は全社対応レベル、Severity 4 は軽微な障害。
- **関連オブジェクトの紐づけ**: Incident は `is_dependent_on`（オブジェクト間の依存リンク）で複数の Ticket や Issue を紐づける。これにより、1 つの障害に関連する複数のチケットを一元管理できる。
- **PIA（Primary Incident Assignee）**: Incident ごとに主担当者を設定し、対応の責任を明確にする。

Incident の詳細なオブジェクト構造とリンクルールは [オブジェクト構造リファレンス](/ja/architecture) を参照。s03 のデータモデルと合わせて読むと理解しやすい。

## このセッションで学ぶこと

**始める前の状態：**
- チケット管理は手動で行っている
- SLAの設定方法を知らない
- トリアージが属人化している

**このセッション後の状態：**
- チケットのライフサイクルを定義し、各ステージの遷移条件を説明できる
- SLAポリシーを優先度別に設定できる
- 基本的な自動トリアージルールを設計できる
- Incident を使ったエスカレーションフローを設計できる

## 試してみる

テストチケットを作成してSLAが正しく動作するか確認しよう。

```
【サポート設計チェックリスト】

□ 1. SLAポリシーを設定した（優先度4段階）
□ 2. 営業時間（Business Hours）を設定した
□ 3. テストチケットを1件作成した
□ 4. SLAタイマーが正しくカウントされていることを確認した
□ 5. 優先度を変更し、SLAが再計算されることを確認した
□ 6. 自動トリアージルールを1つ設定した
□ 7. ルールに合致するチケットを作成し、自動適用を確認した
□ 8. Incident オブジェクトを 1 件作成し、関連 Ticket を紐づけた
□ 9. Severity と PIA を設定した
```

次のセッションでは、ナレッジベースの構築とサポートポータルの設定を学ぶ。
