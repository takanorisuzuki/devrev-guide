# PLuGとObserve Appで顧客を理解する

> "ライブチャットで顧客と話し、セッションリプレイで行動を見る。2つを組み合わせて顧客を深く理解する。"

## 問題

Webサイトやアプリ上でユーザーがどんな行動をとっているかが見えない。

ユーザーが問い合わせてきたとき、「どの画面で何をしていたか」を把握するのに時間がかかる。

ユーザー行動の分析にはHotjarやFullStoryを別途導入し、サポートツールとは切り離された環境で確認する必要がある（日本ではPtengine・UserHeat・SiTestなどが同カテゴリに当たる）。

この分断が、顧客理解のボトルネックになっている。

## 解決策

DevRevには顧客理解のための2つの機能がセットで提供されている。

- **PLuG**: Webサイトに埋め込むライブチャットウィジェット。顧客との接点を直接プラットフォーム内に取り込む
- **Observe App**: セッションリプレイ（ユーザーの画面操作を録画）とファネル分析（コンバージョン経路の分析）を提供する

PLuGで「顧客が何を言っているか」を把握し、Observe Appで「顧客が何をしているか」を把握する。

この2つを組み合わせることで、サポートチケットの背景を深く理解できる。

## 仕組み

### PLuGの導入

PLuGは **PLuG SDK** をWebサイトに組み込むことで利用できる。他のチャットウィジェット（IntercomやZendeskのウィジェットなど）と同様に、数行のコードを追加するだけだ。

**ステップ1: スクリプトの読み込み**

`<head>` または `</body>` 直前に以下のスクリプトタグを追加する。

```html
<script src="https://plug-platform.devrev.ai/static/plug.js" crossorigin="anonymous" async></script>
```

**ステップ2: SDKの初期化**

ページ読み込み後に `window.plugSDK.init()` を呼び出す。`app_id` は DevRev の Settings > PLuG Settings > **Your Unique App ID** から取得する。

```javascript
// 基本的な初期化
window.plugSDK.init({
  app_id: "YOUR_APP_ID",
});
```

Reactの場合は `useEffect` 内で呼び出す。

```javascript
useEffect(() => {
  window.plugSDK.init({
    app_id: "YOUR_APP_ID",
  });
}, []);
```

初期化が完了すると、右下にチャットランチャーが表示される。

**ステップ3: セッション録画の有効化（推奨）**

Observe App と連携するには、セッション録画を明示的に有効化する。

```javascript
window.plugSDK.init({
  app_id: "YOUR_APP_ID",
  enable_session_recording: true,
  session_recording_key: "YOUR_SESSION_RECORDING_KEY",
});
```

`session_recording_key` は Settings > PLuG Settings から取得する。

**`init()` の主要パラメータ**

| パラメータ | 説明 | 型 |
|-----------|------|---|
| `app_id` | PLuG の一意識別子（**必須**） | string |
| `enable_session_recording` | セッション録画を有効化する | boolean |
| `session_recording_key` | セッション録画アカウントの識別キー | string |
| `session_token` | ユーザー識別用トークン（後述） | string |
| `session_id` | クロスドメイン追跡用のセッションID | string |
| `tab_id` | クロスドメイン追跡用のタブID | string |
| `enable_default_launcher` | デフォルトのランチャーアイコンを表示するか | boolean |
| `disable_plug_chat_window` | チャットウィンドウを無効化する | boolean |
| `primary_color` | メインのブランドカラー（hex: `#007bff`） | string |
| `primary_text_color` | ランチャーやボタンの文字色（hex: `#f0f0f0`） | string |

**ユーザー識別（Verified Identification）**

匿名ユーザーの場合でもPLuGは動作するが、RevUser と紐付けることで、過去の問い合わせ履歴やアカウント情報を参照できるようになる。

1. Settings > PLuG Tokens で **Application Access Token (AAT)** を発行する
2. バックエンドで AAT を使い、DevRev API（`auth-tokens.create`）からセッショントークンを取得する
3. フロントエンドで `session_token` を `init()` に渡す

```javascript
window.plugSDK.init({
  app_id: "YOUR_APP_ID",
  session_token: "SESSION_TOKEN_FROM_BACKEND",
});
```

これによりチャット内容や起票されたチケットが特定の RevUser に紐付き、サポート担当者がコンテキストを即座に把握できる。

**ログアウト時の処理**

ユーザーがアプリからログアウトした際は `shutdown()` を呼び出す。会話データがクリアされ、セッション録画も停止する。

```javascript
await window.plugSDK.shutdown();
```

再度ログインした場合は `init()` を再実行すればよい。

**PLuG で使える機能**

- **AIチャットウィジェット**: 顧客からのリアルタイムメッセージとAIによる自動回答
- **ナレッジベース検索**: FAQや記事の自動表示
- **Search Agent**: `initSearchAgent()` でウィジェット内に検索専用UIを提供
- **チケット作成**: チャットから直接チケットを起票
- **Nudges**: ウィジェットからのアウトバウンドメッセージ（プロアクティブな案内やアクション促進）

PLuGの設定はDevRevのSettings > PLuG Settings から行う。デザインのカスタマイズ（ランチャーのロゴ・位置・色・表示ページの制御）も可能。Settings 画面下部の **View in Playground** でプレビューを確認できる。

**対応プラットフォーム**

PLuG SDK は Web だけでなく、モバイルアプリにも対応している。

| プラットフォーム | パッケージ / SDK | 備考 |
|---|---|---|
| Web | PLuG SDK（JavaScript） | 上記の `init()` 方式 |
| iOS | DevRev SDK（Swift） | 最小デプロイターゲット iOS 15.0 |
| Android | `ai.devrev.sdk:devrev-sdk` | 最小 API level 24 |
| Flutter | `devrev_sdk_flutter` | Dart SDK 3.7.0 以上 |
| React Native | DevRev React Native SDK | — |

各 SDK で共通の機能（チャット、セッション録画、ユーザー識別、データマスキング）が利用できる。モバイルアプリに PLuG を組み込むことで、Web と同一のサポート体験を提供できる。

### Observe App の機能

Observe App は「ユーザーが何をしているか」を可視化する分析機能群である。PLuG SDK のセッション録画と連動して動作する。

**1. セッションリプレイ**

ユーザーのマウス操作、クリック、スクロール、画面遷移を録画する。DevRev のナレッジグラフに統合されており、チケットや会話から直接セッションに遷移できる。

録画では以下の情報もキャプチャされる。

- **ページ遷移**: 訪問したすべてのページとその順序
- **クリック / 入力イベント**: UI 要素とのインタラクション
- **Network calls**: API リクエストとレスポンス（エラーを含む）
- **Console errors**: JavaScript エラーのリアルタイムキャプチャ
- **体験品質の問題**: Dead Click（クリックしても反応しない）、Rage Click（連打）を自動検出

> **データマスキング**: 個人情報を含む UI 要素には CSS クラス `rr-mask` を付与する（Web）か、各モバイル SDK の `markSensitiveViews` メソッドを使うことで自動的にマスキングされる。

**2. ファネル分析**

ユーザーが離脱するポイントを可視化する。カスタムイベントベースのファネルビルダーで、任意のステップを定義できる。

- `plugSDK.trackEvent()` でカスタムイベントを発火する
- ファネルビルダーでイベントをステップとして並べる
- ドロップオフ区間をクリックすると、該当するセッション一覧にドリルダウンできる

```javascript
// カスタムイベントの発火例
window.plugSDK.trackEvent("checkout_started", {
  plan: "enterprise",
  source: "pricing_page",
});
```

**3. Session Analytics Dashboard**

設定不要で利用できる組み込みダッシュボード。PLuG のセッション録画を有効化するだけで、以下のメトリクスが自動的に集計される。

- ページビュー数・ユニークユーザー数の推移
- Dead Click が多いページの一覧
- 平均セッション時間
- 各ウィジェットからセッション一覧へのドリルダウン

カスタムダッシュボードの作成も可能（s09 で詳しく扱う）。

**4. Cross-domain Session Tracking**

複数ドメインにまたがるユーザージャーニーを1つのセッションとして追跡できる。

```javascript
// ドメインAでセッション情報を取得
const { sessionId, tabId } = window.plugSDK.getSessionDetails();

// ドメインBに渡して初期化時に設定する
```

### PLuGとObserve Appの連携

PLuGから起票されたチケットには、そのユーザーのセッション情報が自動的に紐付く。サポート担当者はチケットを開くだけで、ユーザーが問題に遭遇した前後の操作をリプレイで確認できる。

この連携は以下のように活用される。

- **Conversation → セッション**: チャットの会話画面から、関連するセッションリプレイに直接遷移
- **ファネルのドロップオフ → セッション**: 離脱したユーザーのセッションを特定して問題を調査
- **Dead Click → セッション**: UI の問題が起きているセッションを特定して修正につなげる

従来のZendesk+Hotjarの組み合わせでは手動で行っていた作業が、同一プラットフォーム内で自動化される。

## このセッションで学ぶこと

**始める前の状態：**
- PLuG SDKの導入方法を知らない
- セッションリプレイの活用経験がない
- 顧客行動分析とサポートツールが分離している
- ファネル分析でカスタムイベントを使う方法を知らない

**このセッション後の状態：**
- PLuG SDKを`init()`で初期化し、ライブチャットとセッション録画を動作させられる
- ユーザー識別（session_token）の仕組みを説明できる
- セッションリプレイで Dead Click / Rage Click / Network errors を確認できる
- `trackEvent()` でカスタムイベントを発火し、ファネル分析に活用できる
- PLuGとObserve Appの連携（チケット→セッションへの自動紐付け）を説明できる
- PLuG が Web / iOS / Android / Flutter に対応していることを知っている

## 試してみる

テストページにPLuGを埋め込んで動作確認しよう。

```
【PLuG導入チェックリスト】

□ 1. Settings > PLuG Settings で App ID を確認した
□ 2. テスト用HTMLページに plug.js スクリプトタグを追加した
□ 3. window.plugSDK.init({ app_id: "..." }) で初期化した
□ 4. ページを開いてPLuGウィジェット（ランチャー）が表示されることを確認した
□ 5. PLuGからメッセージを送信し、DevRevのInboxに届くことを確認した
□ 6. init() に enable_session_recording: true を追加してセッション録画を有効化した
□ 7. Observe Appでセッションリプレイが記録されていることを確認した
□ 8. PLuGから起票したチケットにセッション情報が紐付いていることを確認した

【発展課題】
□ 9. plugSDK.trackEvent() でカスタムイベントを1つ発火してみた
□ 10. Session Analytics Dashboard でメトリクスを確認した
□ 11. （バックエンドがある場合）session_token を使ってユーザー識別を試した
□ 12. shutdown() をログアウト時に呼び出してみた
```

次のセッションでは、開発サイクル（Issue管理からリリースまで）をDevRevで管理する。
