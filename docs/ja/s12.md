# DevRev APIを使ってみる

> "Personal Access Tokenさえ取得すれば、curlで今日からAPIを叩ける。"

## 問題

DevRevをカスタム統合したいが、何から始めればいいかわからない。

APIドキュメントは膨大で、認証方式も複数ある。

公開APIとベータAPIの違いも理解できていない。

## 解決策

Personal Access Token（PAT）を取得し、curlで最初のAPIコールを実行する。

これが最も速い入口である。

APIの体系を把握してから、本格的な統合開発に進む。

## 仕組み

**認証方式**

DevRev APIの認証方式は主に2つ。

| 方式 | 用途 | 取得方法 |
|------|------|---------|
| Personal Access Token (PAT) | 個人の開発・テスト用 | DevRevのSettings画面から発行 |
| Application Access Token (AAT) | アプリがユーザーのセッショントークンを取得するため | Dev Org 管理者が発行 |
| System User Token (SUT) | システムユーザーとしてのAPI呼び出し | Dev Org 管理者が発行 |
| Session Token | PLuG ウィジェットで顧客を識別してAPI呼び出し | AAT を使って STS から取得 |

**トークンの使い分け**

- **PAT** — まずはこれで始める。発行者本人と同じ権限でAPIを呼べる。Settings > API Tokens から発行する。
- **AAT** — PLuG ウィジェットで顧客（Rev User）を識別する際に使用する。AAT 自体ではAPIを呼ばず、Session Token を取得するための認証に使う。
- **SUT** — Snap-in ではなく外部アプリケーションがサービスアカウントとして動作する場合に使用する。
- **Session Token** — AAT で認証した後に取得する短命トークン。特定の顧客に紐づいたAPIコールに使う。PLuG SDK の `session_token` パラメータで Verified Identification を実現する仕組みの裏側がこれ。

まずはPATで始める。PATはDevRevのSettings > API Tokens画面から発行できる。

**オブジェクトの内部型階層（開発者向け）**

API を使いこなすには、DevRev のオブジェクトがどのような型階層で構成されているかを知っておくとよい。すべてのオブジェクトは **Atom** と呼ばれる共通の基底型から派生している。s03 の概念と対応させて読むと理解が早い。

```
Atom（基底型 — id, created_date, modified_date など共通属性）
├── Work
│   ├── Ticket
│   ├── Issue
│   ├── Task
│   └── Incident
├── Part
│   ├── Product / Capability / Feature / Enhancement（customer parts）
│   └── Code / Service / Runnable / Linkable（builder parts）
├── Org
│   ├── DevOrg
│   └── RevOrg
├── User
│   ├── DevUser
│   └── RevUser
└── Link（オブジェクト間の関係）
```

この階層を理解すると、API の設計方針が見えてくる。

- `works.list` で Ticket も Issue も Task も取得できるのは、すべてが **Work** のサブタイプだから。
- レスポンス JSON の共通フィールド（`id`, `created_date`, `modified_date`, `display_id` など）はすべて **Atom** から継承されている。
- `type` フィールドで具体的なオブジェクトタイプ（`ticket`, `issue` など）を識別する。

[オブジェクト構造リファレンス](/ja/architecture) でオブジェクト間のリンクルールを確認できる。API にまだ進まない場合は、この節は後回しにしてよい。

**エンドポイント体系**

DevRev APIはRESTful形式で提供される。ベースURLは以下の通り。

```
https://api.devrev.ai/
```

主要なエンドポイントの例。

| エンドポイント | メソッド | 説明 |
|-------------|--------|------|
| /works.list | POST | チケット・Issueの一覧取得 |
| /works.create | POST | チケット・Issueの作成 |
| /works.update | POST | チケット・Issueの更新 |
| /dev-users.list | POST | Dev Usersの一覧取得 |
| /rev-users.list | POST | Rev Usersの一覧取得 |
| /parts.list | POST | Partsの一覧取得 |

DevRev APIの特徴として、一覧取得や作成もPOSTメソッドを使用する点に注意する。

**公開API vs ベータAPI**

| 区分 | 安定性 | 変更通知 |
|------|--------|---------|
| 公開API (GA) | 安定。破壊的変更には事前通知あり | リリースノートで告知 |
| ベータAPI | 予告なく変更される可能性あり | 保証なし |

本番連携には公開APIのみを使用することを推奨する。ベータAPIはテスト・検証目的に限定する。

**最初のAPIコール例**

PATを取得した後、curlでチケット一覧を取得する例。

```bash
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"]}'
```

レスポンスはJSON形式で返る。各オブジェクトにはIDが付与されており、このIDを使って個別のオブジェクトを操作できる。

**レスポンスの共通フィールド（Atom 基底型）**

すべてのオブジェクトは Atom 基底型を継承しているため、レスポンス JSON に共通のフィールドが含まれる。

```json
{
  "work": [
    {
      "id": "don:core:dvrv-us-1:devo/0:ticket/123",
      "display_id": "TKT-123",
      "type": "ticket",
      "title": "ログインできない",
      "created_date": "2025-01-15T09:00:00Z",
      "modified_date": "2025-01-15T10:30:00Z",
      "created_by": { "id": "don:identity:dvrv-us-1:devo/0:devu/1", "display_name": "Taro Yamada" },
      "owned_by": [{ "id": "don:identity:dvrv-us-1:devo/0:devu/2", "display_name": "Hanako Sato" }],
      "applies_to_part": { "id": "don:core:dvrv-us-1:devo/0:product/1", "display_name": "MyApp" },
      "stage": { "name": "in_progress" },
      "priority": "p1"
    }
  ]
}
```

- `id` — オブジェクトの一意識別子（DON 形式）。すべてのAPI操作でこの値を使う。
- `display_id` — 人間が読める短縮ID（`TKT-123`, `ISS-456` など）。
- `type` — オブジェクトの具体的な型（`ticket`, `issue`, `task` など）。
- `created_date` / `modified_date` — ISO 8601 形式のタイムスタンプ。

**ページネーション**

DevRev API はカーソルベースのページネーションを採用している。レスポンスに `next_cursor` フィールドが含まれていれば、次のページが存在する。

```bash
# 最初のリクエスト
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"]}'

# レスポンスに next_cursor がある場合、次のページを取得
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"], "cursor": "ufhe492s..."}'
```

- `next_cursor` がレスポンスに含まれない場合 → 最後のページに到達済み。
- ページサイズの指定には `limit` パラメータを使う（未指定時はデフォルト値が適用される）。

**TypeScript SDK の活用**

curl で動作確認ができたら、本格的な開発には公式 TypeScript SDK を使用することを強く推奨する。HTTP リクエストを直接書くよりも型安全で、リトライ処理も組み込まれている。

```bash
npm install @devrev/typescript-sdk
```

SDK の初期化パターンは利用シーンによって異なる。

```typescript
import { client } from "@devrev/typescript-sdk";

// パターン1: Snap-in 内部（サービスアカウントトークンで認証）
const devrevSDK = client.setup({
  endpoint: event.execution_metadata.devrev_endpoint,
  token: event.context.secrets.service_account_token,
});

// パターン2: Beta API を使う場合
const devrevBeta = client.setupBeta({
  endpoint: event.execution_metadata.devrev_endpoint,
  token: event.context.secrets.service_account_token,
});
```

`client.setup()` は公開（GA）API のクライアントを返し、`client.setupBeta()` はベータ API を含むクライアントを返す。本番環境では `setup()` を使い、ベータAPIが必要な場合のみ `setupBeta()` を選択する。

SDK のメソッド名は API エンドポイントに対応している。

| API エンドポイント | SDK メソッド |
|------------------|-------------|
| /works.list | `devrevSDK.worksList(...)` |
| /works.create | `devrevSDK.worksCreate(...)` |
| /works.get | `devrevSDK.worksGet(...)` |
| /works.update | `devrevSDK.worksUpdate(...)` |
| /timeline-entries.create | `devrevSDK.timelineEntriesCreate(...)` |
| /dev-users.list | `devrevSDK.devUsersList(...)` |

外部サービスとの連携でも、直接 HTTP リクエストを組み立てるのではなく SDK を使うことでエラーハンドリングが統一される。

- DevRev APIの認証方法を知らない

**このセッション後の状態：**
- PATを発行し、curlでAPIコールを実行できる
- 主要エンドポイント（works / dev-users / rev-users / parts）の用途を説明できる
- 公開APIとベータAPIの違いを理解し、使い分けの判断ができる
- トークン種別（PAT / AAT / SUT / Session Token）の用途を区別できる
- カーソルベースのページネーションで全件取得ができる
- TypeScript SDK を使った開発の入口を理解している
