# DevRev APIを使ってみる

> "Personal Access Tokenさえ取得すれば、curlで今日からAPIを叩ける。"

## 問題

DevRevをカスタム統合したいですが、何から始めればいいかわかりません。

APIドキュメントは膨大で、認証方式も複数あります。

公開APIとベータAPIの違いも理解できていません。

## 解決策

Personal Access Token（PAT）を取得し、curlで最初のAPIコールを実行します。

これが最も速い入口です。

APIの体系を把握してから、本格的な統合開発に進みます。

## 仕組み

**認証方式**

DevRev APIの認証方式は主に2つです。

| 方式 | 用途 | 取得方法 |
|------|------|---------|
| Personal Access Token (PAT) | 個人の開発・テスト用 | DevRevのSettings画面から発行 |
| Application Access Token (AAT) | アプリがユーザーのセッショントークンを取得するため | Dev Org 管理者が発行 |
| System User Token (SUT) | システムユーザーとしてのAPI呼び出し | Dev Org 管理者が発行 |
| Session Token | Computer for Your Customers ウィジェットで顧客を識別してAPI呼び出し | AAT を使って STS から取得 |

**トークンの使い分け**

- **PAT** — まずはこれで始めます。発行者本人と同じ権限でAPIを呼べます。Settings > API Tokens から発行します。
- **AAT** — Computer for Your Customers ウィジェットで顧客（Rev User）を識別する際に使用します。AAT 自体ではAPIを呼ばず、Session Token を取得するための認証に使います。
- **SUT** — Snap-in ではなく外部アプリケーションがサービスアカウントとして動作する場合に使用します。
- **Session Token** — AAT で認証した後に取得する短命トークンです。特定の顧客に紐づいたAPIコールに使います。Computer for Your Customers SDK の `session_token` パラメータで Verified Identification を実現する仕組みの裏側がこれです。

まずはPATで始めてください。PATはDevRevのSettings > API Tokens画面から発行できます。

**オブジェクトの内部型階層（開発者向け）**

API を使いこなすには、DevRev のオブジェクトがどのような型階層で構成されているかを知っておくとよいでしょう。すべてのオブジェクトは **Atom** と呼ばれる共通の基底型から派生しています。s03 の概念と対応させて読むと理解が早くなります。

```
Atom（基底型 — id, created_date, modified_date など共通属性）
├── Work
│   ├── Ticket
│   ├── Issue
│   ├── Task
│   └── Incident
├── Part
│   ├── Product / Capability / Feature / Enhancement（customer parts）
│   └── Code / Service / Runnable / Linkable（builder parts）
├── Org
│   ├── DevOrg
│   └── RevOrg
├── User
│   ├── DevUser
│   └── RevUser
└── Link（オブジェクト間の関係）
```

この階層を理解すると、API の設計方針が見えてきます。

- `works.list` で Ticket も Issue も Task も取得できるのは、すべてが **Work** のサブタイプだからです。
- レスポンス JSON の共通フィールド（`id`, `created_date`, `modified_date`, `display_id` など）はすべて **Atom** から継承されています。
- `type` フィールドで具体的なオブジェクトタイプ（`ticket`, `issue` など）を識別します。

[オブジェクト構造リファレンス](/ja/reference/architecture) でオブジェクト間のリンクルールを確認できます。APIにまだ進まない場合は、この節は後回しにしてください。

**エンドポイント体系**

DevRev APIはRESTful形式で提供されます。ベースURLは以下の通りです。

```
https://api.devrev.ai/
```

主要なエンドポイントの例です。

| エンドポイント | メソッド | 説明 |
|-------------|--------|------|
| /works.list | POST | チケット・Issueの一覧取得 |
| /works.create | POST | チケット・Issueの作成 |
| /works.update | POST | チケット・Issueの更新 |
| /dev-users.list | POST | Dev Usersの一覧取得 |
| /rev-users.list | POST | Rev Usersの一覧取得 |
| /parts.list | POST | Partsの一覧取得 |

DevRev APIの特徴として、一覧取得や作成もPOSTメソッドを使用する点に注意してください。

**公開API vs ベータAPI**

| 区分 | 安定性 | 変更通知 |
|------|--------|---------|
| 公開API (GA) | 安定。破壊的変更には事前通知あり | リリースノートで告知 |
| ベータAPI | 予告なく変更される可能性あり | 保証なし |

本番連携には公開APIのみを使用することをお勧めします。ベータAPIはテスト・検証目的に限定してください。

**最初のAPIコール例**

PATを取得した後、curlでチケット一覧を取得する例です。

```bash
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"]}'
```

レスポンスはJSON形式で返ります。各オブジェクトにはIDが付与されており、このIDを使って個別のオブジェクトを操作できます。

**レスポンスの共通フィールド（Atom 基底型）**

すべてのオブジェクトは Atom 基底型を継承しているため、レスポンス JSON に共通のフィールドが含まれます。

```json
{
  "work": [
    {
      "id": "don:core:dvrv-us-1:devo/0:ticket/123",
      "display_id": "TKT-123",
      "type": "ticket",
      "title": "ログインできない",
      "created_date": "2025-01-15T09:00:00Z",
      "modified_date": "2025-01-15T10:30:00Z",
      "created_by": { "id": "don:identity:dvrv-us-1:devo/0:devu/1", "display_name": "Taro Yamada" },
      "owned_by": [{ "id": "don:identity:dvrv-us-1:devo/0:devu/2", "display_name": "Hanako Sato" }],
      "applies_to_part": { "id": "don:core:dvrv-us-1:devo/0:product/1", "display_name": "MyApp" },
      "stage": { "name": "in_progress" },
      "priority": "p1"
    }
  ]
}
```

- `id` — オブジェクトの一意識別子（DON 形式）。すべてのAPI操作でこの値を使います。
- `display_id` — 人間が読める短縮ID（`TKT-123`, `ISS-456` など）。
- `type` — オブジェクトの具体的な型（`ticket`, `issue`, `task` など）。
- `created_date` / `modified_date` — ISO 8601 形式のタイムスタンプ。

**ページネーション**

DevRev API はカーソルベースのページネーションを採用しています。レスポンスに `next_cursor` フィールドが含まれていれば、次のページが存在します。

```bash
# 最初のリクエスト
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"]}'

# レスポンスに next_cursor がある場合、次のページを取得
curl -X POST https://api.devrev.ai/works.list \
  -H "Authorization: Bearer YOUR_PAT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"type": ["ticket"], "cursor": "ufhe492s..."}'
```

- `next_cursor` がレスポンスに含まれない場合 → 最後のページに到達済みです。
- ページサイズの指定には `limit` パラメータを使います（未指定時はデフォルト値が適用されます）。

**TypeScript SDK の活用**

curl で動作確認ができたら、本格的な開発には公式 TypeScript SDK を使用することを強くお勧めします。HTTP リクエストを直接書くよりも型安全で、リトライ処理も組み込まれています。

```bash
npm install @devrev/typescript-sdk
```

SDK の初期化パターンは利用シーンによって異なります。

```typescript
import { client } from "@devrev/typescript-sdk";

// パターン1: Snap-in 内部（サービスアカウントトークンで認証）
const devrevSDK = client.setup({
  endpoint: event.execution_metadata.devrev_endpoint,
  token: event.context.secrets.service_account_token,
});

// パターン2: Beta API を使う場合
const devrevBeta = client.setupBeta({
  endpoint: event.execution_metadata.devrev_endpoint,
  token: event.context.secrets.service_account_token,
});
```

`client.setup()` は公開（GA）API のクライアントを返し、`client.setupBeta()` はベータ API を含むクライアントを返します。本番環境では `setup()` を使い、ベータAPIが必要な場合のみ `setupBeta()` を選択してください。

SDK のメソッド名は API エンドポイントに対応しています。

| API エンドポイント | SDK メソッド |
|------------------|-------------|
| /works.list | `devrevSDK.worksList(...)` |
| /works.create | `devrevSDK.worksCreate(...)` |
| /works.get | `devrevSDK.worksGet(...)` |
| /works.update | `devrevSDK.worksUpdate(...)` |
| /timeline-entries.create | `devrevSDK.timelineEntriesCreate(...)` |
| /dev-users.list | `devrevSDK.devUsersList(...)` |

外部サービスとの連携でも、直接 HTTP リクエストを組み立てるのではなく SDK を使うことでエラーハンドリングが統一されます。

## 用語

| 用語 | 説明 |
|------|------|
| **PAT（Personal Access Token）** | 個人の開発・テスト用APIトークン。発行者本人と同じ権限でAPIを呼べる |
| **AAT（Application Access Token）** | アプリケーション用トークン。顧客識別のためのSession Token取得に使う |
| **SUT（System User Token）** | システムユーザーとしてAPI呼び出しを行うためのトークン |
| **Session Token** | AATで認証後に取得する短命トークン。特定の顧客に紐づいたAPIコールに使う |
| **DON（DevRev Object Notation）** | DevRevオブジェクトの一意識別子の形式（例: `don:core:dvrv-us-1:devo/0:ticket/123`） |
| **display_id** | 人間が読める短縮ID（TKT-123、ISS-456 など） |
| **Atom** | すべてのDevRevオブジェクトの共通基底型。id / created_date / modified_date などを持つ |
| **ページネーション** | カーソルベースの結果取得方式。`next_cursor` で次のページを取得する |
| **TypeScript SDK** | DevRev公式のTypeScript用SDK（`@devrev/typescript-sdk`）。型安全でリトライ処理内蔵 |
| **ベータAPI** | 予告なく変更される可能性があるAPI。本番利用は非推奨 |
