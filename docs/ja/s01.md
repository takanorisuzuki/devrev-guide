# DevRevとは何か：AIネイティブな作業基盤

> "作り手と使い手をつなぐ作業基盤で、記録に沿った答えと統制を同じデータモデル上に載せる。DevRevが目指す世界を理解する。"

## 問題

多くの企業では、サポートにZendesk、開発管理にJira、AI活用にChatGPT・Copilot・Claudeなどと、役割ごとに別々のツールを使っている。

その結果、チーム間の情報伝達にコストがかかり、顧客の声がエンジニアに直接届かない。

チケットが起票されてからIssueになるまでに何日もかかる。

同じ問題が繰り返し報告されても、誰も気づかない。

ツールの分断は、チームの分断を生む。

加えて、ChatGPT・Copilot・Claude などの汎用アシスタントは会話や要約に強い一方で、CRM やチケットなど複数の記録システム（SoR）にまたがる**事実の一貫性**、**誰が何を見られるか**、**答えの根拠を後から検証する**といった要件は、別レイヤーで設計しがちである。

ツール分断に加え、ここが曖昧だと「便利そうな回答」と「業務として担保したい回答」の間にギャップが生じる。

この章では、**MCP** や **Claude Code**、各プロバイダの **Skills** のように「モデル外でツールや文脈をつなぐ」方式を、ある程度は知っている前提で進める（**Claude Code に限った話ではない**）。

ChatGPT を含むどのフロントモデルでも、**LLM ＋ RAG（ドキュメント等の検索）＋ MCP／Skill** はできることが増える一方、複数 SoR をまたいだ**権限の一貫した適用**、**参照の抜け・ノイズ**、**ツール呼び出しの連鎖に伴うレイテンシとトークン**、**事後に同じ条件で根拠を追う**ことは、設計を誤れば依然として難しい。

これは「より賢いモデル」より手前の、**データと文脈をどう束ねるか**の問題である。

実際、**Claude等の汎用モデルを単体で**API/MCP経由でつなぐ fetch型 は、セッションごとに schema 探索や関係の再構築が発生しやすく、トークン/時間コストが複利に増えがちである。

Computer の **Memory** は権威ある関係（typed edges）を “読む” 前提で、join/filter のような構造化処理は決定的に行い、必要な結果セットだけ返すことで推測とコストを抑える。

CTO office の公開比較では、同一クエリ反復で **約70–95%のトークン削減** と **最大約5.5倍の高速化** が報告されている（要点は [Memory vs fetch型AI：正確性とコスト](/ja/memory-vs-fetch-ai-accuracy-and-cost)）。

## 解決策

DevRevは、サポート（Rev）と開発（Dev）とAIを1つのプラットフォームに統合したSaaSプロダクトである。

重要なのは「既存のツールや運用をすべて置き換える」ことではなく、**今あるSoR（CRM・チケット・開発管理など）を保ったまま**、記録と権限に沿った文脈を束ねる土台を用意する、という捉え方である。

DevRevには3つの主要アプリケーションがある。

- **Support App**: チケット管理、SLAポリシー、顧客対応を担当する
- **Build App**: 課題管理、スプリント、リリースなど開発サイクルを担当する
- **Observe App**: セッションリプレイ、ファネル分析など顧客行動の可視化を担当する

さらに、PLuGというWebサイト埋め込み型のライブチャットウィジェットがあり、顧客接点を直接プラットフォーム内に取り込める。

プラットフォーム全体の正式なブランド名は **Computer by DevRev** である（後述の「仕組み」で触れる **Computer** 機能とセットで覚えるとよい）。

## 仕組み

DevRevの設計思想は「AIネイティブ」にある。後からAI機能を追加したのではなく、プラットフォームの土台にAIが組み込まれている。

DevRevは現在、プラットフォーム全体を**Computer by DevRev**という名称でブランド展開している。「人間エージェントとAIエージェントの両方を支えるオペレーティングシステム」として位置づけられており、単なるAI機能の集合ではなく、プラットフォームそのものが**Computer**である。

その中核に「**Computer**」と呼ばれるAIチームメイト機能が存在する。プラットフォームとして重視するのは、エンタープライズで共通になりやすい次のような価値である（実行やガバナンスの詳細は [s14](/ja/s14) でも扱う）。

- **精密な Search / Answers**: 必要なレコードとフィールドに絞って文脈を組み立て、推論に渡す材料の質を揃える
- **権限に沿った Memory**: 誰が何を見られるかを前提に、ナレッジグラフ上の関係性を参照する
- **監査・説明可能性**: 結論の根拠を、可能な限り記録とトレースしやすい形で残す設計を目指す
- **ワークフローとアクションの分離**: 検証可能な答えと、実行・自動化を設計上切り分け、統制しやすくする

ユーザー近辺で体感しやすい例としては、次のような処理がある。

- チケットの自動分類・トリアージ
- 顧客の過去の問い合わせ履歴の要約
- 類似チケットの検出と関連Issueの提案

ここまでが「Computer がユーザーに近いところで行う処理の一例」である。次に、Computer を支える **全体の地図**として、基盤が 4 つに分かれていることを押さえる（各基盤の深掘りや Agent Studio の詳細は [s14](/ja/s14) で扱う）。

### Computer by DevRev の 4 つの基盤

Computer は単一の AI 機能ではなく、4 つの基盤（Foundation）で構成されている。

| 基盤 | 役割 | 具体例 |
|------|------|--------|
| **AirSync** | 50 以上の外部ツールとの双方向同期 | Jira・Zendesk・Salesforce のデータをリアルタイムで取り込み、DevRev 上のオブジェクトとして統合管理する |
| **Memory** | 特許取得済みの AI ネイティブなナレッジグラフ | 顧客・製品・チケット・Issue の関係を構造的に記憶し、**閲覧権限に沿って**検索やエージェントが参照できる形にする |
| **Foundational Services** | 検索・自動化・分析の共通基盤 | 全文検索、ワークフロー、ダッシュボードなどを横断提供し、**再利用と説明しやすさ**の前提を揃える |
| **Agent Studio** | ノーコードの AI エージェント構築環境 | ゴール・ツール・**制約**を定義して自律エージェントを設計する（実行面の詳細は [s14](/ja/s14)） |

これら 4 つの基盤があることで、上述のチケット自動分類やナレッジベース検索といった個別機能が実現されている。AirSync が外部データを集約し、Memory がそれを構造化し、Foundational Services が処理基盤を提供し、Agent Studio がそれらを束ねた自律エージェントを生み出す——という積み上げ構造になっている。

従来のツールスタックとDevRevの対応関係を整理すると以下のようになる。

| 従来のツール | DevRevの対応機能 |
|------------|----------------|
| Zendesk / Freshdesk | Support App |
| Jira / Linear | Build App |
| Hotjar / FullStory | Observe App |
| Intercom / Drift | PLuG |
| ChatGPT / Copilot / Claude（手動連携） | Computer（プラットフォーム内蔵AI） |

この対応表の「ChatGPT / Copilot / Claude → Computer」は、**汎用アシスタントを捨てる**話ではない。汎用アシスタントは会話・要約・思考の外部化に強く、MCP/Skill と組み合わせれば多くの業務を前に進められる。一方で、複数SoRをまたぐ権限・根拠・統制は「つなぎ方」だけでは揺れやすい。Computer by DevRev は、同一データモデルの上に Memory や Foundational Services が乗る**スタック全体**として、汎用アシスタントをより安全に使う土台にもなる。

重要なのは、Support App などが別々のプロダクトではなく、同一のデータモデル上で動作する点である。サポートチケットと開発Issue、顧客データが同じプラットフォーム内でつながっているため、チーム間の情報断絶が構造的に解消される。

## このコースの全体像（14セッション）

各セッションが何に向かって積み上げるのかを先に把握しておくと、学習が進めやすい。

s01-s04はbeginnerセッション。DevRevの利用経験は不要だ。s05以降はintermediate、s13-s14はadvanced。各セッションは前のセッションの知識を前提として積み上げる。

## このセッションで学ぶこと

**始める前の状態：**
- DevRevの名前は聞いたことがあるが、具体的に何ができるかわからない
- Jira+Zendeskの組み合わせに不満はあるが、代替案を評価する基準がない
- 「AIネイティブ」が何を意味するか曖昧
- ChatGPT・Copilot・Claude などを業務で使っているが、SoR をまたぐ回答の根拠や権限に不安がある
- MCP や Claude Code、Skills で SoR やドキュメントに触れているが、権限・監査・連鎖のコストに限界を感じている

**このセッション後の状態：**
- DevRevの3つの主要アプリ（Support App / Build App / Observe App）の役割を説明できる
- 従来のツールスタックとDevRevの対応関係をマッピングできる
- 「AIネイティブ」の意味を「Computerがプラットフォームに組み込まれている」と具体的に説明できる
- Computer by DevRev の 4 つの基盤（AirSync / Memory / Foundational Services / Agent Studio）の役割を説明できる
- 汎用アシスタントと Computer が担うレイヤー（会話支援 vs 記録に沿った答えと統制）の違いを説明できる
- Computer が目指すエンタープライズ向けの価値（Search/Answers、権限付き Memory、監査、ワークフローとアクションの分離）を列挙できる
- LLM＋RAG＋MCP／Skill の構成でも残りやすい難しさ（権限・抜け・コスト・監査）を例に挙げて説明できる

## 試してみる

自社（または所属組織）の現行ツールスタックを書き出し、「置き換える」ではなく、**どの記録をどの単位で束ね、どこを統制したいか**をマッピングしてみよう。

```
【ツールスタック・マッピング表】

現行ツール（SoR）        → DevRev上で束ねる単位
─────────────────────────────────────
サポート: _____________ → Support App
開発管理: _____________ → Build App
行動分析: _____________ → Observe App
チャット: _____________ → PLuG
AI活用:   _____________ → Computer（権限・根拠・統制の土台）

【考察】
- 統合による最大のメリットは何か？
- 移行の際に最もリスクが高い領域はどこか？
```
