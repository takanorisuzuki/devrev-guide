# DevRevとは何か：AIネイティブな作業基盤

> "作り手と使い手をつなぐ作業基盤で、記録に沿った答えと統制を同じデータモデル上に載せる。DevRevが目指す世界を理解する。"

## 問題

業務でAIを使うと、会話や要約はすぐ役に立ちます。一方で実務では、複数の記録システム（SoR）をまたいで **正しい事実**を参照し、**権限**を守り、**根拠**を後から追えて、必要なら**実行の統制**までできることが求められます。ここが揃わないと、回答がどれだけ「それっぽく」ても業務としては使いにくいです。

### ツールの分断

サポート（Zendesk 等）・開発（Jira 等）・顧客管理（Salesforce 等）・AI（ChatGPT / Copilot / Claude 等）が別々だと、次が起きやすいです。

- 顧客の声がエンジニアに届きにくい／チケット→Issue が遅い
- 同じ問題の再報告に気づきにくい  
  → **ツールの分断はチームの分断**になります。

### 汎用アシスタントと「つなぎ」の限界

会話・要約は強いですが、複数の記録システム（SoR）をまたいだ **事実の一貫性・誰が何を見られるか・事後の根拠確認** は、別レイヤーで設計しがちで、「それっぽい答え」と「業務として正しい答え」がずれます。

**Claude Code** や **Claude Cowork** などでフロントは進んでいます。**MCP / Skills** でもできることは増える一方、データ層が弱いと **権限の一貫性・呼び出し連鎖のコスト・運用の複雑さ** が残りやすいです。

### 確率論の LLM と、企業が欲しい土台

- **LLM＋RAG** は根底が**確率論**。業務として筋の通った判断には、推測ではなく **関係として持つナレッジグラフ** のような土台が別途必要になりやすいです。
- **RBAC** や **アクション後のシステム全体のロールバック**（複数の記録・オブジェクトにまたがる操作を、監査可能に取り消すこと）のような要件は、**汎用モデル＋接続の束だけ**ではまだ厳しい場面が多いです。
- さらに実務では、CRM・チケット・Issue のような**構造化データ**と、メール／通話ログ／チャット／仕様書のような**非構造化データ**を「どの顧客・どの製品・どの事象に紐づくか」まで一貫して結び、権限付きで辿れることが重要になります。検索で断片を拾えるだけでは、関係の取り違えや根拠の断裂が起きやすいです。

## 解決策

DevRevは、サポート（Rev）と開発（Dev）とAIを1つのプラットフォームに統合したSaaSプロダクトです。

重要なのは「既存のツールや運用をすべて置き換える」ことではなく、**今あるSoR（CRM・チケット・開発管理など）を保ったまま**、記録と権限に沿った文脈を束ねる土台を用意する、という捉え方です。

DevRevには3つの主要アプリケーションがあります。

- **Computer for Support Teams**: チケット管理、SLAポリシー、顧客対応を担当します
- **Computer for Builders**: 課題管理、スプリント、リリースなど開発サイクルを担当します
- **Computer for User Insights**: セッションリプレイ、ファネル分析など顧客行動の可視化を担当します

さらに、**Computer for Your Customers** というWebサイト埋め込み型のライブチャットウィジェットがあり、顧客接点を直接プラットフォーム内に取り込めます。

プラットフォーム全体の正式なブランド名は **Computer by DevRev** です（後述の「仕組み」で触れる **Computer** 機能とセットで覚えるとよいでしょう）。

## 仕組み

DevRevの設計思想は「AIネイティブ」にあります。後からAI機能を追加したのではなく、プラットフォームの土台にAIが組み込まれています。

DevRevは現在、プラットフォーム全体を**Computer by DevRev**という名称でブランド展開しています。「人間エージェントとAIエージェントの両方を支えるオペレーティングシステム」として位置づけられており、単なるAI機能の集合ではなく、プラットフォームそのものが**Computer**です。

ここでのポイントは、「より賢いモデルに乗り換える」だけではなく、**社内の記録・権限・根拠を束ねるデータ層**を持てるかどうかです。フロントモデルが進化しても、その土台が薄いと、SoR をまたぐ正しさや統制の難しさは残りやすいです。

その中核に「**Computer**」と呼ばれるAIチームメイト機能が存在します。プラットフォームとして重視するのは、エンタープライズで共通になりやすい次のような価値です（実行やガバナンスの詳細は [s14](/ja/s14) でも扱います）。

- **精密な Search / Answers**: 必要なレコードとフィールドに絞って文脈を組み立て、推論に渡す材料の質を揃えます
- **権限に沿った Memory**: 誰が何を見られるかを前提に、ナレッジグラフ上の関係性を参照します
- **監査・説明可能性**: 結論の根拠を、可能な限り記録とトレースしやすい形で残す設計を目指します
- **ワークフローとアクションの分離**: 検証可能な答えと、実行・自動化を設計上切り分け、統制しやすくします

ユーザー近辺で体感しやすい例としては、次のような処理があります。

### fetch型と Memory の違い（要点）

| 比較の要点 | **fetch型**（API/MCP でモデル単体） | **Computer の Memory**                                                                   |
| :--------- | :----------------------------------- | :--------------------------------------------------------------------------------------- |
| 関係の扱い | セッションごとに探索・再構築しがち   | 権威ある **typed edges** を読み、join/filter は決定的に                                  |
| コスト感   | トークン・時間が複利で増えやすい     | 結果セット中心に抑えやすい（[公開比較の目安](/ja/reference/memory-vs-fetch-ai-accuracy-and-cost)） |

- チケットの自動分類・振り分け
- 顧客の過去の問い合わせ履歴の要約
- 類似チケットの検出と関連Issueの提案

ここまでが「Computer がユーザーに近いところで行う処理の一例」です。次に、Computer を支える **全体の地図**として、基盤が 4 つに分かれていることを押さえます（各基盤の深掘りや Agent Studio の詳細は [s14](/ja/s14) で扱います）。

### Computer by DevRev の 4 つの基盤

Computer は単一の AI 機能ではなく、4 つの基盤（Foundation）で構成されています。

| 基盤                      | 役割                                         | 具体例                                                                                                         |
| ------------------------- | -------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| **Computer AirSync**      | 50 以上の外部ツールとの双方向同期            | Jira・Zendesk・Salesforce のデータをリアルタイムで取り込み、DevRev 上のオブジェクトとして統合管理する          |
| **Memory**                | 特許取得済みの AI ネイティブなナレッジグラフ | 顧客・製品・チケット・Issue の関係を構造的に記憶し、**閲覧権限に沿って**検索やエージェントが参照できる形にする |
| **Foundational Services** | 検索・自動化・分析の共通基盤                 | 全文検索、ワークフロー、ダッシュボードなどを横断提供し、**再利用と説明しやすさ**の前提を揃える                 |
| **Agent Studio**          | ノーコードの AI エージェント構築環境         | ゴール・ツール・**制約**を定義して自律エージェントを設計する（実行面の詳細は [s14](/ja/s14)）                  |

これら 4 つの基盤があることで、上述のチケット自動分類やナレッジベース検索といった個別機能が実現されています。Computer AirSync が外部データを集約し、Memory がそれを構造化し、Foundational Services が処理基盤を提供し、Agent Studio がそれらを束ねた自律エージェントを生み出す——という積み上げ構造になっています。

従来のツールスタックとDevRevの対応関係を整理すると以下のようになります。

| 従来のツール                           | DevRevの対応機能                                    |
| -------------------------------------- | --------------------------------------------------- |
| Zendesk / Freshdesk                    | Computer for Support Teams                          |
| Jira / Linear                          | Computer for Builders                               |
| Hotjar / FullStory                     | Computer for User Insights                          |
| Intercom / Drift                       | Computer for Your Customers                         |
| ChatGPT / Copilot / Claude（手動連携） | Computer（プラットフォーム内蔵AI）                  |

この対応表の「ChatGPT / Copilot / Claude → Computer」は、**汎用アシスタントを捨てる**話ではありません。汎用アシスタントは会話・要約・思考の外部化に強く、Claude Code / MCP / Skill と組み合わせればできることは広がります。一方で、確率論としての LLM、RAG の抜け、複数 SoR をまたぐ **RBAC** や **アクション後の全体ロールバック**までを「モデル＋接続」だけで完結させるのは難しく、権限・根拠・統制は「つなぎ方」だけでは揺れやすいです。Computer by DevRev は、同一データモデルの上に Memory（ナレッジグラフ）や Foundational Services が乗る**スタック全体**として、**どの世代の最新モデルにも**社内知識を、単にプロンプトを長くするのではなく意味を持って接続しつつ、汎用アシスタントをより安全に使う土台にもなります。

重要なのは、Computer for Support Teams などが別々のプロダクトではなく、同一のデータモデル上で動作する点です。サポートチケットと開発Issue、顧客データが同じプラットフォーム内でつながっているため、チーム間の情報断絶が構造的に解消されます。

## このコースの全体像（14セッション）

各セッションが何に向かって積み上げるのかを先に把握しておくと、学習が進めやすいです。

- **s01-s03（DevRev基礎）**: DevRevの全体像、セットアップ、データモデルの理解
- **s04-s09（プラットフォーム活用）**: サポート設計、ナレッジベース、顧客理解、開発サイクル、管理者設定、分析ダッシュボード
- **s10-s14（開発者・拡張）**: ワークフロー自動化、Snap-in活用と自作、API利用、Agent Studio

s01-s04はbeginnerセッションです。DevRevの利用経験は不要です。s05以降はintermediate、s13-s14はadvancedです。各セッションは前のセッションの知識を前提として積み上げます。

## このセッションで学ぶこと

**始める前の状態：**

- DevRevの名前は聞いたことがあるが、具体的に何ができるかわからない
- Jira+Zendeskの組み合わせに不満はあるが、代替案を評価する基準がない
- 「AIネイティブ」が何を意味するか曖昧
- ChatGPT・Copilot・Claude などを業務で使っているが、SoR をまたぐ回答の根拠や権限に不安がある
- MCP や Claude Code、Skills で SoR やドキュメントに触れているが、権限・監査・連鎖のコストに限界を感じている
- 「LLM+RAG はそれっぽい答え」「RBAC やロールバックはアシスタント単体では足りない」と感じている

**このセッション後の状態：**

- DevRevの主要アプリ（Computer for Support Teams / Computer for Builders / Computer for User Insights / Computer for Your Customers）の役割を説明できます
- 従来のツールスタックとDevRevの対応関係をマッピングできます
- 「AIネイティブ」の意味を「Computerがプラットフォームに組み込まれている」と具体的に説明できます
- Computer by DevRev の 4 つの基盤（Computer AirSync / Memory / Foundational Services / Agent Studio）の役割を説明できます
- 汎用アシスタントと Computer が担うレイヤー（会話支援 vs 記録に沿った答えと統制）の違いを説明できます
- Computer が目指すエンタープライズ向けの価値（Search/Answers、権限付き Memory、監査、ワークフローとアクションの分離）を列挙できます
- LLM＋RAG＋MCP／Skill の構成でも残りやすい難しさ（確率論としての限界、権限・抜け・コスト・監査）を例に挙げて説明できます
- RBAC やアクション後の全体ロールバックのような要件を、汎用モデルと接続だけで担う難しさと、Computer / Memory が補う点を説明できます
- モデル世代が変わっても Computer が社内知識をグラフでつなぎ、最新モデルへ意味のある、ポリシーに沿った接続を渡せる、というイメージを説明できます

## 試してみる

自社（または所属組織）の現行ツールスタックを書き出し、「置き換える」ではなく、**どの記録をどの単位で束ね、どこを統制したいか**をマッピングしてみましょう。

```
【ツールスタック・マッピング表】

現行ツール（SoR）        → DevRev上で束ねる単位
─────────────────────────────────────
サポート: _____________ → Computer for Support Teams
開発管理: _____________ → Computer for Builders
行動分析: _____________ → Computer for User Insights
チャット: _____________ → Computer for Your Customers
AI活用:   _____________ → Computer（権限・根拠・統制の土台）

【考察】
- 統合による最大のメリットは何か？
- 移行の際に最もリスクが高い領域はどこか？
```
