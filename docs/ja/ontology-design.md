# オントロジー設計ガイド

このページは、DevRev に自社のデータを入れるとき「何を、どこに、どう入れるか」を自分で判断できるようになるための設計ガイドです。操作手順ではなく **考え方** を伝えます。

読み終えたとき、以下ができるようになることを目指します。

- 自社のデータを見たときに「これはどのオブジェクトに入るか」が判断できる
- 標準オブジェクトで足りるのか、カスタムオブジェクトを作る必要があるのかがわかる
- 複数のデータが「どう繋がるか」を想像でき、その繋がりから得られる価値が見える

オブジェクトの一覧・図・リンクルールは [オブジェクト構造リファレンス](/ja/reference/architecture)、データモデルのチュートリアルは [s03](/ja/s03) を参照してください。

## 5つの問い — ビジネスのデータはこれで整理できる

あなたの会社のすべてのデータは、以下の5つの問いのどれかに答えるものです。

| # | 問い | 入るオブジェクト |
|---|------|--------|
| 1 | 誰が関わっているか？ | Account / RevUser / DevUser |
| 2 | 何を提供しているか？ | Parts（Product / Capability / Feature） |
| 3 | 何が起きているか？ | Ticket / Issue / Incident と Enhancement |
| 4 | 誰に何を売ろうとしているか？ | Opportunity |
| 5 | 何を知っているか？ | Article |

どの問いにも当てはまらないデータだけが「カスタムオブジェクトを作る」対象になります。

### なぜこの5つか

ほとんどのビジネスはこのサイクルで動いています。

```
お客様に売る → お客様が使う → 困ったことが起きる → 社内で対応する → 改善する → また価値を届ける
```

この流れに登場する情報は「人」「提供物」「仕事」「商談」「知識」に集約されます。DevRev はこのサイクル全体をひとつの場所で繋げるために、この5つを基本構造にしています。

## 1. 誰が関わっているか — Account / RevUser / DevUser

ビジネスに関わる人と組織は、大きく3種類に分けて管理します。

- **Account（取引先の会社）** — 顧客企業、パートナー・代理店、仕入先など。1レコード = 1社が基本です
- **RevUser（取引先の担当者・エンドユーザー）** — 窓口担当、問い合わせをしてくる人、商談の意思決定者、製品を実際に使うエンドユーザー
- **DevUser（自社メンバー）** — サポート担当、営業、エンジニア、PM など DevRev を使う全員。チーム・部署はグループにまとめられます

**大事なポイント:** 1人の RevUser は複数の Account に紐づけられます。パートナー企業の営業が複数のエンド顧客を担当している、グループ会社間で兼務している、といったケースでも同じ人を重複して作る必要はありません。また、所属する会社がまだ不明な見込み客やフリーランスも、Account への紐づけなしで RevUser として登録できます。所属先が判明した時点で後から紐づければ OK です。

### 振り分けの目安

| あなたのデータ | どこに入るか |
|---|---|
| 顧客の会社名リスト | Account |
| 顧客の担当者一覧 | RevUser |
| 名刺情報 | RevUser（会社部分は Account） |
| 社員名簿 | DevUser |
| 部署一覧 | グループ |
| 代理店リスト | Account（タグ等で「代理店」と区別） |

## 2. 何を提供しているか — Parts（最大4階層）

あなたの会社が提供しているモノ・サービスを Parts として構造化します。これは単なるカタログではありません。**「問い合わせや作業が、提供物のどの部分に関するものか」を追跡するための地図** です。

| 階層 | オブジェクト | 意味 |
|---|---|---|
| 段階1 | Product | お客様が「御社の○○を使っています」と言うときの「○○」 |
| 段階2 | Capability | 大きな機能領域・モジュール（例: 仕訳入力、帳票・レポート、経費精算） |
| 段階3 | Feature | お客様が「ここが良い」「ここが困る」と指す最小の単位（例: PDF出力、CSVエクスポート） |
| 段階4 | Builder Parts（Sub-feature / Runnable / Linkable） | 裏側の実装やサブ機能（マイクロサービス、API、内部モジュール） |

段階1〜3はお客様から見える構造（customer parts）、段階4は作り手が使う構造（builder parts）です。ソフトウェアに限った概念ではなく、産業機器なら「アーム制御 → 衝突検知」、コンサルなら「戦略立案 → KPI設計」のように、**「お客様が何について話すか」の単位で分解する** だけです。

**段階4（Builder Parts）を使う意味:**

- Issue を「技術的にどのコンポーネントに関するか」まで紐づけ可能
- Pull Request が「お客様のどの機能に影響するか」を自動追跡
- 障害発生時に「このサービスが落ちたら、お客様のどの機能が影響を受けるか」が即座にわかる

段階4は全員が使う必要はありません。開発チームが「コードと製品構造を接続したい」場合に使います。お客様対応だけなら段階1〜3で十分です。

### なぜ構造化するのか

構造があると、以下が自動的にわかるようになります。

- **どこに問い合わせが集中しているか** → 改善の優先度が見える
- **ここに障害が起きたら何社に影響するか** → インパクトが即座にわかる
- **お客様が最も求めている改善はどこか** → ロードマップの判断材料になる

構造が無いと、問い合わせが100件来ても「どこに集中しているか」がわかりません。手動で Excel 集計して、それでも漏れます。

### どこまで細かくすべきか

完璧を目指す必要はありません。

| 状況 | 推奨 |
|---|---|
| 提供物が1つでシンプル | 2階層（Product + Feature）で十分 |
| 提供物が1つで多機能 | 3〜4階層使う |
| 提供物が複数 | それぞれの Product の下に2〜4階層 |
| 技術側も管理したい | 段階4でサービスやコンポーネントを表現 |

**最初は主要な5〜10個だけ** 入れれば十分です。問い合わせが来るたびに「これはどこに属するか」を判断していけば、自然と構造は育ちます。Parts 階層の設計演習は [s03](/ja/s03) にあります。

## 3. 何が起きているか — Ticket / Issue / Enhancement / Incident

日々発生する仕事を「誰のために、何をするか」で分類します。

### Ticket（お客様のための仕事）

お客様から何か言われて「対応します」と約束した仕事です。問い合わせ、サポートリクエスト、不具合報告、要望、クレームが該当します。

**共通する特徴:**

- 必ず「依頼者」がいる
- 「解決しました」とお客様に伝える義務がある
- 対応期限が意識される

### Issue（社内でやる仕事）

自分たちが「これをやろう」と決めた仕事です。お客様はこの仕事の存在を直接知りません。開発タスク、バグ修正、施策タスク、調査などが該当します。

### Ticket と Issue の繋がり — ここが最も重要

**お客様のための仕事から、社内の仕事が生まれることがある。**

```
お客様「レポートのPDFが文字化けします」（Ticket）
     │
     ▼ サポート担当が調査し、開発への作業を起こす
     │
社内「PDF生成ライブラリのフォント処理を修正する」（Issue）
     │
     ▼ 修正完了
     │
お客様に「修正しました」と伝えて Ticket 完了
```

この繋がりが保たれることで:

- お客様への回答漏れが防げる（修正したのに通知忘れ、がなくなる）
- 「今どのお客様の問題が未解決か」が一覧できる
- 「1つのバグが何人のお客様に影響しているか」が数字で見える

繋がりは多対多です。1つの Ticket が複数の Issue に分解されることもあれば、1つの Issue が複数のお客様の問題を同時に解決することもあります。

### Enhancement（改善テーマ）

複数の Issue をまとめる、大きな方向性や改善プロジェクトです。エピック、プロジェクト、イニシアチブと呼んでいるものが該当します。DevRev では Enhancement は Work ではなく **Part** に属し、複数 Issue の親として機能します（[オブジェクト構造リファレンス](/ja/reference/architecture) 参照）。

```
Enhancement: 「モバイル対応」
  ├── Issue: 「レスポンシブデザイン対応」
  ├── Issue: 「タッチ操作の最適化」
  ├── Issue: 「モバイル用APIの開発」
  └── Issue: 「オフライン対応」
```

これにより「モバイル対応は何%進んでいるか」が、配下の Issue の進捗から自動で見えます。

### Incident(障害)

システムやサービスの異常事態です。通常の仕事とは区別されます。障害は緊急性が異なり、影響範囲の即時把握が必要だからです。

### 仕事の振り分けフロー

```
それは「やるべきこと」ですか？
│
├── Yes → 誰が依頼した？
│   │
│   ├── お客様 → Ticket
│   │
│   └── 自分たち → 規模は？
│       │
│       ├── 小さい作業 → Issue
│       │
│       └── 複数人で数週間以上 → Enhancement
│              └── その下に個別の Issue を作る
│
├── システム障害 → Incident
│
└── No → 他のオブジェクトを検討
```

## 4. 商談とナレッジ — Opportunity / Article

### Opportunity（商談）

営業活動で「まだ決まっていないが、進めている取引」を管理します。どの Account への提案か、金額、成約確度、進行ステージ、担当営業、クローズ予定日を持てます。

**Opportunity は「約束前」の状態です。** 成立後の契約書や注文書を管理したい場合は、カスタムオブジェクト（後述）を作る対象になります。

### Article（ナレッジ）

組織に蓄積された知識やドキュメントです。FAQ、マニュアル、社内ナレッジ、リリースノートが該当します。

**他のオブジェクトと繋がります:**

- Ticket の回答として Article を案内する
- よくある質問をまとめて再利用する
- ナレッジを充実させることで問い合わせ自体を減らす

アクセス制御の詳細は [Articleアクセス制御](/ja/reference/article-access-control) を参照してください。

## 5. 繋がりの力 — データが連携すると見えること

DevRev の最大の価値は「オブジェクトの中身」ではなく「オブジェクト同士の繋がり」にあります。

### 全体の繋がり

```
RevUser（田中部長）
    │ 所属
    ▼
Account（A社）
    │                          │
    ▼                          ▼
Opportunity                 Ticket
「年間契約更新」            「レポートが遅い」
                                │
                                ▼
                            Issue
                            「SQLクエリの最適化」
                                │
                                ▼
                            Feature
                            「月次レポート自動生成」
                                │
                                ▼
                            Product
                            「クラウド会計ソフト」
```

### この繋がりから答えられる問い

| 経営やマネージャーが聞くこと | 繋がりが答えてくれる |
|---|---|
| 契約更新が近い顧客で未解決の問題があるのはどこ？ | Opportunity（更新期限） + Account + 未解決の Ticket |
| 今期一番問い合わせが多い領域は？ | Ticket → Parts で集計 |
| この障害で何社に影響が出た？ | Incident → Parts → Ticket → Account |
| お客様の要望が多い改善は？ | Ticket（要望）→ Issue → Enhancement |
| あの改善をリリースしたら何件の問い合わせが解消する？ | Enhancement → Issue → Ticket 件数 |

### 繋がりを活かすためにやること

**やるべきこと:**

- Ticket を登録するとき「Parts のどの部分についてか」を選ぶ
- Issue を作るとき「どの Ticket がきっかけか」を紐づける

**自動で繋がるもの（手動不要）:**

- RevUser と Account の紐づけ（メールドメインから自動判定）
- Ticket がどの Account のものか（問い合わせ者から自動判定）

## 6. 標準で足りないとき — カスタムオブジェクトを作る判断基準

拡張手段そのもの（カスタムフィールド・サブタイプ・カスタムオブジェクトの設定方法）は [オブジェクト構造リファレンス](/ja/reference/architecture) の「拡張の仕組み」を参照してください。このセクションは **「どれを選ぶべきか」の判断基準** を扱います。

### 基本原則

**まず「標準オブジェクト + カスタムフィールド」で対応できないか考える。**

多くの場合、カスタムオブジェクトを作らなくても解決します。

| やりたいこと | 正しいアプローチ |
|---|---|
| 顧客情報に「業界」「契約プラン」を足したい | Account にカスタムフィールドを追加 |
| Ticket を「返品」「障害」「質問」に分けたい | Ticket のサブタイプを作成 |
| Issue にバグ固有の情報を足したい | サブタイプ「バグ」を作り、専用フィールドを追加 |

### カスタムオブジェクトが必要になる5つのパターン

**パターン1: どの概念にも当てはまらない「モノ」がある**

管理したいものが人でも、Parts でも、Work でも、Opportunity でも、Article でもない場合。

- 保守契約（有効期間、カバー範囲、SLA 条件を持つ）
- ライセンスキー（発行日、有効期限、割り当て先を持つ）
- 物理的な資産・機器（シリアル番号、設置場所、保証期間を持つ）
- 社内の申請書類（出張申請、購買申請）

**パターン2: 1つの親に対して複数のレコードが独立して存在する**

「Account に契約情報をフィールドとして追加すればいいのでは？」→ 1社に対して契約が複数あり、それぞれ別の期間・条件で管理が必要なら、フィールド追加では表現できません。

- 1社に複数の保守契約がある
- 1つの Opportunity に複数の見積書がある

**パターン3: 固有のライフサイクル（状態の流れ）がある**

そのデータが独自のステージ遷移を持つ場合。

- 見積書: 下書き → 提出 → 承認 → 受注/失注
- 稟議: 起案 → 部長承認 → 経理承認 → 完了
- 採用候補者: 応募 → 書類選考 → 面接 → オファー → 入社/辞退

**パターン4: 外部システムに対応する標準オブジェクトがない**

外部ツールから取り込むデータの中で、標準オブジェクトに対応するものがない場合。

- CRM の見積（Quote）、注文（Order）、請求書（Invoice）
- 社内 ERP の在庫データ

**パターン5: 大量データとして独立して検索・分析したい**

標準オブジェクトにフィールドとして持たせることは可能だが、量や性質が合わない場合。

- IoT センサーからのアラートデータ
- EC の注文データ

### カスタムオブジェクトを作らないほうがよい場合

| やりたいこと | カスタムオブジェクトは不要。こうする |
|---|---|
| Ticket の種類を細かく分けたい | サブタイプを作る |
| Issue に「原因」「影響範囲」を付けたい | カスタムフィールドを追加する |
| 「VIP顧客」を区別したい | タグやカスタムフィールドを追加する |
| Opportunity の種類を分けたい | サブタイプを作る |

### 判断フロー

```
管理したいデータがある
│
├── 人や組織？ → Account / RevUser / DevUser
│
├── 提供物の構造？ → Parts（最大4階層）
│
├── やるべき仕事？ → Ticket / Issue / Enhancement / Incident
│   │
│   └── 足りない情報がある？
│       ├── フィールドを数個足せば済む → カスタムフィールド
│       └── 種類によって情報が違う → サブタイプ作成
│
├── 商談？ → Opportunity
│
├── ドキュメント？ → Article
│
└── 上のどれでもない独立した「モノ」？
    ├── 1つの親に複数レコードが必要？ → カスタムオブジェクト
    ├── 固有のライフサイクルがある？ → カスタムオブジェクト
    └── 大量データとして独立検索が必要？ → カスタムオブジェクト
```

## 7. 外部ツールからの取り込み — 何がどこに入るか

用語レベルの対照表は [s03](/ja/s03) に、AirSync/ADaaS の実装詳細は [ADaaS リファレンス](/ja/reference/adaas-reference) にあります。ここでは「データの入り先」を整理します。

### CRM系（Salesforce / HubSpot / Dynamics 等）

| ツール側のデータ | DevRev での入り先 | 備考 |
|---|---|---|
| 取引先 / Company | Account | |
| 連絡先 / Contact | RevUser | |
| リード / Lead | RevUser | まだ会社が不明でも OK |
| 商談 / Opportunity / Deal | Opportunity | |
| 見積 / Quote | カスタムオブジェクト | 標準に無い概念 |
| 注文 / Order | カスタムオブジェクト | 同上 |

### サポート系（Zendesk / Freshdesk / Intercom 等）

| ツール側のデータ | DevRev での入り先 | 備考 |
|---|---|---|
| チケット / Case / Conversation | Ticket | |
| 組織 / Organization | Account | |
| ユーザー / End User | RevUser | |
| エージェント / Agent | DevUser | |
| ナレッジ記事 | Article | |

### 開発系（Jira / Azure DevOps / Linear 等）

| ツール側のデータ | DevRev での入り先 | 備考 |
|---|---|---|
| Issue (Bug / Story / Task) | Issue（サブタイプで区別） | |
| Epic | Enhancement | 複数 Issue の親 |
| User | DevUser | |
| Sprint | DevRev の Sprint 機能 | |

### コード管理（GitHub / GitLab）

| ツール側のデータ | DevRev での入り先 | 備考 |
|---|---|---|
| Repository | Builder Parts | |
| Pull Request / Commit | Issue にリンク | 進捗を示すイベント |

### データウェアハウス（Snowflake / BigQuery 等）

| ツール側のデータ | DevRev での入り先 | 備考 |
|---|---|---|
| 顧客テーブル | Account / RevUser | |
| 商談テーブル | Opportunity | |
| それ以外のテーブル | カスタムオブジェクト | テーブルごとに自動生成可能 |

### マッピング時の考え方

**「同じ概念か？」で判断する:**

- ツール側の「取引先」と DevRev の Account → 同じ概念 → そのまま対応
- ツール側の「Issue」と DevRev の Issue → 同じ概念 → そのまま対応
- ツール側の「Quote（見積）」→ DevRev に対応する標準オブジェクトが無い → カスタムオブジェクト

**対応がない場合の選択肢:**

1. カスタムオブジェクトを作る
2. 既存オブジェクトのサブタイプ + カスタムフィールドで表現する
3. そもそも取り込まない（DevRev で管理する必要がないデータ）

## 8. 業種別の設計例

### 例1: BtoB SaaS 企業

```
Account  = 契約顧客企業
RevUser  = 顧客企業の担当者、エンドユーザー
DevUser  = 自社の営業、CS、開発者

Product    = 「マーケティングオートメーション」
Capability = 「メール配信」「リード管理」「分析」
Feature    = 「ABテスト」「スコアリング」「ファネル分析」

Ticket      = 問い合わせ・要望・不具合報告
Issue       = 開発チームの作業
Enhancement = 四半期ごとの改善テーマ
Opportunity = 新規契約・アップセル
Incident    = サービスダウン

Article = ヘルプセンターの記事、社内対応ナレッジ

→ カスタムオブジェクトは不要なケースが多い
  （契約管理が複雑なら「サブスクリプション」を検討）
```

### 例2: 製造業（産業機器メーカー）

## 9. 設計の進め方

### ステップ1: データの棚卸し

社内で管理しているデータをリストアップします。CRM、サポートツール、プロジェクト管理ツール、スプレッドシート、基幹システム、紙の台帳が確認対象です。「顧客マスター」「商談一覧」「問い合わせ履歴」「開発バックログ」のようなレベルで十分です。

### ステップ2: 振り分け

各データを5つの問いに沿って振り分けます。

| あなたのデータ | どのオブジェクトか |
|---|---|
| 顧客マスター | Account + RevUser |
| 商談一覧 | Opportunity |
| 問い合わせ履歴 | Ticket |
| 開発バックログ | Issue |
| ナレッジ記事 | Article |
| 保守契約台帳 | → 既存に当てはまらない → カスタムオブジェクトの候補 |
| 社員名簿 | DevUser |
| 製品カタログ | Parts（最大4階層） |

### ステップ3: カスタムオブジェクトの候補を精査

「既存に当てはまらない」と判断したものについて確認します。

1. **本当に標準オブジェクトで表現できないか？** → カスタムフィールドやサブタイプで代用できないか再検討
2. **DevRev で管理する必要があるか？** → 他のオブジェクトとの繋がりが無いなら、外部システムのままでも良いかもしれない
3. **どのオブジェクトと繋がるべきか？** → カスタムオブジェクトを作る場合、Account や Ticket とどうリンクするか考える

## よくある設計上の疑問

### 1人の担当者が複数の会社に関わっている

よくあるケースです。DevRev では、1つの RevUser レコードを複数の Account に紐づけられます。同じ人を会社ごとにコピーして作る必要はありません。パートナー企業の営業が複数のエンド顧客を担当している、グループ会社間で兼務している、転職して取引先の会社が変わった、といった場合が典型例です。

### 最初から完璧に設計しないとダメか

いいえ。後からいくらでも拡張できます。最初は Account（顧客企業）、Parts を2〜4階層、Ticket、Issue だけで始めて問題ありません。使い始めてから「こういう情報も必要だ」とわかった時点でカスタムフィールドを追加したり、カスタムオブジェクトを作れます。

### 既に他のツールで管理しているデータも DevRev に入れるべきか

「DevRev の他のデータとの繋がりから価値が生まれるか」で判断してください。

- **入れるべき:** 顧客データ、問い合わせ、開発作業、商談 → これらが繋がることで全体像が見える
- **入れなくてもよい:** 給与データ、経費精算データなど → 顧客対応や製品開発との繋がりが薄い

### 1つのデータが2つのオブジェクトに当てはまる気がする

以下の基準で判断してください。

- お客様が直接関わるか？ → Ticket 寄り
- 社内だけで完結するか？ → Issue 寄り
- お金の話か？ → Opportunity 寄り

迷った場合は「最も多い使い方」に合わせてください。例外的なケースはサブタイプやカスタムフィールドで吸収できます。

### サブタイプとカスタムオブジェクトの違い

- **サブタイプ** = 同じオブジェクトだけど、種類によって追加情報が少し違う（例: Issue のサブタイプ「バグ」には「原因分類」「再現手順」がある）
- **カスタムオブジェクト** = 既存のどのオブジェクトとも根本的に性質が異なるもの（例: 「保守契約」は Work でも Opportunity でもない独立した概念）

判断基準: 既存オブジェクトの「バリエーション」ならサブタイプ。既存オブジェクトとは「まったく別の概念」ならカスタムオブジェクト。

## まとめ

1. **5種類のオブジェクトがある** — 人（Account / RevUser / DevUser）、Parts、Work（Ticket / Issue / Incident）、Opportunity、Article
2. **ほとんどのデータは標準オブジェクトに入る** — 足りない情報はカスタムフィールドやサブタイプで補う
3. **標準に入らないものだけカスタムオブジェクトを作る** — 「概念として別物かどうか」で判断する
4. **オブジェクト同士の繋がりが最大の価値** — 個別管理なら既存ツールで十分。DevRev の力は繋がりにある
5. **最初から完璧にしなくてよい** — 使いながら育てる
