# AirSyncコネクタを自作する（ADaaS）

> "既製コネクタにない外部システムとの同期も、ADaaS SDKを使えば自分で構築できる。"

## このセッションで学ぶこと

**始める前の状態：**
- ADaaS/AirSyncコネクタ開発の経験がない
- カスタムSnap-inとAirSyncコネクタの違いを知らない
- 大量データの同期が必要だが方法がわからない

**このセッション後の状態：**
- カスタムSnap-in（s13）とAirSyncコネクタ（ADaaS）の使い分けを説明できます
- ADaaS SDKを使ったコネクタのプロジェクト構造を理解しています
- Extractionの4フェーズとその役割を説明できます
- Initial Domain Mapping（IDM）の目的と作成方法を理解しています
- 制御プロトコル（DONE/PROGRESS/ERROR）とタイムアウト管理の仕組みを説明できます
- Batch-then-Pushパターンと状態リセットパターンを実装できます
- Computer Readyコネクタの5つの要件を説明できます
- よくある失敗パターンとその対策を知っています

## 問題

マーケットプレイスのAirSyncコネクタは主要ツールをカバーしていますが、社内システムや特殊なSaaSとの同期には対応していません。

「自社の基幹システムのデータをDevRevに取り込みたい」「特定のSaaSの履歴データを丸ごとインポートしたい」「双方向同期を独自システムに対応させたい」こうしたニーズを、既製コネクタだけでは解決できません。

また、s13で学んだカスタムSnap-in（Webhook/イベント系）はリアルタイムな処理が得意ですが、大量の履歴データの取り込みや双方向同期には向いていません。

## 解決策

**ADaaS（Airdrop as a Service）SDK** を使ってAirSyncコネクタを自作します。

ADaaS SDKは `@devrev/ts-adaas` パッケージとして提供されており、テンプレートリポジトリをベースにTypeScriptでコネクタを開発します。AirSyncプラットフォームのOrchestrator・状態管理・リトライなどの複雑な制御が、SDKとプラットフォームによって自動的に処理されます。

## 仕組み

### AirSyncコネクタ vs カスタムSnap-in

| | カスタムSnap-in（s13） | AirSyncコネクタ（ADaaS） |
|---|---|---|
| **使うSDK** | Snap-in SDK（一般） | `@devrev/ts-adaas` |
| **動作トリガー** | Webhook、イベント、Cron | AirSync Orchestratorからのイベント |
| **データフロー** | 任意 | 構造化されたETLパイプライン |
| **状態管理** | 自前で実装 | SDKが提供（S3バックエンド） |
| **フィールドマッピング** | なし | あり（IDM + Blueprint + レシピUI） |
| **長時間実行** | 自前で対応 | Orchestrator + 制御プロトコルで自動管理 |

**ADaaS（AirSyncコネクタ）を選ぶとき:**

| この要件があれば ADaaS を使う | カスタムSnap-inで十分 |
|---|---|
| 定期的な大量データのインポートが必要 | リアルタイム/イベント駆動で少量のデータを都度取得・送信する |
| 履歴データを丸ごと取り込みたい | データエンリッチメント（既存レコードに情報を付加するだけ） |
| 双方向同期が必要 | 外部からのWebhook受信→1件処理のような単発フロー |
| フィールドマッピングUI（レシピ）が必要 | 履歴データ不要で、新規イベントのみ処理すれば十分 |

### 用語の整理

AirSync関連には歴史的経緯から複数の用語が混在しています。

| 用語 | 意味 |
|------|------|
| **AirSync** | データ同期エンジン全体のブランド名（旧称: Airdrop） |
| **ADaaS** | AirSyncプラットフォームにアクセスするためのSDK/サービス層の技術名称。開発者向けに残存 |
| **AirSyncコネクタ** | ADaaS SDKを使って構築されたSnap-in（ADaaSコネクタと同義） |
| **ネイティブコネクタ** | Jira/SFDCなど、プラットフォームに直接組み込まれたコネクタ（ADaaS登場前） |

### 全体アーキテクチャ

```
[外部システム] ←→ [ADaaS Snap-in (Lambda)] ←→ [AirSync Platform]
                                                       │
                                              ┌────────┴────────┐
                                              │                 │
                                        [Orchestrator]    [Recipe Manager]
                                              │                 │
                                        制御プロトコル     マッピング管理
                                              │
                                          [DevRev]
                                       (Knowledge Graph)
```

ADaaS Snap-inはAWS Lambda上で動作し、AirSync Orchestratorの指示に従いフェーズごとに起動されます。

### ドメインモデル

#### 外部システム側

| 概念 | 説明 | 例 |
|------|------|-----|
| **External System Name** | ツールの識別子 | Jira, Salesforce, Zendesk |
| **External System Instance** | ツールの個別テナント | yourcompany.atlassian.net |
| **External Sync Unit** | インスタンス内のデータコンテナ | Jiraのプロジェクト, SFDCのorg全体 |

外部システムには2種類あります:

- **Single Workspace型**（Salesforce, Zendesk）: org全体が1つのExternal Sync Unit
- **Multiple Workspace型**（Jira, Linear）: プロジェクト/チームごとに複数のExternal Sync Unit

#### DevRev側

| 概念 | 説明 |
|------|------|
| **Sync Unit** | External Sync Unit + Sync Run設定 + Blueprint で構成される抽象単位 |
| **Sync Run** | Sync Unitに基づく1回の同期実行 |
| **Blueprint** | 複数レコードタイプの変換関数を定義 |
| **Keyring** | 認証情報の保管（UIでは「Connection」として表示） |

---

### Extractionライフサイクル（4フェーズ）

E2DR（External to DevRev）方向の同期は4フェーズで構成されます。

| # | フェーズ | 目的 | 実行タイミング |
|---|---------|------|--------------|
| 1 | **External Sync Units Extraction** | 同期対象ユニットの取得 | 初回同期のみ |
| 2 | **Metadata Extraction** | 外部システムのスキーマ取得 | 毎回 |
| 3 | **Data Extraction** | レコードデータの抽出 | 毎回 |
| 4 | **Attachments Extraction** | 添付ファイルの抽出 | 毎回 |

DR2E（DevRev to External）方向はExtractionフェーズを持ちません。Data Loading → Attachments Loading の2フェーズで構成されます。

### 設計の進め方

コードを書き始める前に、以下を明確にしてください。

**1. 対象の外部システムのAPI調査**

| 調査項目 | 確認すること |
|---------|------------|
| 認証方式 | OAuth / APIキー / PAT。Keyringの種類を決定する |
| レートリミット | 1分あたりの上限、`Retry-After`ヘッダーの有無。リトライ設計に直結 |
| ページネーション | カーソル / オフセット / ページ番号。状態管理の設計に影響 |
| 変更検出 | `updated_at`フィールドの有無。増分同期の可否を決定する |

**2. 同期対象の設計**

- どのオブジェクト（レコードタイプ）を同期するか
- 各レコードタイプのスコープ（External System Scoped / External Sync Unit Scoped）
- DevRevのどのオブジェクトタイプにマップするか（issue / ticket / account / custom_object）

**3. 同期方向と頻度の決定**

| 判断項目 | 選択肢 |
|---------|--------|
| 方向 | E2DR only（一方向）/ 2-way（双方向） |
| 初回 | 全量インポート / Time-scoped（直近N日分でテスト後に全量） |
| 増分 | 外部APIがdelta取得をサポートするか → high-water mark設計 |

**4. IDの設計**

| ID | 何を使うか |
|----|----------|
| `external_org_id` | サブドメイン / テナントGUID / org ID（不変であること） |
| External Sync Unitの`id` | プロジェクトID / workspace GUID（不変であること） |

### 開発の始め方

**テンプレートと必要なツール**

```bash
# テンプレートリポジトリをクローン（SDKはpackage.jsonに含まれている）
git clone https://github.com/devrev/airdrop-template
cd airdrop-template/code
npm install
```

**プロジェクト構造**

```
manifest.yaml                          # Snap-in設定
code/
  src/
    functions/
      external-system/
        external_domain_metadata.json  # 外部ドメインメタデータ
        initial_domain_mapping.json    # 初期ドメインマッピング（IDM）
        http-client.ts                 # 外部API通信
        data-normalization.ts          # データ正規化（E2DR方向）
        data-denormalization.ts        # データ非正規化（DR2E方向）
      extraction/
        workers/
          external-sync-units-extraction.ts
          metadata-extraction.ts
          data-extraction.ts
          attachments-extraction.ts
      loading/
        workers/
          load-data.ts
          load-attachments.ts
```

**エントリーポイントの実装例**

```typescript
import { AirdropEvent, spawn } from "@devrev/ts-adaas";
import initialDomainMapping from "../external-system/initial_domain_mapping.json";

export interface ExtractorState {
  todos: { completed: boolean };
  users: { completed: boolean };
  attachments: { completed: boolean };
}

export const initialExtractorState: ExtractorState = {
  todos: { completed: false },
  users: { completed: false },
  attachments: { completed: false },
};

const run = async (events: AirdropEvent[]) => {
  for (const event of events) {
    await spawn<ExtractorState>({
      event,
      initialState: initialExtractorState,
      initialDomainMapping,
      baseWorkerPath: __dirname,  // SDKがイベントタイプに基づきworkerを自動解決
    });
  }
};

export default run;
```

> **v1.13.0以降**: `workerPath` は非推奨。`baseWorkerPath: __dirname` を使用すること。

**開発の推奨順序**

1. 外部システムからデータを取得するHTTPクライアントを実装する
2. External Domain Metadata（EDM）の初期バージョンを作成し、`chef-cli`で検証する（[chef-cli ドキュメント](https://developer.devrev.ai/airsync/initial-domain-mapping)）
3. Initial Domain Mapping（IDM）を準備する
4. **まず1つのアイテムタイプ（ユーザー推奨）から開始**し、動作確認後に他のタイプへ進む

> **EDM（External Domain Metadata）とは**: 外部システムのスキーマ（レコードタイプ名、フィールド名、フィールドの型、enum値、参照関係）をJSON形式で宣言するファイルです。IDMはこのEDMを前提に作成されます。詳細は [Metadata Extraction ドキュメント](https://developer.devrev.ai/airsync/metadata-extraction) を参照してください。

### Initial Domain Mapping（IDM）

IDMは外部データスキーマとDevRevネイティブレコードタイプの関係を定義するファイルです。一度設定すると、このSnap-inをインストールした全ユーザーに適用されます。

```json
{
  "additional_mappings": {
    "record_type_mappings": {
      "todos": { "default_mapping": { "...": "..." }, "possible_record_type_mappings": [] },
      "users": { "default_mapping": { "...": "..." }, "possible_record_type_mappings": [] }
    },
    "format_version": "v0.2.0",
    "devrev_metadata_version": 13
  }
}
```

IDMの作成方法:

| 方法 | コマンド | 用途 |
|------|---------|------|
| **Chef UI** | `chef-cli configure-mappings --env prod --idm path/to/idm.json` | 対話的。プロダクション推奨 |
| **MCP** | AI支援マッピング（実験的） | ローカルで高速プロトタイピング |

`spawn()` に `initialDomainMapping` を渡すと、SDKがSnap-inバージョン変更を検出した際、次回のstatefulイベント（metadata extraction等）のタイミングで自動的にインストールします。

### Object Mappers

外部オブジェクトとDevRevエンティティの対応関係を追跡する双方向ルックアップテーブルです。`adapter.mappers` 経由でアクセスします。

```typescript
// 外部ID → DevRevエンティティ検索
await adapter.mappers.getByExternalId({
  sync_unit: adapter.event.payload.event_context.sync_unit,
  external_id: "EXT-123",
  target_type: SyncMapperRecordTargetType.WORK,
});

// DevRevエンティティ → 外部ID検索（Loading時に使用）
const record = await adapter.mappers.getByTargetId({
  sync_unit: adapter.event.payload.event_context.sync_unit,
  target: devrevWorkId,
});
const externalId = record.data.sync_mapper_record.external_ids[0];

// 新規マッピング作成
await adapter.mappers.create({
  sync_unit: adapter.event.payload.event_context.sync_unit,
  external_ids: [externalSystemId],
  targets: [devrevEntityId],
  status: SyncMapperRecordStatus.OPERATIONAL,
});
```

> **2-way syncを実装する場合**: Loading ワーカー（`load-data.ts`）で DevRev からの変更データを受け取り、外部システムに書き戻します。`getByTargetId` で外部IDを取得し、外部APIを呼び出してデータを更新します。更新後は `external_versions` に `modified_date` を記録してループを防止します。詳細は [Load data ドキュメント](https://developer.devrev.ai/airsync/load-data) を参照してください。

双方向同期でのループ防止（DevRev起因の変更が外部から再抽出されるのを防ぐ）:

```typescript
await adapter.mappers.update({
  id: syncMapperRecordId,
  sync_unit: adapter.event.payload.event_context.sync_unit,
  external_versions: {
    add: [{ recipe_version: 0, modified_date: "2024-01-15T10:30:00Z" }],
  },
});
```

### 制御プロトコルと長時間実行

ADaaS Lambdaは数百万レコードの処理に15分以上かかることがあります。SDKは3つのメカニズムで対応しています。

**1. 状態永続化（State Persistence）**

処理の進捗状態をS3バックエンドに保存します。Lambda再起動後も前回の続きから処理を再開できます。

**2. 制御プロトコル**

各Lambda実行の完了時（正常終了またはタイムアウト発生時）に、Snap-inはプラットフォームにステータスを返します:

| メッセージ | 動作 |
|-----------|------|
| **DONE** | 処理完了 |
| **PROGRESS** | 追加処理が必要。同じ入力でLambdaを再起動 |
| **ERROR** | エラー発生。エンドユーザーに表示 |

**3. タイムアウト管理**

SDKがLambdaタイムアウトの終了前（約90%経過時点）にソフトタイムアウトをセットします。タイムアウト発生時は `adapter.isTimeout` が `true` になります。

```
1. Lambda起動
2. SDKがソフトタイムアウトタイマーをセット（ハードタイムアウトの90%）
3. ソフトタイムアウト → adapter.isTimeout = true
4. onTimeoutコールバックが呼ばれる
5. onTimeout は必ず PROGRESS イベントを発行（ERROR/DONE ではない）
6. プラットフォームがLambdaを再起動し、保存した状態から再開
```

> **重要**: `adapter.isTimeout` が `true` になった後の `repo.push()` はサイレントに無視されます（S3にアップロードされない）。タイムアウト後にpushしてもデータは消えます。

### 同期の種類

`event_context.mode` で現在の同期種別（`SyncMode.INITIAL` / `SyncMode.INCREMENTAL`）を判定します。各モードの抽出ロジックの詳細は [ADaaS 実装リファレンス — Sync Modes](/ja/adaas-reference#sync-modes-リファレンス) を参照してください。

> **注**: AirSyncでは「2-way sync」はDR2E方向（逆方向）の同期を指します。E2DR（Forward）と DR2E（Reverse）の両方を有効にすることで双方向同期が実現します。

| 種類 | 方向 | 説明 |
|------|------|------|
| **Initial Sync** | E2DR | 初回。全データを抽出して取り込む |
| **Incremental Sync** | E2DR | 前回同期以降の差分のみ取り込む |
| **2-way Sync (Reverse)** | DR2E | DevRevの変更を外部システムに書き戻す |
| **Time-scoped Sync** | E2DR | 指定タイムスタンプ以降のデータのみ抽出 |

**Time-Scoped Syncの実装**

Computer Readyコネクタには `TIME_SCOPED_SYNCS` のサポートが必須です。`event_context` から取得できる2つのパラメータを正しく処理します。

| パラメータ | 説明 |
|-----------|------|
| `extract_from` | 抽出の開始タイムスタンプ |
| `reset_extract_from` | `true` のとき、増分同期でも `extract_from` から再抽出する |

```typescript
const { extract_from, reset_extract_from, mode } = adapter.event.payload.event_context;

let sinceTimestamp: string | null;
if (mode === SyncMode.INITIAL) {
  // 初回同期: extract_from があればそこから、なければ全データ抽出
  sinceTimestamp = extract_from ?? null;
} else if (reset_extract_from) {
  // 増分同期 + リセット指示: extract_from から再抽出
  sinceTimestamp = extract_from ?? null;
} else {
  // 通常の増分同期: 前回成功時のタイムスタンプを使用
  sinceTimestamp = adapter.state.lastSuccessfulSyncStarted;
}

// sinceTimestamp が null の場合は全データ抽出（フィルタなし）
```

manifestへの宣言も必要です:

```yaml
imports:
  - slug: airdrop-my-connector-snap-in
    capabilities:
      - TIME_SCOPED_SYNCS
```

### 実装パターンとベストプラクティス

**Batch-then-Push パターン（推奨）**

ページ全体を処理してからpushし、push成功後にカーソルを進めます。

```typescript
for (const item of page.items) {
  if (adapter.isTimeout) {
    // タイムアウト: pushせずreturn → 次回呼び出しで同じページから再開
    return;
  }
  batch.push(normalize(item));
}

// ページ全体の処理が完了した場合のみpush
await adapter.getRepo('entity').push(batch);
state.cursor = page.nextCursor; // push成功後にのみカーソルを進める
```

**状態リセットパターン**

新しいSync Run開始時に完了フラグをリセットしますが、増分同期ではhigh-water markを保持します。

```typescript
if (event_type === EventType.StartExtractingData) {
  // 完了フラグとカーソルはリセット。lastModified（high-water mark）は上書きしない
  adapter.state.entityA = { ...adapter.state.entityA, completed: false, nextPage: 0 };
  adapter.state.entityB = { ...adapter.state.entityB, completed: false, nextPage: 0 };
}
```

**リトライとバックオフ**

最大リトライ待ち時間の合計がLambdaタイムアウトを超えないよう注意します（5回×最大30秒=150秒は安全、5回×60秒=300秒は危険）。

### Manifest設定の重要ポイント

**external_org_id の決定**

`external_org_id` はSync Mapperレコードのスコーピングに使われます。間違えると顧客間でデータが混在します。manifestの `external_system_name` とセットで設定します。

| 条件 | 設定方法 |
|------|---------|
| サブドメインでインスタンスを識別する場合 | `is_subdomain: true`（subdomain値が自動的に `external_org_id` になる） |
| それ以外 | `organization_data` でAPIを呼び出し、レスポンスからユニークIDを抽出 |

- 顧客インスタンスごとにユニークかつ**不変**であること
- 表示名など変更可能な値の使用はNG
- 静的文字列のハードコードはNG

**token_verification**

接続作成時にクレデンシャルを検証するエンドポイントを指定します。認証必須のエンドポイントを使うこと（常に200を返すパブリックエンドポイントではNG）。

**Computer Readyコネクタの要件**

ユーザー介入なしで完全セットアップできる「Computer Ready」コネクタには以下が必要です。

| # | 要件 | 内容 |
|---|------|------|
| 1 | TIME_SCOPED_SYNCS | manifestで宣言し、`extract_from` を正しく処理する |
| 2 | OAuth認証 | 手動APIキーではなくOAuthフローを使用 |
| 3 | パーミッション認識 | 接続の権限を検証し、不足時にエラーを報告 |
| 4 | デフォルト設定 | 必須カスタム入力なしでインストール可能 |
| 5 | 自動レシピ送信 | 完全な `initial_domain_mapping.json` を `spawn()` に渡す |

```yaml
imports:
  - slug: airdrop-my-connector-snap-in
    capabilities:
      - TIME_SCOPED_SYNCS
    allowed_connection_types:
      - my-system-oauth
```

### よくある失敗

特に影響が大きい3つを挙げます。全10パターンは [ADaaS 実装リファレンス](/ja/adaas-reference) を参照してください。

| 失敗 | 影響 | 対策 |
|------|------|------|
| `external_org_id` をハードコードまたは変更可能な値を使用 | 顧客間でデータ重複 / リネーム時にデータ重複 | 外部APIから不変のGUID等を動的に取得する |
| Sync Run間で完了フラグをリセットしない | 2回目以降のSyncが何も抽出しない | `StartExtractingData`で必ずリセット |
| `onTimeout`でErrorイベントを発行 | 再開可能なタイムアウトが致命的エラーになる | 必ずProgressイベントを発行 |

## 試してみる

サンプルコネクタを動かして、ADaaS SDKの動作を体験しましょう。

```
【ADaaS コネクタ開発演習】

基礎確認
□ 1. カスタムSnap-in vs AirSyncコネクタの使い分け表を見て、自社のユースケースを分類した
□ 2. https://github.com/devrev/airdrop-template のREADMEを読んだ
□ 3. プロジェクト構造（manifest.yaml / workers / IDM）を把握した

開発環境
□ 4. テンプレートリポジトリをクローンした
□ 5. npm install を実行し、@devrev/ts-adaas が依存に含まれていることを確認した
□ 6. spawn() にbaseWorkerPath と initialDomainMapping を渡す実装を確認した

Extractionフェーズ
□ 7. 4つのフェーズ（ESU/Metadata/Data/Attachments）のworkerファイルの場所を確認した
□ 8. adapter.isTimeout を各ループ先頭でチェックするコードを書いた
□ 9. push() 成功後にのみカーソルを進めるBatch-then-Pushパターンを実装した

状態管理
□ 10. StartExtractingDataイベントで完了フラグをリセットするコードを書いた
□ 11. INCREMENTAL時はhigh-water markを保持することを確認した
□ 12. onTimeout が ERROR ではなく PROGRESS を発行することを確認した

Manifest設定
□ 13. external_org_id を動的に取得する仕組みを設計した（is_subdomain or organization_data）
□ 14. token_verification に認証必須のエンドポイントを設定した
□ 15. Computer Ready要件のチェックリストを確認した

【設計メモ】
対象の外部システム: ______________
External Sync Unitの種類: Single Workspace型 / Multiple Workspace型
external_org_idの取得方法: is_subdomain / organization_data（エンドポイント: ______________）
同期するレコードタイプ: ______________
各レコードタイプのスコープ: External System Scoped / External Sync Unit Scoped
双方向同期が必要か: はい / いいえ
```

このセッションは [s13（Snap-inを自作する）](/ja/s13) の応用です。Snap-inの基本開発フローを先に学んでから取り組むことを推奨します。

実装中に参照したい詳細情報（エンティティスコープ、サブタイプ管理、Mixins一覧、バリデーションチェックリスト完全版）は [ADaaS 実装リファレンス](/ja/adaas-reference) にまとめています。
