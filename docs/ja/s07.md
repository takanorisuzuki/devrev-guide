# 開発サイクルを管理する

> "TicketからIssueを作り、Partに紐付け、リリースまで追跡する。サポートと開発が1つのツールでつながる。"

## 問題

サポートチームが受けた顧客の不具合報告が、エンジニアに届くまでに何日もかかる。届いたとしても、元のチケットとの紐付けが切れてしまい、修正が完了しても顧客に通知するのを忘れる。Jiraのチケット番号とZendeskのチケット番号を手動で紐付ける運用は、規模が大きくなると破綻する。

## 解決策

DevRevのBuild Appでは、サポートチケットから直接Issueを作成し、そのIssueをPartに紐付けてリリースまで追跡できる。Ticket→Issue→Part→Releaseの一連の流れが同一プラットフォーム内で完結する。

## 仕組み

**Ticket→Issue連携**

サポートチケットから「Create Issue」を実行すると、そのチケットに紐付いたIssueが作成される。この連携により以下が実現する。

- Issueが解決されると、紐付いたTicketの起票者に自動通知できる
- 同じ問題を報告している複数のTicketを1つのIssueに集約できる
- Issueの進捗がTicket側からも確認できる

**Issue管理**

DevRevのIssueには以下の属性がある。

| 属性 | 説明 |
|------|------|
| Type | Bug / Feature Request / Task など |
| Priority | Blocker / High / Medium / Low |
| Part | 紐付くProduct/Feature |
| Assignee | 担当者 |
| Sprint | 所属するスプリント |
| Stage | Open / In Progress / In Review / Done |

**スプリント管理**

Build Appではスプリント（開発サイクル）を定義できる。スプリントにIssueを割り当て、バーンダウンチャートで進捗を追跡する。バーンダウンチャートとは、スプリント期間中の残作業量を時間軸で表したグラフで、予定通りに消化できているかを一目で確認できる。

**リリース管理**

リリースはIssueの集合として管理する。リリースノートの自動生成も可能で、そのリリースに含まれるIssueの一覧がまとめられる。

この一連の流れにより「顧客が報告した問題が、いつのリリースで修正されるか」をトラッキングできるようになる。

## このセッションで学ぶこと

**始める前の状態：**
- サポートと開発が別ツールで管理されている
- Ticket→Issueの連携が手動
- リリースに含まれる修正内容を把握するのに時間がかかる

**このセッション後の状態：**
- TicketからIssueを作成し、Partに紐付ける操作ができる
- スプリントにIssueを割り当て、進捗を確認できる
- リリース管理の流れを理解し、リリースノートの自動生成を利用できる

## 試してみる

既存のテストチケットからIssueを1件作成し、Partに紐付けてみよう。

```
【開発サイクル演習】

□ 1. テストチケットを1件選択した
□ 2. チケットからIssueを作成した
□ 3. Issueに適切なPartを紐付けた
□ 4. IssueにPriorityとTypeを設定した
□ 5. スプリントを作成し、Issueを割り当てた
□ 6. IssueのStageをIn Progress→Doneに変更した
□ 7. チケット側からIssueの状態変更を確認した
```
