# 開発サイクルを管理する

> "TicketからIssueを作り、Partに紐付け、リリースまで追跡する。サポートと開発が1つのツールでつながる。"

## 問題

サポートチームが受けた顧客の不具合報告が、エンジニアに届くまでに何日もかかります。

届いたとしても、元のチケットとの紐付けが切れてしまい、修正が完了しても顧客に通知するのを忘れます。

Jiraのチケット番号とZendeskのチケット番号を手動で紐付ける運用は、規模が大きくなると破綻します。

## 解決策

DevRevのComputer for Buildersでは、サポートチケットから直接Issueを作成し、そのIssueをPartに紐付けてリリースまで追跡できます。

Ticket→Issue→Part→Releaseの一連の流れが同一プラットフォーム内で完結します。

## 仕組み

**Ticket→Issue連携**

サポートチケットから「Create Issue」を実行すると、そのチケットに紐付いたIssueが作成されます。この連携により以下が実現します。

- Issueが解決されると、紐付いたTicketの起票者に自動通知できます
- 同じ問題を報告している複数のTicketを1つのIssueに集約できます
- Issueの進捗がTicket側からも確認できます

**Issue管理**

DevRevのIssueには以下の属性があります。

| 属性 | 説明 |
|------|------|
| Type | Bug / Feature Request / Task など |
| Priority | Blocker / High / Medium / Low |
| Part | 紐付くProduct/Feature |
| Assignee | 担当者 |
| Sprint | 所属するスプリント |
| Stage | Open / In Progress / In Review / Done |

**Enhancement（エピック相当）**

**Enhancement** は **Part**（Product → … → Feature の下の customer RevPart）として階層に属し、複数の Issue を束ねる親としても使えます。Jira の Epic に近いロードマップ単位のまとまりです。Issue / Ticket 型の Work とは別オブジェクトですが振る舞いはハイブリッドになりえます。詳細は s03 と [オブジェクト構造リファレンス](/ja/reference/architecture) を参照してください。

**スプリント管理**

Computer for Buildersではスプリント（開発サイクル）を定義できます。スプリントにIssueを割り当て、バーンダウンチャートで進捗を追跡できます。バーンダウンチャートとは、スプリント期間中の残作業量を時間軸で表したグラフで、チームの進捗が予定通りであるかを一目で確認できます。

**リリース管理**

リリースはIssueの集合として管理します。リリースノートの自動生成も可能で、そのリリースに含まれるIssueの一覧がまとめられます。

この一連の流れにより「顧客が報告した問題が、いつのリリースで修正されるか」をトラッキングできるようになります。

## このセッションで学ぶこと

**始める前の状態：**
- サポートと開発が別ツールで管理されている
- Ticket→Issueの連携が手動
- リリースに含まれる修正内容を把握するのに時間がかかる

**このセッション後の状態：**
- TicketからIssueを作成し、Partに紐付ける操作ができます
- スプリントにIssueを割り当て、進捗を確認できます
- リリース管理の流れを理解し、リリースノートの自動生成を利用できます

## 試してみる

既存のテストチケットからIssueを1件作成し、Partに紐付けてみましょう。

```
【開発サイクル演習】

□ 1. テストチケットを1件選択した
□ 2. チケットからIssueを作成した
□ 3. Issueに適切なPartを紐付けた
□ 4. IssueにPriorityとTypeを設定した
□ 5. スプリントを作成し、Issueを割り当てた
□ 6. IssueのStageをIn Progress→Doneに変更した
□ 7. チケット側からIssueの状態変更を確認した
```

次のセッションでは、管理者設定とアクセス制御を学びます。

## 用語

| 用語 | 説明 |
|------|------|
| **Issue** | 開発チームが対応すべき課題。Bug / Feature Request / Task などのTypeを持つ |
| **Sprint** | 開発サイクルの期間単位。Issueをスプリントに割り当てて進捗を管理する |
| **バーンダウンチャート** | スプリント期間中の残作業量を時間軸で表したグラフ |
| **Stage** | Issueの進捗状態。Open / In Progress / In Review / Done など |
| **Enhancement** | Part階層の末端で、複数Issueを束ねるEpic相当のまとまり |
| **Release** | Issueの集合として管理されるリリース単位。リリースノートの自動生成が可能 |
| **Ticket→Issue連携** | サポートチケットから直接Issueを作成し、紐付ける仕組み。顧客の声を開発に届ける核心機能 |
