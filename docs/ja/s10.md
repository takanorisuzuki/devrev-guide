# ワークフローと自動化を設計する

> "トリガー・条件・アクション。シンプルな自動化から始め、段階的に高度化する。"

## 問題

繰り返し作業が多い。チケットが作成されるたびに手動でタグを付け、担当者をアサインし、通知を送る。こうした定型作業に1日の何割かの時間を費やしている。

手動オペレーションはコストがかかるだけでなく、ミスの原因にもなる。

## 解決策

DevRevのAutomations機能を使って、ルールベースの自動化を設計する。基本的なルール自動化はシンプルに始められるが、条件分岐やAIノードを組み合わせると非常に複雑にもなる。

このセッションの知識は、s14で学ぶAgent Studioの直接の前提知識になる。

## 仕組み

**自動化の基本構造**

DevRevの自動化は3つの要素で構成される。

1. **トリガー（Trigger）**: 何が起きたら自動化を発動するか
2. **条件（Condition）**: トリガーが発動した後、追加の条件判定を行うか
3. **アクション（Action）**: 条件を満たした場合に何を実行するか

**トリガーの例**

| トリガー | 説明 |
|---------|------|
| Ticket Created | チケットが作成されたとき |
| Ticket Updated | チケットが更新されたとき |
| Issue Stage Changed | Issueのステージが変わったとき |
| SLA Breached | SLA違反が発生したとき |

**条件の例**

- チケットのPriorityがBlockerである
- チケットのPartが「課金」に紐付いている
- チケットの本文に特定のキーワードが含まれている

**アクションの例**

- タグを追加する
- 担当者をアサインする
- Slackチャンネルに通知を送る
- チケットの優先度を変更する

**シンプルなワークフロー例**

「チケットが作成されたら、本文に『緊急』が含まれていれば、Priorityを自動でBlockerに設定し、専用Slackチャンネルに通知する」

これをトリガー・条件・アクションに分解すると以下のようになる。

- トリガー: Ticket Created
- 条件: 本文に「緊急」を含む
- アクション1: PriorityをBlockerに設定
- アクション2: Slackの#urgent-ticketsに通知

**複雑化の段階**

基本のルール自動化は直感的だが、以下の要素を組み合わせると複雑度が急上昇する。

- **AI条件ノード**: AIがチケットの内容を解析して条件判定する（キーワードマッチではなく意味ベースの判定）
- **並列パス**: 条件分岐後に複数の処理を同時実行する
- **待機ノード**: 一定時間待ってから次のアクションを実行する
- **ループ**: 条件が満たされるまで繰り返す

これらの高度な機能を理解しておくことが、s14のAgent Studioを学ぶ前提になる。

## このセッションで学ぶこと

**始める前の状態：**
- 自動化の設計経験がない、または手動ルールに依存している
- トリガー・条件・アクションの概念が整理されていない
- AI条件ノードの存在を知らない

**このセッション後の状態：**
- トリガー・条件・アクションの3要素で自動化を設計できる
- シンプルなワークフローを実際に作成できる
- AI条件ノードや並列パスなど高度な要素の存在と用途を理解している

## 試してみる

「チケットが作成されたら自動でタグを付ける」シンプルなワークフローを作ってみよう。

```
【ワークフロー設計演習】

ワークフロー名: ______________

トリガー: ______________
条件:     ______________
アクション: ______________

□ 1. ワークフローを作成した
□ 2. テストチケットを作成して動作確認した
□ 3. タグが自動的に付与されたことを確認した

【発展課題】
以下の自動化を設計してみよう（実装は任意）。

シナリオ: 「SLA違反が発生したら、チケットの担当者と
そのマネージャーにSlack通知を送り、Priorityを
1段階引き上げる」

トリガー: ______________
条件:     ______________
アクション1: ______________
アクション2: ______________
アクション3: ______________
```
