# ワークフローと自動化を設計する

> "トリガー・条件・アクション。シンプルな自動化から始め、段階的に高度化する。"

## 問題

繰り返し作業が多い状況です。

チケットが作成されるたびに手動でタグを付け、担当者をアサインし、通知を送ります。

こうした定型作業に1日の何割かの時間を費やしています。

手動オペレーションはコストがかかるだけでなく、ミスの原因にもなります。

## 解決策

DevRevのAutomations機能を使って、ルールベースの自動化を設計します。

基本的なルール自動化はシンプルに始められますが、条件分岐やAIノードを組み合わせると非常に複雑にもなります。

このセッションの知識は、s14で学ぶAgent Studioの直接の前提知識になります。

## 仕組み

**自動化の基本構造**

DevRevの自動化は3つの要素で構成されます。

1. **トリガー（Trigger）**: 何が起きたら自動化を発動するか
2. **条件（Condition）**: トリガーが発動した後、追加の条件判定を行うか
3. **アクション（Action）**: 条件を満たした場合に何を実行するか

**トリガーの例**

| トリガー | 説明 |
|---------|------|
| Ticket Created | チケットが作成されたとき |
| Ticket Updated | チケットが更新されたとき |
| Issue Stage Changed | Issueのステージが変わったとき |
| SLA Breached | SLA違反が発生したとき |

**条件の例**

- チケットのPriorityがBlockerである
- チケットのPartが「課金」に紐付いている
- チケットの本文に特定のキーワードが含まれている

**アクションの例**

- タグを追加する
- 担当者をアサインする
- Slackチャンネルに通知を送る
- チケットの優先度を変更する

**シンプルなワークフロー例**

「チケットが作成されたら、本文に『緊急』が含まれていれば、Priorityを自動でBlockerに設定し、専用Slackチャンネルに通知する」

これをトリガー・条件・アクションに分解すると以下のようになります。

- トリガー: Ticket Created
- 条件: 本文に「緊急」を含む
- アクション1: PriorityをBlockerに設定
- アクション2: Slackの#urgent-ticketsに通知

**設定場所**

ワークフローは **Settings > Workflows** から作成・管理します。s02 の Settings 地図で「Workflows」カテゴリとして確認済みの場所です。

---

**ワークフローノードの種類**

DevRev のワークフローエンジンは、単純な if-then だけでなく、以下のノード種別を組み合わせて複雑なフローを構築できます。

| ノード | 説明 | 用途例 |
|--------|------|--------|
| **Condition** | 条件分岐（if-then-else） | Priority が Blocker なら特別対応パスへ |
| **AI Condition** | AI がテキストを解析して意味ベースで判定 | 「怒っている顧客」「単なる質問」を分類 |
| **Parallel** | 複数のパスを同時実行 | 通知送信と担当者変更を同時に |
| **Wait** | 一定時間待機してから次のアクションを実行 | 24時間後に未応答チケットをリマインド |
| **Loop** | 条件が満たされるまで繰り返す | 承認が得られるまでリトライ |
| **Action** | 具体的な操作を実行 | タグ追加、担当者アサイン、通知送信 |

**複雑化の段階**

基本のルール自動化は直感的ですが、上記のノードを組み合わせると複雑度が急上昇します。

- **AI条件ノード**: AIがチケットの内容を解析して条件判定します（キーワードマッチではなく意味ベースの判定）
- **並列パス**: 条件分岐後に複数の処理を同時実行します
- **待機ノード**: 一定時間待ってから次のアクションを実行します
- **ループ**: 条件が満たされるまで繰り返します

これらの高度な機能を理解しておくことが、s14のAgent Studioを学ぶ前提になります。

---

**ワークフローの永続性とバージョニング**

DevRev のワークフローエンジンには、運用上重要な2つの特性があります。

1. **Durable State（永続的な状態）**: ワークフロー実行中にエラーやポーズが発生しても、状態が保持されます。問題解決後に**実行を再開（resume）**できるため、途中で止まっても最初からやり直す必要がありません。

2. **バージョニング**: ワークフローの新バージョンを publish しても、**既に実行中のランは旧バージョンで完走します**。新バージョンは次のトリガー以降に適用されます。これにより、運用中のフローを壊すことなく設計を改善できます。

```
ワークフロー v1 で実行開始
  ↓ 途中で v2 を publish
  ↓ 実行中のランは v1 のまま完走
  ↓ 次のトリガーから v2 が適用
```

この仕組みにより、本番環境でも安心してワークフローを反復改善できます。

## 試してみる

「チケットが作成されたら自動でタグを付ける」シンプルなワークフローを作ってみましょう。

```
【ワークフロー設計演習】

ワークフロー名: ______________

トリガー: ______________
条件:     ______________
アクション: ______________

□ 1. ワークフローを作成した
□ 2. テストチケットを作成して動作確認した
□ 3. タグが自動的に付与されたことを確認した

【発展課題】
以下の自動化を設計してみましょう（実装は任意）。

シナリオ: 「SLA違反が発生したら、チケットの担当者と
そのマネージャーにSlack通知を送り、Priorityを
1段階引き上げる」

トリガー: ______________
条件:     ______________
アクション1: ______________
アクション2: ______________
アクション3: ______________
```

次のセッションでは、Snap-inとマーケットプレイスを活用します。

## まとめ

このセッションを終えると、次のことができるようになっているはずです。

- Settings > Workflows からワークフローを作成できる
- トリガー・条件・アクションの3要素で自動化を設計できる
- シンプルなワークフローを実際に作成できる
- ノードの種類（Condition / AI Condition / Parallel / Wait / Loop / Action）を説明できる
- バージョニングの仕組み（既存実行は旧バージョンで完走）を理解している
- ワークフローの永続性（エラー後の再開が可能）を理解している

## 用語

| 用語 | 説明 |
|------|------|
| **Automations** | DevRevのルールベース自動化機能。トリガー・条件・アクションで構成される |
| **トリガー（Trigger）** | 自動化を発動させるイベント（Ticket Created、SLA Breached など） |
| **条件（Condition）** | トリガー発動後に追加で判定する論理条件 |
| **アクション（Action）** | 条件を満たした場合に実行する操作（タグ追加、担当者アサイン、通知送信など） |
| **AI Condition** | AIがテキストを解析し、意味ベースで条件判定するノード |
| **Parallel** | 複数のパスを同時実行するワークフローノード |
| **Wait** | 一定時間待機してから次のアクションを実行するノード |
| **Loop** | 条件が満たされるまで繰り返すノード |
| **Durable State（永続的な状態）** | ワークフロー実行中にエラーが発生しても状態が保持され、再開可能な仕組み |
| **バージョニング** | 新バージョンをpublishしても既存実行は旧バージョンで完走する仕組み |
