# 管理者設定とアクセス制御

> "ロール設計と権限制御を正しく設定すると、運用が大きく楽になる。"

## 問題

権限設定が雑な状態では、2つの問題が起きます。ゆるすぎると、誰でも何でも見えてしまいセキュリティリスクになります。

厳しすぎると、チームメンバーが必要な操作をできずに管理者への依頼が頻発します。

組織が大きくなるほど、適切なロール設計の重要性が増します。

## 解決策

DevRevのロールとメンバーシップを正しく設計します。

設計の基本方針は「最小権限の原則」に従いつつ、チームの業務効率を損なわないバランスをとることです。

## 仕組み

Settings の左メニューでどのカテゴリがあるかの全体像は [s02](/ja/s02) で先に押さえておくと、ここ（User Management や権限）にすぐ辿り着きやすいです。

**システムロール**

DevRevには以下のシステムロール（変更不可）が用意されています。

| ロール | 権限概要 |
|--------|---------|
| Owner | ワークスペース全体の管理権限。請求情報を含むすべての設定を変更可能 |
| Admin | ワークスペース設定の変更、メンバー管理、ロール割り当てが可能 |
| Member | 通常のチケット・Issue操作が可能。設定変更は不可 |

**カスタムロール**

システムロールだけでは不十分な場合、カスタムロールを作成できます。たとえば以下のようなケースです。

- サポートリーダー: チケット管理とSLA設定は可能だが、開発側の設定は変更不可
- 外部パートナー: 特定のPartに紐付くチケットのみ閲覧可能

**メンバーシップ管理**

DevRevでは、ユーザーを以下の単位で管理します。

- **個人**: 1人ずつロールを割り当てます
- **グループ**: 複数ユーザーをグループにまとめ、グループ単位でロールを割り当てます

グループを活用すると、チーム編成の変更時にロール設定を一括変更できるため、管理コストが大幅に下がります。

**ワークスペース設定の主要項目**

| 設定カテゴリ | 主な設定内容 |
|------------|------------|
| General | ワークスペース名、タイムゾーン、言語 |
| Members | メンバー招待、ロール割り当て、グループ管理 |
| Parts | プロダクト構造の定義 |
| SLA Policies | 優先度別の応答・解決目標時間 |
| Integrations | 外部サービスとの連携設定 |

自社の組織構造に合わせたロール設計を紙に書いてみましょう。

```
       必要な権限: ______________
       不要な権限: ______________

チーム名: ______________
  └── ロール: ______________
       必要な権限: ______________
       不要な権限: ______________

【考察】
- システムロールだけで足りるか？カスタムロールが必要か？
- グループはいくつ必要か？
```

次のセッションでは、分析ダッシュボードを活用してKPIを可視化します。

## まとめ

このセッションを終えると、次のことができるようになっているはずです。

- システムロール（Owner / Admin / Member）の違いを説明できる
- 自組織に合ったカスタムロールを設計できる
- グループを使ったメンバーシップ管理の利点を理解している
- 最小権限の原則に基づいたロール設計の考え方を説明できる

---

## 用語

| 用語 | 説明 |
|------|------|
| **Owner** | ワークスペース全体の管理権限を持つシステムロール。請求情報を含むすべての設定を変更可能 |
| **Admin** | ワークスペース設定の変更・メンバー管理・ロール割り当てが可能なシステムロール |
| **Member** | 通常のチケット・Issue操作が可能なシステムロール。設定変更は不可 |
| **カスタムロール** | システムロールでは不十分な場合に作成する独自のロール定義 |
| **グループ** | 複数ユーザーをまとめた管理単位。グループ単位でロールを割り当てられる |
| **最小権限の原則** | 業務に必要な最低限の権限のみを付与するセキュリティ設計方針 |
