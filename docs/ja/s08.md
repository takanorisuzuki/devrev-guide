# 管理者設定とアクセス制御

> "ロール設計と権限制御を正しく設定すると、運用が大きく楽になる。"

## 問題

権限設定が雑な状態では、2つの問題が起きる。ゆるすぎると、誰でも何でも見えてしまいセキュリティリスクになる。

厳しすぎると、チームメンバーが必要な操作をできずに管理者への依頼が頻発する。

組織が大きくなるほど、適切なロール設計の重要性が増す。

## 解決策

DevRevのロールとメンバーシップを正しく設計する。

設計の基本方針は「最小権限の原則」に従いつつ、チームの業務効率を損なわないバランスをとることである。

## 仕組み

**システムロール**

DevRevには以下のシステムロール（変更不可）が用意されている。

| ロール | 権限概要 |
|--------|---------|
| Owner | ワークスペース全体の管理権限。請求情報を含むすべての設定を変更可能 |
| Admin | ワークスペース設定の変更、メンバー管理、ロール割り当てが可能 |
| Member | 通常のチケット・Issue操作が可能。設定変更は不可 |

**カスタムロール**

システムロールだけでは不十分な場合、カスタムロールを作成できる。たとえば以下のようなケース。

- サポートリーダー: チケット管理とSLA設定は可能だが、開発側の設定は変更不可
- 外部パートナー: 特定のPartに紐付くチケットのみ閲覧可能

**メンバーシップ管理**

DevRevでは、ユーザーを以下の単位で管理する。

- **個人**: 1人ずつロールを割り当てる
- **グループ**: 複数ユーザーをグループにまとめ、グループ単位でロールを割り当てる

グループを活用すると、チーム編成の変更時にロール設定を一括変更できるため、管理コストが大幅に下がる。

**ワークスペース設定の主要項目**

| 設定カテゴリ | 主な設定内容 |
|------------|------------|
| General | ワークスペース名、タイムゾーン、言語 |
| Members | メンバー招待、ロール割り当て、グループ管理 |
| Parts | プロダクト構造の定義 |
| SLA Policies | 優先度別の応答・解決目標時間 |
| Integrations | 外部サービスとの連携設定 |

## このセッションで学ぶこと

**始める前の状態：**
- 全員がAdmin権限を持っている状態
- ロール設計を考えたことがない
- メンバー管理が場当たり的

**このセッション後の状態：**
- システムロール（Owner / Admin / Member）の違いを説明できる
- 自組織に合ったカスタムロールを設計できる
- グループを使ったメンバーシップ管理の利点を理解している

## 試してみる

自社の組織構造に合わせたロール設計を紙に書いてみよう。

```
【ロール設計シート】

  └── ロール: ______________
       必要な権限: ______________
       不要な権限: ______________

チーム名: ______________
  └── ロール: ______________
       必要な権限: ______________
       不要な権限: ______________

チーム名: ______________
  └── ロール: ______________
       必要な権限: ______________
       不要な権限: ______________

【考察】
- システムロールだけで足りるか？カスタムロールが必要か？
- グループはいくつ必要か？
```
