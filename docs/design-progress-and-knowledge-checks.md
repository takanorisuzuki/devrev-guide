# 設計ドラフト: 進捗管理と知識チェックの追加

> 作成日: 2026-06-23
> 対象プロジェクト: `devrev-guide`（Next.js 16 / App Router / Vercel）
> ステータス: ドラフト（レビュー前）
> 目的: Anthropic Academy（Skilljar）水準のセルフトレーニング体験に近づけるため、(1) 学習者向け進捗管理と (2) 知識チェックを段階的に追加する。

---

## 背景

GAP分析の結果、コンテンツの骨格（15セッション、統一テンプレート、冒頭の学習目標、persona別パス）は既に整っている。一方で、学習者の「定着」と「継続」を支える2つの仕組みが欠けている。

| 観点 | 現状 | 備考 |
|------|------|------|
| 学習目標の明示 | あり（全15本） | `keyInsight` を冒頭表示＋末尾「まとめ」に到達目標。GAPではない |
| 運営者向け完了分析 | あり | `SessionCompleteTracker.tsx` が GA4 に `session_complete` を送信。**学習者には何も見せない** |
| 学習者向け進捗管理 | **なし** | 完了状態の保存・進捗バー・再訪時の復元がない |
| 知識チェック | **なし** | クイズ／理解度確認がコード・コンテンツとも0件 |

本ドラフトはこの2つの「なし」を埋めるための実装方針を定める。

### 設計原則

- **ログイン不要を維持する。** 本サイトは認証なしの静的配信。進捗・採点はクライアント（`localStorage`）で完結させ、バックエンド・DBを増やさない。
- **既存テンプレートを壊さない。** セッションMarkdownの `問題→解決策→仕組み→試してみる→まとめ→用語→参考` 構造はそのまま。知識チェックは「試してみる」と「まとめ」の間に差し込む。
- **段階導入。** まず進捗管理を単独でリリース。知識チェックは1〜2セッションで試作してから全体展開する。
- **修了認定はスコープ外（フェーズ3で再検討）。** 受講証明・合格基準・スコア連動は当面作らない。理解の自己確認に絞る。

---

## フェーズ1: 学習者向け進捗管理

### 目的

学習者が「15本中どこまで終えたか」を一目で把握でき、再訪時に続きから戻れるようにする。長い学習パスからの離脱を防ぐ。

### 振る舞い

- 各セッションページの末尾に「このセッションを完了にする」ボタンを置く（または末尾到達で自動マーク＋手動トグル可）。
- 完了状態を `localStorage` に保存する（locale非依存。`s01` を ja で完了したら en でも完了扱い、を基本とする。要確認）。
- ホームの `SessionGrid` と persona path ページに進捗を反映：
  - 完了済みセッションにチェック表示。
  - 「全体 15本中 N本完了」「あなたのパス 8本中 3本完了」の進捗バー。
- 「進捗をリセット」操作を1つ用意する（設定的な小リンクで可）。

### データ仕様

`localStorage` キー: `devrev-guide:progress:v1`

```json
{
  "version": 1,
  "completed": { "s01": "2026-06-23T12:00:00Z", "s03": "2026-06-23T12:30:00Z" },
  "updatedAt": "2026-06-23T12:30:00Z"
}
```

- `completed` は `SessionId → ISO完了日時` のマップ。存在＝完了。
- `version` でスキーマ移行に備える。読み込み時に未知バージョンなら破棄して初期化。

### 実装対象（想定）

| ファイル | 変更内容 |
|----------|----------|
| `src/lib/progress.ts`（新規） | `localStorage` の読み書き、`getCompleted()` / `markComplete(id)` / `clear()`、SSRガード（`typeof window`）、変更通知（custom event or Zustand等）。**`src/lib/analytics.ts` とは別物**として実装し、GA4トラッカーと役割を混ぜない |
| `src/components/session/SessionCompleteButton.tsx`（新規） | 完了トグルUI。`'use client'`。`SessionCompleteTracker`（GA4用）はそのまま残す |
| `src/components/home/SessionGrid.tsx` | 完了バッジ・進捗バーの表示 |
| persona path ページ | パス単位の進捗バー |
| `src/components/session/PrevNextNav.tsx`（任意） | 「続きから」導線 |

### 受け入れ基準

- セッションを完了→別ページ遷移→戻ると完了状態が保持される。
- ブラウザ再起動後も保持される。
- `localStorage` が使えない環境（プライベートモード等）でもエラーで落ちず、進捗なしとして動作する。
- 既存の GA4 `session_complete` イベントは従来通り送られ続ける（リグレッションなし）。

### 工数感

1〜2日（バックエンドなし、UI＋localStorageのみ）。

## フェーズ2: 知識チェック（記録なしの自己確認）

### 目的

各セッションの要点が定着したかを、学習者自身がその場で確認できるようにする。**採点記録・合否判定は行わない**（フェーズ1の進捗管理とは独立）。

### 振る舞い

- 各セッション末尾「まとめ」の直前に、3〜5問の選択式クイズを表示。
- 回答すると即座に正誤と短い解説を表示（答え合わせ型）。
- スコアは画面に出すが**保存しない**。何度でもやり直せる。
- まずは **s01 と s03 の2本のみ** に実装して反応を見る（試作）。良ければ全15本へ展開。

### コンテンツ仕様

クイズはMarkdown本文に混ぜず、データとして分離する（多言語・保守性のため）。

`src/data/quizzes.ts`（新規）:

```ts
export type QuizQuestion = {
  id: string;              // 例: "s01-q1"
  question: { ja: string; en: string };
  choices: { ja: string; en: string }[];
  answerIndex: number;     // 正解の選択肢インデックス
  explanation: { ja: string; en: string };
};

export const QUIZZES: Partial<Record<SessionId, QuizQuestion[]>> = {
  s01: [ /* 3-5問 */ ],
  s03: [ /* 3-5問 */ ],
};
```

- 設問はセッションの「まとめ（到達目標）」に1対1で対応させて作る。目標達成の確認になる。
- 製品UI依存の細かい操作手順ではなく、概念理解を問う（メンテ頻度を下げるため）。

### 実装対象（想定）

| ファイル | 変更内容 |
|----------|----------|
| `src/data/quizzes.ts`（新規） | 設問データ（日英） |
| `src/components/session/KnowledgeCheck.tsx`（新規） | 出題・採点・解説表示。`'use client'`。記録なし |
| セッションページ | 「まとめ」直前に `QUIZZES[sessionId]` があれば描画 |

### 受け入れ基準

- s01 / s03 で出題・回答・即時フィードバックが動く。
- クイズ未定義のセッションでは何も表示されない（段階展開が安全にできる）。
- 日英で設問・選択肢・解説が正しく切り替わる。

### 工数感

試作（2本＋コンポーネント）で2〜3日。全15本展開は設問作成が主コスト。

### 保守上の注意

- 設問は製品仕様変更の影響を受ける。展開前に「概念問題に寄せる」方針を徹底し、UI手順の暗記問題は避ける。
- 全15本×数問は継続メンテ負荷になる。試作の反応次第で「全展開する／主要セッションのみ」を判断する。

## フェーズ3（将来・スコープ外）: 修了認定

フェーズ1・2の利用状況を見てから検討する。実施する場合の検討事項のみ列挙：

- 合格基準（クイズ正答率）と進捗の連動。
- persona別パス完走時の修了表示・バッジ・受講証明。
- persona別の総合演習（capstone）の追加。
- 記録の永続化（ログイン／メール送付）の要否。

---

## 実装順序の推奨

1. フェーズ1（進捗管理）を単独でリリース — 確実に効き、安く、リスクが低い。
2. フェーズ2をs01/s03で試作 → 反応を見て全展開を判断。
3. フェーズ3は1・2の成果を見て再検討。

## 留意点（要レビュー）

- 進捗のlocale扱い（ja/en共有 or 分離）を決める必要がある。
- `npm run build` と `npm run lint` で確認（テストスイートはなし）。
- 既存 `SessionCompleteTracker`（GA4）と新規進捗管理は**目的が異なる別系統**。両立させ、混同しないこと。
